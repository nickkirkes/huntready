# Implementation Plan: S03.11 — Confidence calibration audit + ADR-017 finalization

Plan-format-version: 1

## Story summary

S03.11 closes the loop on Q11 (confidence calibration). No production code is written.
Three artifacts ship:

1. **Synthesis report** at `docs/planning/epics/E03-confidence-calibration-synthesis.md` (PM-owned, new file).
2. **F12 retrofit edits** to existing per-story working notes — each `regulation_record`-producing story's note must record `(entity_id → assigned_confidence) + (source format / extraction op / transformation steps / correction-touched flag)` so the audit is reproducible.
3. **Conditional ADR-017 amendment DRAFT** at `docs/adrs/ADR-017-amendment-DRAFT.md` — only if any §7 defer-trigger fires. PM **does not** commit the amendment autonomously per the user's explicit instruction and ADR-017 §7. The DRAFT file is staged for user review.

The audit subject is **`regulation_record.confidence` only** (the sole entity with a confidence column per ADR-017 §3). The audit is sourced from the deterministic committed artifacts (`dea-2026.json`, `black-bear-2026.json`) rather than the live DB, which is empty per S03.10's T0 probe. This is acceptable because every artifact-to-DB transformation in `load_regulation_records.py` is deterministic; the artifact-derived values are byte-equivalent to what a live `regulation_record` row would carry.

The load-bearing PM decision in this story is the **adjudication of ADR-017 §7's "any tier has 0 rows" trigger** against V1 Montana data, where `low` is confirmed to have 0 rows by data property (not by framework gap). This decision is made in T8 and ships in the synthesis report.

### T6: Run audit — verify each sampled row against ADR-017 framework (~15 min)
**Files:** docs/planning/epics/E03-confidence-calibration-synthesis.md
**Depends on:** T1, T2, T3, T4, T5 (T6 reads the retrofitted F12 blocks from T1-T4 and the sample table from T5)
**Action:** Append §3 Audit results to the synthesis report.
**Details:**
- **CRITICAL — audit independence guard (anti-circularity):** the artifact's `extraction_confidence` field was produced by `_assign_row_confidence` in `extract_dea.py`. If you compute `framework_predicted` by re-reading that same field, the audit becomes a tautology (100% pass-rate by construction) and Trigger 3 can never fire. The audit must be **independent**: you derive `framework_predicted` from the row's F12 inputs (`source_format`, `extraction_op`, `transformation_steps`, `correction_touched`) using ADR-017's rules, WITHOUT looking at the artifact's `extraction_confidence` value first. Only after `framework_predicted` is recorded do you reveal the artifact's value and compare. This is the load-bearing methodological discipline of the audit.
- For each row in T5's sample table, perform the verification in this strict order:
  1. **Pull F12 inputs only.** Find the row's edge case in the appropriate retrofitted working note (from T1-T4) and read the 4-field F12 tuple: `source_format`, `extraction_op`, `transformation_steps`, `correction_touched`. For rows that aren't in the F12 sample blocks (rows beyond the per-story samples), derive the 4-field tuple from the edge case's general rules documented in §1 of the synthesis report. Do NOT read `extraction_confidence` from the artifact at this step.
  2. **Apply ADR-017's rules independently.** Read ADR-017 §4 "Three-tier framework" and §5 "Aggregation rule". The rules to apply (derived from the ADR, not from `_assign_row_confidence` directly):
    - Structured-table-cell extraction with HD-code present in license_code → HIGH
    - Prose-only license row (no HD-code in license_code) → MEDIUM
    - Heading-anchored prose extraction (not table-cell) → MEDIUM
    - Empty license_code or empty opportunity → LOW
    - Correction-touched row → demote_one_tier (HIGH→MEDIUM, MEDIUM→LOW, LOW→LOW per `pdf.py:141`)
    - Section-level MIN-aggregation over row tiers per ADR-017 §5
    Write `framework_predicted` in the audit table based on this derivation.
  3. **Reveal artifact value.** Read the row's `extraction_confidence` field from `dea-2026.json` / `black-bear-2026.json` / the working note. Write this as `assigned_confidence` in the audit table.
  4. **Compare.** Record match (✓) or non-match (✗). For non-matches, derive the failure mode in the evidence column (which rule was applied vs. what the artifact recorded, and why).
- Write the audit results as a follow-on table extending T5's sample table with two new columns: `framework_predicted` and `match` (✓/✗). The cleanest approach is to write a parallel table referencing T5's row numbers: `| # | extraction_confidence (assigned) | framework_predicted | match | evidence (ADR-017 rule applied) |`
- Compute the audit pass-rate: `match_count / 50` × 100. State it explicitly.
