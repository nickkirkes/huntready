# Implementation Plan: S06.10 — CO `jurisdiction_binding` generation

Plan-format-version: 1

## Summary

The final E06 DB-write story. Extends the existing S05.6 scaffold
`ingestion/states/colorado/load_jurisdiction_bindings.py` into a full cross-cutting
binding loader, the CO analog of MT's `load_jurisdiction_bindings.py`. Reads the
committed `geometry-overlays.json` × CO `regulation_record` cross product, walks 4
conceptual builders (statewide / overlay self-row / overlay portions / no-hunt-zone
nearby), and UPSERTs in one atomic three-phase transaction. Also bundles the
S06.0/D3 `CO_STATE_CODE → _STATE` rename across the 4 geometry-era CO loaders.

## Key design decisions (resolved from discovery — NOT open questions)

1. **Statewide builder → 0 rows for CO V1.** CO wrote 0 `CO-STATEWIDE-*` regulation_records
   (S06.6 reality; spec line 1078 assumed they exist). The statewide builder exists as a
   code path; the guard asserts `len(statewide_rrs) == 0` and fails loud on ANY
   `CO-STATEWIDE-*` jurisdiction_code (mirrors MT's symmetric unexpected-statewide guard,
   but with `expected_statewide_pairs = frozenset()`).

2. **Overlay builder → self-row (`primary_unit`) only; `restricted_area` fixture rows SKIPPED.**
   The CO `geometry-overlays.json` contains 15 `gmu→restricted_area` rows
   (`role_for_e03='restricted_area'`) covering all 10 `EXPECTED_CO_RA_ORPHAN_IDS`. Per AC
   #1103 ("overlay **self-row**") + AC #1104 (no-hunt-zone builder iterates the 10 zones
   from `EXPECTED_CO_RA_ORPHAN_IDS` via the nearby spatial query), the `restricted_area`
   fixture rows are NOT turned into bindings by the overlay builder — the dedicated
   hardcoded `_build_no_hunt_zone_bindings_co` is the SOLE producer of zone bindings (via
   the 5 km nearby spatial query, not the fixture overlap rows). The overlay builder's
   eligibility function recognizes `restricted_area` as a known role and returns
   `False` (seen-and-skipped, never raises).

3. **Known Issue #6 → hardcoded path (S06.0/D2).** CO's role gate
   `_VALID_ROLE_FOR_E03_CO = frozenset({"primary_unit", "portion", "restricted_area"})`
   stays narrow; `no_hunt_zone` / `other_overlay` are produced ONLY by the hardcoded
   no-hunt-zone builder, never via the fixture gate. (CO has no `cwd_management_zone`.)

4. **AFA 9+1 split (Known Issue #12 RESOLVED, hard constraint AC #1040/#1104).** The
   no-hunt-zone builder iterates all 10 `EXPECTED_CO_RA_ORPHAN_IDS`; the AFA id
   `CO-restricted-united-states-air-force-academy-geom` binds `role='other_overlay'`;
   the other 9 NPS/NM rows bind `role='no_hunt_zone'`. Locked by
   `test_afa_bound_other_overlay_not_no_hunt_zone`.

5. **`_STATE` wins (S06.0/D3).** Rename `CO_STATE_CODE → _STATE` in the 4 geometry-era CO
   loaders + README + stale docstring naming-notes. `CO_STATE_SLUG` companions untouched.

6. **Count-guard band is PROVISIONAL** `(300, 1200)`. Dry-run requires a live/dev DB
   (build phase reads reg_records + runs spatial queries), which is operator-gated.
   Estimated baseline ≈ 398 self-row `primary_unit` bindings (one per reg_record whose GMU
   has a geometry) + no-hunt-zone fan-out. Band is documented as provisional pending the
   operator's first dry-run empirical count → narrow to ±30% (the S04.2 T16-narrowing
   analog the spec anticipates at line 1064). Tests verify band *logic*, not the exact count.

## CO deltas vs the MT template (`montana/load_jurisdiction_bindings.py`)

- `_derive_parent_geometry_id_co(rr)`: `f"{rr.jurisdiction_code}-geom"` for any
  `CO-GMU-{int}` code; fail loud on `CO-STATEWIDE-*` (no statewide bindings in V1) or any
  other pattern. (No species-prefix partitioning — every CO GMU serves all species.)
- `is_binding_eligible_co(species_group, parent_gid, row)`: self-row → True;
  `portion` → True (CO has zero; code path preserved per AC #1103); `restricted_area` →
  False (skipped — handled by no-hunt-zone builder); non-self `primary_unit` → False;
  unknown role → raise. CO has no species-axis geometry partitioning, so there is no
  Step-B species-prefix check.
- No-hunt-zone builder uses the scaffold's `query_nearby_gmus_for_zone(conn, zone_geom_wkt)`
  (WKT-based, NOT zone_id JOIN like MT). Therefore pre-fetch each zone's WKT via
  `_fetch_zone_wkts` (`SELECT id, extensions.ST_AsText(geom) FROM geometry WHERE id = ANY(%s)`)
  before calling it. The scaffold function + `_QUERY_NEARBY_GMUS_FOR_ZONE_SQL` are LOCKED
  by `test_co_binding_reference.py` — **reuse them as-is, do not modify**.
- AFA dispatch in the no-hunt-zone builder (see decision #4).
- `EXPECTED_CO_RA_ORPHAN_IDS` imported from `states.colorado.build_overlay_fixture`.
- `_JURISDICTION_BINDING_ID_FORMAT` imported from `states.colorado.load_regulation_records`
  (DO NOT redefine).
