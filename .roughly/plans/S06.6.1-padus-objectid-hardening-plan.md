# Implementation Plan: S06.6.1 — PAD-US OBJECTID outFields hardening

Plan-format-version: 1

## Summary

PAD-US 4.1 was republished and now omits the top-level GeoJSON `id` field unless `OBJECTID` is in
the request's `outFields`. `load_restricted_areas.py`'s `_RA_OUT_FIELDS` doesn't request OBJECTID,
and `lib/arcgis._read_objectid` silently falls back to `feature["id"]` (now absent) → returns `None`
→ the RA manifest-hash loop fails loud with 0 rows written. This story (a) adds `"OBJECTID"` to the
outFields tuple (immediate workaround), (b) removes the silent fallback from the OID-critical paths
by introducing a strict `_require_objectid` helper and routing the two manifest-hash loops through it,
(c) audits all CO `_*_OUT_FIELDS` tuples, and (d) locks both with regression tests.

## Design decision (load-bearing — drives T2/T3/T4)

`_read_objectid` is called from **8 sites**:
- **6 error-message-context callers** (`arcgis.py:335, 490, 499, 510, 527, 559`) build diagnostic
  strings and legitimately tolerate a best-effort value or `None`. These must NOT start raising.
- **2 OID-critical manifest-hash loops** that already treat `None` as fatal:
  `fetch_features` (`arcgis.py:822`) and `load_restricted_areas.py:374`.

Making `_read_objectid` itself raise would break the 6 error-message callers (a feature reported for
some *other* error that also lacks a resolvable OID would raise a masking OBJECTID error). The epic's
AC #624 explicitly permits the raise to live in "its raising caller." So:

- **Add a new strict `_require_objectid(feature, *, oid_field=None) -> int | str`** to `lib/arcgis.py`
  that scans `properties`/`attributes` for the candidate OID keys and **does NOT fall back to
  `feature["id"]`** — raising `ArcGISError` with a rich diagnostic when no OBJECTID is found.
- **Leave `_read_objectid` unchanged** (best-effort, retains the `feature["id"]` fallback) for the
  error-message callers and the dedup-key path (`arcgis.py:744`, which intentionally tolerates).
- **Route only the two OID-critical manifest-hash loops** through `_require_objectid`.

This localizes the behavior change to exactly the manifest-hash gate, which is where the silent
fallback masked the republish. Test (b) "top-level `id` only → raises" locks the new strictness:
previously the manifest loop would hash on the `feature["id"]` fallback; now it raises, forcing
`outFields` to carry OBJECTID.

## Spec deviations (confirmed in discovery — surfaced at plan gate, baked into plan)

1. **`_GMU_OUT_FIELDS` does not exist.** `load_gmus.py` uses `arcgis.fetch_features()`, which builds
   `outFields` from `metadata.out_fields` (server `fields[]`), so OBJECTID is naturally requested.
   The grep finds **only** `_RA_OUT_FIELDS`. T2-audit → no GMU code change; confirm in closure note.
2. **No pinned-SHA constant in `test_load_co_restricted_areas.py`.** That test mocks
   `arcgis.compute_feature_hash` → `"aa"*32`; no real hash is asserted. AC #629's "update the pinned
   SHA constant" references a constant that does not exist (carried over from the extraction-artifact
   stories). → Documented N/A in closure note; nothing to re-pin.
3. **No committed PAD-US fixtures + no `--dry-run` flag.** `git ls-files` shows zero committed PAD-US
   fixtures (S05.4 left them operator-pending Group B); `load_restricted_areas.py` has no `--dry-run`
   (argparse only `--fetch-year`) and a run goes straight to `db.connect()`. Regenerating the fixture
   (T6 / AC #627–628) requires live PAD-US network + `DATABASE_URL` — exactly the operator-pass
   resume. → **T6 fixture regen + AC #627/#628 deferred to the operator-pass resume** (Group-B-adjacent);
   the build session ships code + tests + pitfall. Documented in closure note.

These three deviations mean the build-session scope is **T1, T2, T3, T4, T5** (+ closure memo + pitfall
candidate). T6/AC #627–629 are operator-pass artifacts.

### T2: Add strict `_require_objectid` helper to lib/arcgis.py (~5 min)
**Files:** ingestion/ingestion/lib/arcgis.py
**Action:** Add a new module-level function `_require_objectid` (place it immediately after
`_read_objectid`, ~line 631) and update `_read_objectid`'s docstring to point OID-critical callers at it.
**Details:**
- Signature: `def _require_objectid(feature: dict[str, Any], *, oid_field: str | None = None) -> int | str:`
- Reuse the same candidate-key derivation as `_read_objectid` (`common_keys = ("OBJECTID", "objectid", "FID")`;
  if `oid_field` is not None and not in common_keys, prepend it). Scan `("properties", "attributes")`
  dicts for the candidate keys; on first hit return `val if isinstance(val, (int, str)) else str(val)`.
- **Do NOT fall back to `feature["id"]`.** If no candidate key is found in any attrs dict, `raise ArcGISError(msg)`.
- The diagnostic `msg` MUST contain the literal strings `"OBJECTID"` and `"outFields"`, plus:
  (a) the failure mode ("no resolvable OBJECTID in feature properties/attributes");
  (b) the typical root cause (upstream ArcGIS republish — the service omits the top-level GeoJSON `id`
      unless `OBJECTID` is in the request `outFields`; add it to the loader's `_*_OUT_FIELDS`);
  (c) forensic context: the feature's top-level keys (`sorted(feature.keys())`) and the first ~10
      property keys (from `properties` if a dict, else `attributes`, else `"<none>"`).
  Build the message with the module's existing f-string style (see `arcgis.py:824–828`). Keep it a
  single-arg `ArcGISError(msg)`.
- Update `_read_objectid`'s docstring: add a sentence that OID-critical paths (manifest-hash, anywhere
  a missing OID must be fatal) MUST call `_require_objectid` instead — `_read_objectid` stays
  best-effort (retains the `feature["id"]` fallback) for error-message context and dedup keys.
- Do NOT modify `_read_objectid`'s body. Do NOT touch the 6 error-message callers or the dedup loop (line 744).
**Verify:** `cd ingestion && .venv/bin/ruff check ingestion/lib/arcgis.py && .venv/bin/mypy ingestion/lib/`
**UI:** no

### T5: `TestStateAdapterOutFieldsIncludeObjectid` AST-walk test (~5 min)
**Files:** ingestion/tests/test_load_co_restricted_areas.py
**Depends on:** T1
**Action:** Add a new `TestStateAdapterOutFieldsIncludeObjectid` class that AST-walks every `.py` file
in the CO state-adapter dir, finds every module-level `_*_OUT_FIELDS` tuple assignment, and asserts
`"OBJECTID"` is a member. A future CO loader that adds an `_*_OUT_FIELDS` tuple without OBJECTID fails
this test. `ast`, `importlib.util`, and `Path` are already imported at the top of this file.
**Details:**
- Locate the CO adapter dir via `Path(importlib.util.find_spec("states.colorado.load_restricted_areas").origin).parent`.
- Iterate `sorted(co_dir.glob("*.py"))`. For each, `ast.parse(path.read_text(encoding="utf-8"))`, then
  `ast.walk` for `ast.Assign` nodes whose target is an `ast.Name` ending with `_OUT_FIELDS` and whose
  `node.value` is an `ast.Tuple` (or `ast.List`). Collect each into a `dict[str, list[str]]` keyed by
  `"<filename>:<const_name>"` with the string-constant elements (`elt.value for elt in elts if
  isinstance(elt, ast.Constant) and isinstance(elt.value, str)`).
- `assert collected`, with a message, so an empty result (e.g. the glob/AST-walk silently finding
  nothing) fails loud rather than passing vacuously — and pins that at least `_RA_OUT_FIELDS` is found.
- For each collected tuple, `assert "OBJECTID" in elts, f"{key} missing OBJECTID"`.
- Add a brief class docstring citing S06.6.1 and the PAD-US-republish failure mode.
**Verify:** `cd ingestion && .venv/bin/pytest tests/test_load_co_restricted_areas.py -q`
