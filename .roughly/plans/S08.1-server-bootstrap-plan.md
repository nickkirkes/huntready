# Implementation Plan: S08.1 — Server bootstrap + Streamable HTTP transport on Cloudflare Workers

Plan-format-version: 1

## Context & Key Decisions (read first)

This is the **first TypeScript serving story** in HuntReady. It replaces the `mcp-server/src/index.ts`
placeholder with a Cloudflare Workers entrypoint serving MCP over Streamable HTTP via the SDK's
**stateless `createMcpHandler` path** (no Durable Objects), and stands up the serving test harness +
first CI workflow.

**Verified facts from discovery + live introspection (against installed packages):**
- `createMcpHandler` is exported by the **Cloudflare Agents SDK** (`import { createMcpHandler } from "agents/mcp"`), NOT by `@modelcontextprotocol/sdk`. Signature: `createMcpHandler(server, options?) => (request, env, ctx) => Promise<Response>`. It wraps `WorkerTransport` (extends the SDK's `WebStandardStreamableHTTPServerTransport`).
- Installed + pinned versions (already `npm install`ed into `mcp-server/node_modules`): `agents@0.17.0`, `wrangler@4.105.0`, `@cloudflare/workers-types@4.20260627.1`, `vitest@4.1.9`, `@modelcontextprotocol/sdk@1.29.0` (agents pins SDK exactly 1.29.0 — satisfies the ≥1.26 per-request constraint; no version conflict with `^1.9.0`).
- `agents/mcp` transitively imports `cloudflare:`-protocol modules (`node:async_hooks`, `cloudflare:workers` via `agents/index.js`) → **it is NOT importable in plain Node**. Therefore `src/index.ts` (which imports `agents/mcp`) cannot be imported by a Node-pool vitest test.
- `McpServer`, the SDK `Client`, and `InMemoryTransport` ARE Node-importable. `McpServer` declares its tool registry privately as `_registeredTools`; the public, type-safe way to assert "zero tools" is to connect an SDK `Client` over an in-memory transport and call `client.listTools()`.
- `McpServer` constructor is `(serverInfo, options?)`. **Empirically verified across all three candidate approaches (SDK 1.29):**
  - `{ capabilities: { tools: {} } }` alone declares the capability during `initialize` but does NOT register a `tools/list` handler → `client.listTools()` throws `-32601` AND a deployed server would error on `tools/list` (non-conformant). Rejected.
  - Registering an empty handler via the PUBLIC `server.server.setRequestHandler(ListToolsRequestSchema, …)` works for E08 BUT is **forward-incompatible**: E09/E10's first `server.tool()` call then throws `"A request handler for tools/list already exists"` (the public path does not set `_toolHandlersInitialized`). Rejected.
  - **CHOSEN — call the SDK's idempotent initializer `setToolRequestHandlers()`** (it is typed `private` but exists exactly to install the `tools/list`/`call_tool` handlers once and set `_toolHandlersInitialized = true`). Empirically: `tools/list` returns `[]`, `_registeredTools` stays empty, AND a later `server.tool()` (E09/E10) composes cleanly because the init flag short-circuits the re-register. Call it via a typed `unknown` narrowing (NO `any`): `(server as unknown as { setToolRequestHandlers(): void }).setToolRequestHandlers();`. Note: `getServerCapabilities()` then reports `{ tools: { listChanged: true } }` (the init adds `listChanged`) — tests must assert `caps?.tools` is *defined*, not deep-equal `{}`. This is the only approach that is both conformant for E08 AND forward-compatible for E09's tool registration; the trade is a documented private-method call (justified by the empirical comment).
- **Empirically verified:** `LATEST_PROTOCOL_VERSION` is exported from `@modelcontextprotocol/sdk/types.js` and equals `"2025-11-25"` (matches the spec target). The SDK `Client` over `InMemoryTransport` does NOT expose the negotiated `protocolVersion` publicly (no `getProtocolVersion()`; `InMemoryTransport` doesn't implement `setProtocolVersion`), so the SDK-owns-the-version assertion is made against `LATEST_PROTOCOL_VERSION` + the source-scan guard in T7, not via a client getter.

**Test-harness architecture decision (load-bearing):**
- **Primary harness = Node-pool `vitest`.** Group A protocol-conformance is proven by driving the REAL MCP `initialize`/`tools/list` exchange through the SDK `Client` ↔ `InMemoryTransport` ↔ our `createMcpServer()` factory. This proves SDK-negotiated `protocolVersion`, `serverInfo`, declared `capabilities`, and an empty `tools` array — without needing workerd.
- The **workerd-specific wiring** (`createMcpHandler` + Streamable HTTP over the live Worker) is proven in **Group B** (MCP Inspector against the deployed preview), matching the epic's explicit Group A (local/static at-merge) / Group B (deploy-verified) split. This is why `src/index.ts` is kept a thin, per-request shim over the Node-testable `createMcpServer()` factory.
- The `src/index.ts` per-request-instantiation requirement (which cannot be unit-imported in Node) is locked by a **source-scan test** asserting the server is constructed inside `fetch`, not at module scope.
- A `@cloudflare/vitest-pool-workers` integration test (real workerd handshake) is **explicitly out of scope for S08.1** to keep the foundation low-risk; recorded as a forward-note for the closure (E09 may add it). Stage-4 may promote it if desired.

**Module split:**
- `src/server.ts` — `createMcpServer(): McpServer` factory. MCP-SDK-only imports → Node-testable. Source of server identity + capabilities + the (empty) tool registry. Callable per-request.
- `src/index.ts` — Worker `fetch` entrypoint. Per request: `createMcpServer()` then `createMcpHandler(server)(request, env, ctx)`.

### T3: Worker entrypoint (per-request, stateless, no DO) (~4 min)
**Files:** mcp-server/src/index.ts
**Depends on:** T2
**Action:** Replace the placeholder `console.log` with the Cloudflare Workers `fetch` entrypoint that serves MCP over Streamable HTTP via `createMcpHandler`, constructing the server + handler **per request**.
**Details:**
- Delete the existing `console.log("HuntReady MCP Server — scaffold");`.
- Import `createMcpHandler` from `agents/mcp`. Import `createMcpServer` from `./server.js` (`.js` extension for Bundler resolution).
- Define a minimal `interface Env {}` (no bindings in S08.1; S08.2 adds the DB binding). Use `@cloudflare/workers-types` globals (`ExportedHandler`, `Request`, `Response`, `ExecutionContext`).
- Export default handler:
  ```ts
  export default {
    async fetch(request, env, ctx): Promise<Response> {
      // Per-request instantiation (SDK >=1.26 + stateless workerd requirement):
      // construct the MCP server + handler INSIDE fetch, never in module/global scope.
      const server = createMcpServer();
      const handler = createMcpHandler(server);
      return handler(request, env, ctx);
    },
  } satisfies ExportedHandler<Env>;
  ```
- Add a comment above the `const server = ...` line explicitly stating per-request instantiation is required and why (no module-scope server; workerd cannot share state safely across concurrent invocations). The tests/boundary.test.ts source-scan asserts this.
- Stateless: no `Mcp-Session-Id` handling in app code (the SDK/transport owns protocol-version + session headers). Do NOT hard-code `protocolVersion` anywhere.
- No `any`; imports nothing from `ingestion/`; does not import `@supabase/supabase-js`.
**Verify:** `cd mcp-server && npx tsc --noEmit` (must pass clean — this is the real compile gate for index.ts importing agents/mcp + workers-types).

- Helper: create a linked pair `const [clientTransport, serverTransport] = InMemoryTransport.createLinkedPair();`, build `const server = createMcpServer(); await server.connect(serverTransport);`, build `const client = new Client({ name: "test", version: "0.0.0" }); await client.connect(clientTransport);`.
