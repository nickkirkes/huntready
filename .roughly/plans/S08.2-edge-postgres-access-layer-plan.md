# Implementation Plan: S08.2 — Edge-Postgres driver spike + read-only-enforced access layer

Plan-format-version: 1

## Spike decision baked into this plan (Group A AC #1 / #2)

**Chosen path: `postgres.js` over a direct connection, SELECT-only-role DSN held as a Workers Secret, connect-per-request, targeting the Supabase Supavisor transaction pooler.** Hyperdrive is recorded as the considered-but-deferred alternative in the working note (it changes only *where the connection string is sourced* — the access-layer code is identical; it can be adopted later without a code change).

Rationale (recorded as ADR-024 addendum content in the working note, T6):
- **`postgres.js`, not `@neondatabase/serverless`.** Neon's `fetch`-HTTP mode only speaks to Neon's SQL-over-HTTP endpoint; its WebSocket `Pool` mode needs a `wsproxy` in front of Postgres. Neither works against **Supabase**. `postgres.js` speaks the standard Postgres wire protocol and works against Supabase from both Node (TCP) and workerd (`cloudflare:sockets`, enabled by the already-present `nodejs_compat` flag).
- **Direct + Workers Secret, not Hyperdrive, for the V1 foundation.** Lighter: no Cloudflare cloud resource to provision (smaller Group B operator step — just `wrangler secret put`). The access layer takes a plain connection-string, so switching to Hyperdrive later only changes how `index.ts` sources the DSN.
- **Group A / Group B split de-risks workerd:** Group A proves the access layer + role-level write-rejection **in Node** (vitest against a CI Postgres with the committed GRANT). Group B proves the same code runs **from the deployed edge runtime** (live MT read + provably-rejected write). workerd-`cloudflare:sockets` compatibility is a Group B verification, not an at-merge blocker.

**Stage-4 plan review must confirm:** (a) `postgres.js`'s current Cloudflare-Workers support against `compatibility_date 2025-11-25` + `nodejs_compat`; (b) that neon was correctly ruled out for Supabase; (c) the transaction-pooler `prepare: false` requirement. If review finds direct-socket support shaky, the fallback is Hyperdrive + `postgres.js` (same code; T3/T5 unchanged; only T3's `Env`/wrangler wiring + T6's recorded decision differ).

### T2: Author the SELECT-only role GRANT SQL (~5 min)
**Files:** supabase/grant-readonly-role.sql
**Action:** Create the committed, idempotent GRANT SQL provisioning a dedicated SELECT-only Postgres role, plus RLS ALLOW-SELECT policies (required because every table is `FORCE ROW LEVEL SECURITY`).
**Details:**
- Header comment: this is **operator-applied DB provisioning, NOT a migration** (it lives outside `supabase/migrations/` and is applied by hand to each environment, like the `supabase db query` operator steps in M1/M2). The SELECT-only-role DSN is held in Workers Secrets, never in source. The role is created without a password here; the operator/CI sets the password out-of-band (`ALTER ROLE ... PASSWORD ...`) and composes the DSN.
- Make the whole script **idempotent and table-set-agnostic** so the *same file* applies cleanly on the dev Supabase project (all 10 tables, FORCE RLS on) AND on the minimal CI substrate (only `geometry` exists):
  1. `DO $$ BEGIN IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'huntready_readonly') THEN CREATE ROLE huntready_readonly LOGIN; END IF; END $$;` (LOGIN, no password).
  2. `GRANT CONNECT ON DATABASE` — note: hardcoding a DB name is environment-specific; instead grant connect dynamically OR document that `GRANT CONNECT` is environment-applied. Use a `DO` block: `EXECUTE format('GRANT CONNECT ON DATABASE %I TO huntready_readonly', current_database());`.
  3. `GRANT USAGE ON SCHEMA public TO huntready_readonly;` and `GRANT USAGE ON SCHEMA extensions TO huntready_readonly;` (PostGIS functions live in `extensions`).
  4. `GRANT SELECT ON ALL TABLES IN SCHEMA public TO huntready_readonly;` (table-set-agnostic — grants on whatever tables exist).
  5. `ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO huntready_readonly;` (future tables).
  6. RLS ALLOW policies via a `DO` block that loops `FOR t IN SELECT tablename FROM pg_tables WHERE schemaname='public' LOOP` and, **only if no such policy already exists** (`IF NOT EXISTS (SELECT FROM pg_policies WHERE schemaname='public' AND tablename=t.tablename AND policyname='huntready_readonly_select')`), `EXECUTE format('CREATE POLICY huntready_readonly_select ON public.%I FOR SELECT TO huntready_readonly USING (true)', t.tablename);`. This makes reads work under FORCE RLS on the dev project; on the CI substrate (RLS not forced) the policy is harmless.
- **Critically: grant ONLY SELECT.** No INSERT/UPDATE/DELETE grant anywhere — that absence is what produces SQLSTATE 42501 on a write attempt (T5).
- No passwords, no DSNs, no secrets in this file.
**Verify:** File is syntactically valid SQL (psql `-f` parse). Full validation is in T4/T5 (applied against CI Postgres). `grep -iE "INSERT|UPDATE|DELETE|password|postgres://" supabase/grant-readonly-role.sql` returns no grant of write privileges and no credentials.

### T3: Implement the read-only access layer + wire the credential (~5 min)
**Files:** mcp-server/src/db.ts (create), mcp-server/src/index.ts (modify), mcp-server/wrangler.jsonc (modify), mcp-server/.gitignore (create)
**Action:** Create `src/db.ts` — the single read path (`createDbClient(dsn)` factory using postgres.js); extend `Env` with the secret; document the secret in `wrangler.jsonc`; gitignore `.dev.vars`.
**Details:**
- `mcp-server/src/db.ts` (no `any`; imports only `postgres`):
  - `import postgres from "postgres";` — **import ONLY the default export. No `import type { ... } from "postgres"` named type imports** (e.g. `postgres.Sql`, `Row`, `ParameterOrJSON`).   - Export `createDbClient(dsn: string): DbClient`. Inside: `const sql = postgres(dsn, { prepare: false, max: 1 });` (`prepare: false` is required by Supabase's transaction-mode pooler/pgBouncer — a load-bearing comment must say so; `max: 1` honors workerd's no-cross-invocation-socket-reuse — one connection per request, no shared pool). `query` calls `sql.unsafe(text, params as unknown[])` and returns it (postgres.js `.unsafe(text, params)` **binds** params positionally — `$1`, `$2` — it is injection-safe despite the name; add a comment stating this). `close` calls `await sql.end({ timeout: 5 })`.
  - Type carefully so `tsc` strict (workers-types only, `skipLibCheck` on) passes with no `any`. Cast postgres.js's row result to `T[]` via `as unknown as T[]` at the single boundary if its generic doesn't line up; keep the public surface `any`-free.
  - Export two named smoke-query SQL constants the access layer / tests reuse (so the proven idiom is defined once):
    - `READ_SMOKE_SQL = "SELECT id, state, kind FROM geometry WHERE state = $1 LIMIT 1"` (plain parameterized read; GiST-index-eligible; no cast).
    - `ST_VALID_SMOKE_SQL` = the **WKT round-trip** validity check honoring BOTH Supabase quirks: every `ST_*` is `extensions.`-prefixed AND the geometry-only `ST_IsValid` round-trips via WKT (no direct `geom::geometry` cast):
      `"SELECT id, extensions.ST_IsValid(extensions.ST_GeomFromText(extensions.ST_AsText(geom), 4326)) AS valid FROM geometry WHERE state = $1 LIMIT 1"`.
      Comment: references `.roughly/known-pitfalls.md` "geom::geometry direct cast not enabled" + the `extensions.`-prefix rule; this is the idiom E09/E10 inherit (it bit M2 twice — S05.7). Describe the broken idiom in prose in the comment, do NOT embed the literal `geom::geometry` token adjacent to the corrected SQL (per the known-pitfalls cubic-token-match lesson).
  - `createDbClient` MUST NOT be invoked at module scope anywhere (locked by T5 boundary guard).
- `mcp-server/src/index.ts`: extend `Env` to `interface Env { SUPABASE_READONLY_DSN: string; }` with a comment: SELECT-only-role DSN, held as a Workers Secret (`wrangler secret put SUPABASE_READONLY_DSN`); `.dev.vars` for local dev. **Do NOT call `createDbClient` here yet** — the live call site (internal health check) is S08.3; S08.2 ships the access layer + credential wiring, exercised by `db.test.ts`. (Leaving the field unused is fine; tsc does not flag unused interface members.)
- `mcp-server/wrangler.jsonc`: add a comment block documenting the secret (name `SUPABASE_READONLY_DSN`, set via `wrangler secret put`, `.dev.vars` for `wrangler dev`) — **never the literal DSN**. Workers Secrets are runtime `env`, not a `wrangler.jsonc` binding entry, so this is a documentation comment, not a config key. Keep the no-DO posture intact (do not add `durable_objects`/`migrations`).
- `mcp-server/.gitignore` (new): `.dev.vars` and `.dev.vars.*` (wrangler local-secret file — must never be committed).
