# Implementation Plan: S08.3 — Shape C response builder + envelope/section types + reusable mechanisms

Plan-format-version: 1

Story: `docs/planning/epics/E08-mcp-server-foundation.md` §S08.3. First-class context: this is the TypeScript serving stack (`mcp-server/`). Hard constraints: imports **nothing** from `ingestion/`; **no `any`**; `tsc` strict (both `tsconfig.json` src + `tsconfig.test.json`); `mcp-server/src/types/schema.ts` has a **ZERO git diff**; `supabase/`/`ingestion/`/`web/` untouched. Test baseline floor (S08.2): 18 passed + 5 skipped (local) / 23 (CI) — S08.3 grows it additively.

## Design decisions (load-bearing — reviewed at Stage 4)

1. **Two representations, drift-guarded.** Response types are plain TS **interfaces** in a new `src/types/response.ts` that mirror `architecture.md §"Response shape"` **token-for-token** (inline literal unions, no local aliases — makes the Shape-C reviewer's interface-by-interface check trivial) and **import** shared types (`SourceCitation`, `DrawSpec`, `ReservedPool`, `ClosurePredicate`, `WeaponType`, `Residency`, `GeometryRole`) from `./schema.js` rather than redefining. A parallel **zod** schema (`src/output-schema.ts`) provides runtime validation + the MCP `outputSchema` E09 inherits. Each zod schema is **drift-guarded against its interface** so the two cannot diverge silently (a mismatch becomes a `tsc` error).
2. **zod is the validation mechanism.** zod `4.3.6` is already resolved (deduped, single copy) as a transitive dep of `@modelcontextprotocol/sdk@1.29.0`; the SDK's `registerTool({ outputSchema })` accepts a zod schema (`ZodRawShapeCompat | AnySchema`). Add `zod` as a **direct** dependency (don't rely on a transitive dep that could vanish on an SDK bump). The validating helper uses `schema.safeParse`. This is the idiomatic MCP foundation E09/E10 inherit; the negative test exercises a real validation rejection (not a hand-rolled checker).
3. **Internal health check = an HTTP `/healthz` route, not an MCP tool.** `runHealthCheck(dsn)` (in `src/health-check.ts`, Node-importable) does a real DB read (S08.2's `READ_SMOKE_SQL`) + builds & validates a Shape C envelope, returning a **minimal status object** (no regulation data leaked). `src/index.ts` routes `/healthz` to it inside `fetch` (this is the `createDbClient` call site the S08.2 T5 guard was written to lock). `tools/list` stays empty — the S08.1 registry-empty lock holds untouched.
4. **Barrel convention (the AC's explicit decision):** re-export the response types through `src/types/index.ts` with `export type { … } from "./response.js"`, matching the existing `schema.ts` re-export pattern. Recorded in the closure note.

### T3: `output-schema.ts` — zod schemas + drift guards (~5 min)
**Files:** mcp-server/src/output-schema.ts (Create)
**Depends on:** T1, T2
**Action:** Build zod schemas for the embedded shared types and the full response envelope; export `getRegulationsResponseSchema`; drift-guard each schema against its interface so it cannot diverge.
**Details:** `import { z } from "zod";` plus `import type { … } from "./types/response.js"` and `import type { SourceCitation, ClosurePredicate, ReservedPool, DrawSpec, PointSystem, ResidencyCap, ChoiceConfig, AllocationPool } from "./types/schema.js"`. Define shared zod mirrors (closure of what the envelope embeds): `weaponTypeSchema`, `residencySchema`, `geometryRoleSchema` (`z.enum([...])` matching schema.ts unions exactly), `sourceCitationSchema`, `closurePredicateSchema`, `reservedPoolSchema`, `pointSystemSchema`, `residencyCapSchema`, `choiceConfigSchema`, `allocationPoolSchema`, `drawSpecSchema`. Then response schemas: `confidenceSchema = z.enum(["high","medium","low"])`, `coverageSchema`, `warningSchema`, `resolvedSeasonWindowSchema`, `seasonsSectionSchema`, `resolvedTagSchema`, `tagsSectionSchema`, `methodsSectionSchema`, `resolvedReportingObligationSchema`, `reportingSectionSchema`, `contactSchema`, `contactsSectionSchema`, and `getRegulationsResponseSchema` (the exported envelope schema). Nullable fields use `.nullable()`; the meta `schema_version` is `z.literal(2)`.
**Drift guard (the key foundation pattern):** for every zod schema whose output must equal a known interface, add a compile-time assertion that `z.infer<typeof schema>` equals the interface, via a local
`type AssertEqual<A, B> = [A] extends [B] ? ([B] extends [A] ? true : never) : never;`
helper plus `const _assertSourceCitation: AssertEqual<z.infer<typeof sourceCitationSchema>, SourceCitation> = true;` (one per mirrored shared type AND per response type).
**Which idiom where (don't fight the type checker):** prefer the bidirectional `AssertEqual` for **flat schemas with no optional fields** (`sourceCitationSchema`, `warningSchema`, `coverageSchema`, the enum mirrors, and the response sections that have no optionals) — it catches both missing and extra fields. For schemas with **optional fields or nested object composites** — specifically `reservedPoolSchema` (`eligibility` optionals), `allocationPoolSchema` (`tie_break?` + nested `eligibility` optionals), and `drawSpecSchema` (embeds `choiceConfigSchema`/`allocationPoolSchema`) — zod v4's internal optional/`undefined` representation can make bidirectional `AssertEqual` brittle; for those, annotate the schema `… satisfies z.ZodType<Interface>` instead. Do NOT spend time forcing `AssertEqual` where `satisfies` compiles cleanly. Every assertion must hold with zero `any` and **zero `@ts-expect-error`** — if a guard needs `@ts-expect-error` to pass, that is the signal to switch that schema to `satisfies`.
**Verify:** `npm run lint` passes (this is the gate that proves the zod↔interface drift guards hold).
**UI:** no

### T4: `response-builder.ts` — reusable mechanisms (~5 min)
**Files:** mcp-server/src/response-builder.ts (Create)
**Depends on:** T2, T3
**Action:** Implement the reusable serving mechanisms E09/E10 inherit.
**Details:** `import type { GetRegulationsResponse, Warning } from "./types/response.js"; import { getRegulationsResponseSchema } from "./output-schema.js";`. Export:
- `READ_ONLY_TOOL_ANNOTATIONS = { readOnlyHint: true, idempotentHint: true, openWorldHint: false } as const` — the MCP read-only annotation set E09 passes to `registerTool`.
- `SUPPORTED_SCHEMA_VERSIONS: readonly number[] = [2]` — the gating allow-list (response `meta.schema_version` is `2`).
- `STALE_THRESHOLD_DAYS = 180`.
- `buildDataFreshness(sources: { publication_date: string }[], generatedAt: string): { most_recent_source_date: string; stalest_source_date: string; is_stale: boolean }` — find max/min `publication_date` (ISO strings; lexicographic compare is correct for `YYYY-MM-DD`); `is_stale = (Date.parse(generatedAt) - Date.parse(stalest)) / 86_400_000 > STALE_THRESHOLD_DAYS`. Fail loud (throw) on empty `sources` — a freshness block with no contributing source is a builder misuse, not a silent default.
- `gateBySchemaVersion<T extends { schema_version: number }>(rows: readonly T[], supported: readonly number[] = SUPPORTED_SCHEMA_VERSIONS): { included: T[]; warnings: Warning[] }` — partition rows; for each excluded row emit `{ code: "UNSUPPORTED_SCHEMA_VERSION", section: "overall", message: \`schema_version \${row.schema_version} is unsupported (supported: \${supported.join(", ")}); row excluded\` }`. Never throws on an unsupported row (exclude-and-warn, never silent-drop, never hard-error).
- `renderThinText(response: GetRegulationsResponse): string` — a **thin** markdown derivative assembled from structured fields only (e.g. a one-line coverage summary + per-section presence); explicitly NOT a server-composed `overview`/`headline`. Keep it minimal and mechanical.
- `buildStructuredToolResult(response: GetRegulationsResponse): { structuredContent: Record<string, unknown>; content: { type: "text"; text: string }[] }` — validate `response` with `getRegulationsResponseSchema.safeParse(response)`; on failure **throw** an Error naming the zod issue path(s) (the server-MUST-conform obligation); on success return `{ structuredContent: response as unknown as Record<string, unknown>, content: [{ type: "text", text: renderThinText(response) }] }` (the `as unknown as Record<string,unknown>` boundary cast mirrors the established `db.ts` pattern — the only any-free seam to the SDK's `structuredContent: Record<string, unknown>` type). No normalization of `verbatim_rule` or `confidence` anywhere — copy through byte-identically.
**Verify:** `npm run lint` passes.
**UI:** no

### T5: `health-check.ts` — internal end-to-end health check (~4 min)
**Files:** mcp-server/src/health-check.ts (Create)
**Depends on:** T4
**Action:** Implement `runHealthCheck(dsn)` exercising a real DB read + the envelope builder end-to-end, returning a minimal status object.
**Details:** `import { createDbClient, READ_SMOKE_SQL } from "./db.js"; import { buildStructuredToolResult, buildDataFreshness } from "./response-builder.js"; import type { GetRegulationsResponse } from "./types/response.js";`. Export `interface HealthCheckResult { ok: boolean; db_reachable: boolean; envelope_valid: boolean }` and `async function runHealthCheck(dsn: string): Promise<HealthCheckResult>`:
1. `const client = createDbClient(dsn);` (this is the per-request `createDbClient` call site the S08.2 T5 guard locks — it is inside this function, satisfying the guard).
2. `try { const rows = await client.query(READ_SMOKE_SQL, ["US-MT"]); db_reachable = rows.length >= 0; }` (a successful query proves connectivity — `rows.length >= 0` is ALWAYS true even for zero rows; the probe tests **connectivity, not data existence**, so it must not be flipped to `>= 1`, which would falsely fail against an empty substrate) `finally { await client.close(); }`.
3. Construct a minimal hand-built `GetRegulationsResponse` fixture (all sections `null`; `resolved.jurisdiction` `null`; `sources: [<one citation>]`; `meta` built via `buildDataFreshness` + `coverage` all `"none"`; `schema_version: 2`; empty `warnings`), pass it through `buildStructuredToolResult` — `envelope_valid = true` if it does not throw.
4. Return `{ ok: db_reachable && envelope_valid, db_reachable, envelope_valid }`. Catch DB errors into `db_reachable = false` (the health check reports unhealthy; it does not crash the Worker). Do NOT leak regulation data or DSN contents in the result.
