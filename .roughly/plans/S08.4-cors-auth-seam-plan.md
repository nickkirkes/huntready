### T1: Create `src/cors.ts` — pure CORS module (~5 min)
**Files:** mcp-server/src/cors.ts
**Action:** Create a pure module (Web Fetch API types only; NO import of `agents/mcp`, NO import of the `Env` interface) exporting CORS primitives.
**Details:**
Export exactly three functions plus shared header-name/value constants:

1. `buildCorsHeaders(requestOrigin: string | null, allowedOrigins: string | undefined): Headers`
   - Returns a fresh `Headers` object carrying the CORS response headers.
   - **Origin policy:** if `allowedOrigins` is `undefined`, empty string, or the literal `"*"` → set `Access-Control-Allow-Origin: *` (permissive default for the open V1 endpoint). Otherwise parse `allowedOrigins` as a comma-separated list (trim each entry, drop empties); if `requestOrigin` is non-null AND matches a list entry → set `Access-Control-Allow-Origin: <requestOrigin>` AND `Vary: Origin` (per-origin echo); if no match → do NOT set `Access-Control-Allow-Origin` at all (browser will block; this is correct deny behavior) but still set `Vary: Origin`.
   - Always set: `Access-Control-Allow-Methods: GET, POST, OPTIONS`; `Access-Control-Allow-Headers: Authorization, Content-Type, MCP-Protocol-Version, Accept`; `Access-Control-Expose-Headers: Mcp-Session-Id`; `Access-Control-Max-Age: 86400`.
   - **NEVER set `Access-Control-Allow-Credentials`** (MCP uses `Authorization` bearer tokens, not cookies; `*` + credentials is silently rejected by browsers — see Blast Radius). Add a load-bearing comment explaining the omission.
2. `isCorsPreflightRequest(request: Request): boolean`
   - `true` iff `request.method === "OPTIONS"` AND `request.headers.has("Origin")` AND `request.headers.has("Access-Control-Request-Method")`. (A bare OPTIONS without those headers is not a CORS preflight.)
3. `applyCorsHeaders(response: Response, corsHeaders: Headers): Response`
   - Returns a NEW `Response` preserving `response.status`, `response.statusText`, and `response.body` (a `ReadableStream` body — e.g. the MCP SSE stream — is preserved by passing `response.body` straight through), with the original response headers merged first, then the CORS headers layered on top (CORS wins on collision). Implement by constructing a `Headers` from `response.headers` then `corsHeaders.forEach((v, k) => merged.set(k, v))`.
   - Add a comment: this exists because `createMcpHandler()` returns its own `Response` that knows nothing about CORS, so CORS must be re-applied to it (and to `/healthz`/404 responses) on the way out.

Use Web API types throughout (`Request`, `Response`, `Headers`, `URL`). File-level docstring: pure module, no `agents/mcp` import, unit-tested in the Node vitest pool (the established `server.ts`/`health-check.ts` split).
**Verify:** `cd mcp-server && npm run lint` (tsc strict on src + test configs) passes.
**UI:** no

### T2: Create `tests/cors.test.ts` — unconditional CORS unit tests (~5 min)
**Files:** mcp-server/tests/cors.test.ts
**Depends on:** T1
**Action:** Mirror `tests/response.test.ts` structure exactly: `import { describe, it, expect } from "vitest"`, no `describe.skipIf`, no live DB, all tests unconditional (run identically local + CI).
**Details:** Cover, at minimum:
- `buildCorsHeaders(null, undefined)` → `Access-Control-Allow-Origin: *`.
- `buildCorsHeaders("https://app.example.com", "*")` → `*` (explicit star is permissive).
- `buildCorsHeaders("https://app.example.com", "https://app.example.com,https://other.example.com")` → echoes `https://app.example.com` AND sets `Vary: Origin`.
- `buildCorsHeaders("https://evil.example.com", "https://app.example.com")` → NO `Access-Control-Allow-Origin` header (deny), but `Vary: Origin` present.
- `buildCorsHeaders(null, "https://app.example.com")` (configured list, no request Origin) → no `Access-Control-Allow-Origin`.
- **`Access-Control-Allow-Credentials` is null in every case** (the wildcard-vs-credentials trap lock): `expect(headers.get("Access-Control-Allow-Credentials")).toBeNull()`.
- `Access-Control-Allow-Methods` contains `GET`, `POST`, `OPTIONS`.
- `Access-Control-Allow-Headers` contains `Authorization`, `Content-Type`, `MCP-Protocol-Version`.
- `Access-Control-Expose-Headers` contains `Mcp-Session-Id`.
- `isCorsPreflightRequest`: true for OPTIONS + Origin + Access-Control-Request-Method; false for OPTIONS-without-those-headers; false for a GET/POST with Origin.
- `applyCorsHeaders`: status, statusText, and body text preserved; a CORS header from `corsHeaders` is present on the result; a pre-existing non-CORS header on the input response (e.g. `content-type`) survives; on key collision the CORS value wins.

### T3: Create `src/auth.ts` — pure auth-seam module (~5 min)
**Files:** mcp-server/src/auth.ts
**Depends on:** T1
**Action:** Create a pure module (Web Fetch API types only; NO `agents/mcp`, NO `Env` import) exporting the wired-but-unenforced OAuth-2.1 seam primitives.
**Details:**
Export exactly three functions:

1. `isAuthSeamEnabled(authSeamEnabledRaw: string | undefined): boolean`
   - `true` iff `authSeamEnabledRaw === "true"`. Anything else (absent, `undefined`, `"false"`, `"1"`, etc.) → `false`. The **deployed V1 default is disabled** (the secret is unset), so this returns `false` in production. Comment this.
2. `isAuthorized(request: Request, expectedToken: string | undefined): boolean`
   - **Fail-closed:** if `expectedToken` is `undefined` or empty string → return `false` (a seam enabled without a configured credential admits nobody — never silently open). Otherwise return `true` iff the `Authorization` header is exactly `Bearer ${expectedToken}`. No timing-safe compare needed (test fixture, not a V1 boundary — comment this), but use a strict `===`.
3. `buildUnauthorizedResponse(requestUrl: URL, corsHeaders: Headers): Response`
   - Returns a `401` whose `WWW-Authenticate` header is the OAuth-2.1 / RFC 9728 drop-in shape (NOT a bare static-token string — this is the spec-shaped advertisement the AC requires):
     `Bearer realm="HuntReady MCP", resource_metadata="${requestUrl.origin}/.well-known/oauth-protected-resource"`.
     The `resource_metadata` URI is **derived from `requestUrl.origin`, never hardcoded** (the well-known endpoint itself is NOT built in S08.4 — it's the Q22 drop-in; only the pointer is advertised).
   - Merge `corsHeaders` onto the 401 (a browser must be able to read the 401). Set `content-type: application/json`; body a small JSON object like `{ "error": "unauthorized", "error_description": "Authentication required (auth seam enabled)." }`.
   - Module/function docstring states explicitly: **the credential is a test fixture, not a V1 enforcement boundary** — the deployed V1 endpoint is open (seam disabled); on a single open endpoint a token is unenforceable. `@cloudflare/workers-oauth-provider` is the named drop-in for real enforcement (Q22), NOT built here.

Web API types throughout. Do NOT install `@cloudflare/workers-oauth-provider` or import `express`-based SDK auth routers (workerd-incompatible).
**Verify:** `cd mcp-server && npm run lint` passes.
**UI:** no

### T5: Wire CORS + auth seam into `src/index.ts` (~5 min)
**Files:** mcp-server/src/index.ts
**Depends on:** T1, T3
**Action:** Extend the inline `Env` interface and rewrite the `fetch` handler to the 4-step dispatch with CORS applied to every response. Keep `index.ts` a thin orchestrator (the testable logic lives in `cors.ts`/`auth.ts`).
**Details:**
- Add imports: `buildCorsHeaders`, `isCorsPreflightRequest`, `applyCorsHeaders` from `./cors.js`; `isAuthSeamEnabled`, `isAuthorized`, `buildUnauthorizedResponse` from `./auth.js`.
- Extend `Env` with three OPTIONAL fields (comment each, matching the existing `HEALTHCHECK_TOKEN` comment style):
  - `CORS_ALLOWED_ORIGINS?: string` — comma-separated allowed origins; unset = `*` (permissive V1 default; M4 tightens).
  - `AUTH_SEAM_ENABLED?: string` — `"true"` enables the wired-but-unenforced auth seam; unset/absent = disabled (V1 deployed default).
  - `AUTH_SEAM_TOKEN?: string` — the test-fixture bearer credential checked when the seam is enabled; a Workers Secret, never committed. NOT a V1 enforcement boundary (the deployed endpoint is open).
- New `fetch` body, in this EXACT order:
  1. Compute `const requestOrigin = request.headers.get("Origin");` and `const corsHeaders = buildCorsHeaders(requestOrigin, env.CORS_ALLOWED_ORIGINS);` once at the top.
  2. **CORS preflight short-circuit (must be first, before auth):** `if (isCorsPreflightRequest(request)) return new Response(null, { status: 204, headers: corsHeaders });` (204 No Content; preflight carries no credentials and must NOT be 401'd).
  3. **Auth seam (after preflight):** `if (isAuthSeamEnabled(env.AUTH_SEAM_ENABLED) && !isAuthorized(request, env.AUTH_SEAM_TOKEN)) return buildUnauthorizedResponse(new URL(request.url), corsHeaders);`
  4. **`/healthz` block** (unchanged logic) — but wrap each returned `Response` with `applyCorsHeaders(resp, corsHeaders)`.
  5. **MCP handler** — `const response = await handler(request, env, ctx); return applyCorsHeaders(response, corsHeaders);` (re-apply CORS to the handler's own Response).
- Preserve all existing comments/behavior (per-request `createMcpServer()`, the healthz gate semantics, the no-DB-on-disabled-healthz posture). Do NOT introduce `any`. Do NOT reference `protocolVersion` (boundary.test.ts Test 4 forbids it). Keep `createMcpServer()` and `createDbClient()` (if any) inside the handler function (boundary.test.ts Tests 3 + 5).
**Verify:** `cd mcp-server && npm run lint && npm run test` — full suite green, baseline grew additively, boundary guards (Tests 1–7) still pass.

## Blast Radius

- **Do NOT modify:** `src/server.ts`, `src/db.ts`, `src/health-check.ts`, `src/response-builder.ts`, `src/output-schema.ts`, `src/types/**`, `tests/boundary.test.ts` (its existing `collectTsFiles(srcDir)` recursive walk in Test 2 + Test 5 auto-covers the new `src/cors.ts` + `src/auth.ts` — AC #4 is satisfied with NO new guard test; document this in the closure note), `tests/server.test.ts`, `tests/db.test.ts`, `tests/health-check.test.ts`, `tests/response.test.ts`.
- **Do NOT touch:** anything under `ingestion/`, `supabase/migrations/`, `web/`, `.github/workflows/ci.yml` (CORS + auth-seam tests must pass with DEFAULTS — no new CI secret required; the seam-enabled test sets its own toggle/token inline, the seam-disabled state is the default).
- **Do NOT install** `@cloudflare/workers-oauth-provider`; do NOT adopt the transitively-present `hono` or `cors` packages as direct deps (Node-only / oversized for ~50 lines).
- **Watch for:**
  - **Wildcard-vs-credentials CORS trap** — never set `Access-Control-Allow-Credentials: true`; MCP uses bearer tokens, not cookies; `*` + credentials is silently rejected by browsers. Locked by a T2 test.
  - **OPTIONS preflight must short-circuit BEFORE the auth seam** — rejecting an unauthenticated preflight with 401 violates RFC 6454 and breaks every browser MCP client.
  - **CORS on every response** — the `createMcpHandler()` response, the 401, `/healthz`, and 404 all need CORS re-applied via `applyCorsHeaders`.
  - **Pure modules must not import `agents/mcp` or the `Env` interface** — that would transitively pull workerd-only code and break the Node vitest pool. Pure functions take narrow primitive/Web-API args.
  - **Base tsconfig types = `@cloudflare/workers-types` only** — `src/cors.ts` + `src/auth.ts` use Web API types (`Request`/`Response`/`Headers`/`URL`), never Node-only types.
  - **detect-secrets** — if any test bearer-token literal trips the scanner, annotate with `# pragma: allowlist secret` and refresh `.secrets.baseline` (repo root) line numbers as a routine commit.
