## §1 Edge-case catalogue

Ten cases surfaced during the S03.11 discovery phase. Each case maps to one or more working notes in `E03-confidence-findings/` (all delete at m1 tag per ADR-017 §6; this synthesis file survives). Cases are ordered by their influence on `regulation_record.confidence`, with confidence-irrelevant cases (#6, #9) included for completeness.

---

**1. DEA species fan-out (deer → mule_deer + whitetail).** A single DEA `deer` section produces two `regulation_record` rows with the same MIN-aggregated confidence. The fan-out loop in `_build_dea_records` (`load_regulation_records.py:315`) emits `mule_deer` first, `whitetail` second, both sharing the same `confidence` value captured before the loop at `:306-310`. This is a structural guarantee: the same MIN result is assigned to both fan-out targets without a separate re-computation. V1 produces 129 mule_deer + 129 whitetail = 258 rows, all `medium` (no deer section is all-HIGH because every deer section contains at least one prose-only General Deer License row, which is MEDIUM per the HD-coded/prose-only rule in `_assign_row_confidence`).

The fan-out symmetry is load-bearing for audit reproducibility. S03.6.md documents that the MIN over per-row extraction tiers is computed once per DEA section (`load_regulation_records.py:306-310`) before the `for target_species_group in target_species:` loop at `:315`. Both fan-out records share an identical `confidence` variable; any change that adds a per-species re-computation would silently break parity and would need a test to detect it. The test `TestBuildDeaRecords::test_deer_fanout_confidence_symmetry` in `test_load_regulation_records.py` locks this guarantee. The S03.11 audit's EC1 rows (sample rows 1–5) verify the mule_deer fan-out side; the whitetail side is structurally identical and confirmed by the symmetry guarantee alone. Working note: `S03.6.md` (§ "Deer fan-out symmetry guarantee" and § "F12 per-record trace table").

**2. DEA MIN-tier aggregation (high vs. medium within a section).** Elk sections commonly mix HIGH (HD-coded B License rows, e.g., `Elk B License: 124-00`) and MEDIUM (prose-only General Elk License rows). `pdf.ConfidenceTier.min_tier` aggregates across all rows in the section and returns the most uncertain tier. V1 elk result: 2 sections of 112 emit `high`; 110 emit `medium`. No antelope sections are mixed (all antelope rows within a given numeric-HD section are uniformly HIGH — only the STATEWIDE row is MEDIUM, and it is a 1-row section). The lexicographic-min trap (`min(["high","low","medium"])` → `"high"` because h<l<m) is explicitly guarded by `test_confidence_min_tier_trap_case_high_low_returns_low` in `tests/test_load_regulation_records.py`.

The per-row confidence rule (documented in S03.3.md's "Confidence assignment per ADR-017" table) distinguishes two table-source paths: (a) a cell whose `license_code` contains an HD-code pattern (`\b\d{3}-\d{2}\b`, matched via `re.search` not `re.match` because the actual cell form is compound prose+code like `"Elk B License: 124-00"`) yields HIGH; (b) a prose-only license name like `"General Elk License"` (no embedded HD code) yields MEDIUM. The full-string-match form (`re.match(r"^\d{3}-\d{2}$", ...)`) would incorrectly rate all 678 compound-form cells as MEDIUM. S03.6.md's cross-tab (S03.6's first dry-run, 2026-05-14) confirms the V1 outcome at the regulation_record level: elk has 2 `high` + 110 `medium`; pronghorn 30 `high` + 1 `medium`; all other species uniformly `medium`. The two all-HIGH elk sections are HD 250 (three HD-coded rows: Elk B License 250-00, Elk Permit 250-45, Elk Permit 250-46) and HD 622 — the only two V1 elk HDs with no prose-only General Elk License row. Working notes: `S03.3.md` (§ "Confidence assignment per ADR-017") + `S03.6.md` (§ "Per-record aggregation outcomes" and § "F12 per-record trace table").

**3. Correction-touched demote-one-tier (bear).** All 35 bear BMU rows are tagged `applied_correction=True` in `black-bear-2026.json` because S03.4's `_merge_with_corrections` touched every BMU's `hound_training_season` cell. ADR-017 §4 mandates a single-step demotion (HIGH → MEDIUM, floor-clamped); `demote_one_tier` is called exactly once per BMU (post Stage 2 of the merge algorithm, locked by `TestMergeWithCorrections::test_demote_fires_exactly_once_per_bmu`). The result is that ALL 35 bear rows carry `confidence="medium"` in the artifact and are passed through unchanged by `load_regulation_records.py`'s direct-pass-through path at `:405-406`.

S03.4.md's "Confidence Assignment Patterns" table records the cause: base BMU table rows from the structured rotated table would have been HIGH without any correction (deterministic column-position mapping, pdfplumber table extraction, no manual interpretation). The 2026-03-18 Black Bear correction PDF rewrote the `hound_training_season` field for all 35 BMUs, propagating `correction_touched=True` to every row. The demotion must fire exactly once per BMU — not once per field touched. The original implementation of `_merge_with_corrections` incorrectly re-applied `demote_one_tier` once per corrected field within a BMU; a cubic-review fix in Stage 3 of S03.4 refactored into three sequential stages (Stage 1 collects fields to update; Stage 2 applies field updates; Stage 3 calls `demote_one_tier` once per touched BMU). A test `TestMergeWithCorrections::test_demote_fires_exactly_once_per_bmu` locks this constraint. This edge case is the direct downstream consequence of edge case #5 (ADR-019 date-arbitration): the correction document wins via doc-type-precedence, and the demotion then fires as mandated by ADR-017 §4. Working notes: `S03.4.md` (§ "Confidence Assignment Patterns" and § "Confidence-assignment inputs (F12 sample pairs)") + `S03.6.md` (§ "Per-record aggregation outcomes").

**4. Statewide overlay anchor rows.** Two statewide `regulation_record` rows exist in V1: `MT-STATEWIDE-antelope` (DEA section `hd_number="STATEWIDE"`, emitted via `is_statewide_overlay=True` flag in `extract_dea.py`; confidence=`medium` from the single MEDIUM row `Antelope License: 900-20`) and `MT-STATEWIDE-bear` (S03.6.1 anchor, extracted from page-2 right-column of the bear booklet; confidence=`medium` because all bear rows are correction-demoted). Both are flagged per ADR-018 §3 flag-and-surface; no additional statewide candidates were surfaced during V1 implementation.

The `MT-STATEWIDE-antelope` path is confidence-driven: the `900-20` row contains an HD-code-pattern string (`900-20`) that would have yielded HIGH under the standard `"table"` source branch, but the orchestrator in `extract_dea.py` forces `source="prose"` for statewide overlay rows (`extract_dea.py:1120-1125`), overriding the structured-table-cell signal. ADR-017 §4's prose-source rule then mandates MEDIUM, and the MIN over a 1-element list is identity — so the resulting `MT-STATEWIDE-antelope` record carries `confidence="medium"`. S03.6.md § "MT-STATEWIDE-antelope path" documents this explicitly as a deliberate policy override, not an extraction weakness.

The `MT-STATEWIDE-bear` path (S03.6.1) is confidence-driven differently: the row is extracted from page-2 right-column prose via regex anchors `_BEAR_ID_START_RE` and `_BEAR_ID_END_RE`, which is heading-anchored prose extraction — MEDIUM per ADR-017 §4 regardless of correction state. Critically, the Bear ID Test paragraph lives on page 2 of the booklet, which the 2026-03-18 correction PDF did not modify; therefore `correction_touched=False` for this row despite the parent document being correction-touched at the BMU-table level. S03.6.1.md § "Confidence-assignment inputs (F12 sample pairs)" records this `correction_touched=False` determination explicitly. Working notes: `S03.6.md` (§ "MT-STATEWIDE-antelope path" and § "Additional statewide candidates surfaced") + `S03.6.1.md` (§ "Confidence-assignment inputs (F12 sample pairs)").

**5. Correction PDF date-arbitration (doc-type-precedence rule, ADR-019).** Bear rows cite the correction document (`source_id="mt-fwp-black-bear-2026-correction-2026-03-18"`, `document_type='correction'`, `publication_date='2026-03-18'`) rather than the base booklet (`publication_date='2026-04-27'`). The booklet's later `Last-Modified` date would have won under a naive MAX-date rule (Option A), silently no-op-ing the correction. ADR-019 (doc-type-precedence) was accepted at S03.4 close to formalize that `document_type='correction'` always defeats `'annual_regulations'` regardless of dates; date is tiebreaker only within the same document type. This is the root cause of edge case #3 — correction-priority wins, then demote-one-tier fires.

S03.4.md § "Date-Arbitration Rule: Option B (doc-type-precedence), Not Option A" explains the causal chain. A printed correction is distributed before the booklet goes to press: the correction was finalized in March 2026 and the booklet in April 2026, so the chronological order is a deliberate feature of FWP's publication workflow (corrections precede press-final versions), not a sign the correction is stale. Option A (naive MAX-date) would have inverted operator intent, with the correction — which self-identifies as "Corrections to the 2026 Printed Black Bear Regulations" — being silently suppressed in favor of the later-dated booklet it was written to amend. Option B is the only defensible read. The locking test is `TestMergeWithCorrections::test_real_world_option_b_correction_wins_despite_earlier_date` in `tests/test_extract_black_bear.py`, which asserts that correction content wins even when the booklet has a later `publication_date`. Edge case #5 is therefore both an ADR-019 design record and the prerequisite trigger for edge case #3's demotion. Working note: `S03.4.md` (§ "Date-Arbitration Rule: Option B (doc-type-precedence), Not Option A").

**6. Per-row window divergence (season-level faithfulness).** DEA sections containing multiple license rows can have each row carry distinct `season_windows` (e.g., the three rows in HD 170 elk have different season-type key sets). This was the UAT D1 defect (2026-05-08): the original extractor collapsed all rows in a section to the first row's windows. The fix deleted `_aggregate_section_season_windows` and rewrote row emission to read each row's own season-column cells inline. Per-row window divergence does NOT affect `regulation_record.confidence` (confidence is aggregated at the section level, not the window level). It is relevant to S03.11 because `season_definition` and `license_season` rows derived in S03.7 depend on these per-row windows being faithful.

S03.3.md § "UAT fix log (2026-05-08)" records the root cause: the original `_aggregate_section_season_windows` helper computed section-level windows via first-observation-wins, then `_row_season_windows` filtered them per `season_coverage`. Real PDF rows have per-row date variation — HD 124 deer has three distinct General Season windows (`Oct 24-Oct 30`, `Oct 24-Nov 29`, `Oct 31-Nov 29`) across its five rows; the artifact before the fix carried only the first observation across all five. The wrong assumption, per S03.3.md § "Wrong assumptions in the original implementation," was: "section-level windows are sufficient because all rows in an HD share the same date structure." ADR-008 requires the per-row variance be preserved verbatim. The confidence audit is not impacted by this edge case because the MIN aggregation operates over the per-row `extraction_confidence` tier values (HIGH or MEDIUM, driven by the license_code form), not over window consistency. A section where all rows have identical windows produces the same MIN result as a section with maximally divergent windows, as long as the per-row tier assignments are unchanged. Working note: `S03.3.md` (§ "UAT fix log (2026-05-08)" → D1 defect + fix).

**7. Region 7 portions sub-sections (antelope multi-HD emission).** DEA antelope sections for HDs 700-705 are emitted from "Portions of HDs..." rows on the antelope pages. The S03.3 D2 fix (UAT-discovered) added `_parse_portions_hd_list` and `_is_portions_subsection_row` to walk pages directly. HDs 701 and 703 appear in BOTH the North Yellowstone and South Yellowstone portions sub-sections, so they are emitted twice — once per sub-section. The resulting regulation_record rows for these HDs carry `confidence="high"` (both portions-emitted rows for a given HD have uniform HIGH tiers; no MEDIUM rows in the R7 antelope sections). Fan-out to pronghorn is identical to other antelope sections.

The D2 defect is documented in S03.3.md § "UAT fix log (2026-05-08)": the original antelope orchestrator used `_iter_hd_sections` + `_extract_hd_table`, which only stopped at HD-numbered headers. Region 7's `"Portions of HDs A, B, C ... [directional qualifier]"` sub-sections were absorbed into the preceding HD-numbered section's data rows — HD 690 carried 4 phantom rows (`007-21`, `007-31`, `007-20`, `007-30`); Region 7 portions sub-sections were absent entirely. The fix (option (a) per the UAT directive) rewrites the antelope orchestrator to walk pages directly with three delimiter types: `HD NNN - Name`, `Portions of HDs ... [qualifier]`, and `Region N` (skipped). For Portions sub-sections, one `DeaSectionExtraction` is emitted per listed HD with the same row content; the license codes (`007-21`, `007-31`, etc.) disambiguate at S03.6 ingestion. HD 701 and HD 703 each appear in two sub-sections (North Yellowstone and South Yellowstone), producing two emission events per HD — both all-HIGH (no prose-only rows exist in R7 antelope portions sections), so both produce `confidence="high"` at the regulation_record level. The duplicate artifact-layer entries resolve at the DB layer via `ON CONFLICT DO NOTHING` on the link tables. Working note: `S03.3.md` (§ "UAT fix log (2026-05-08)" → D2 defect + fix, and § "Live artifact (post-UAT-fix)").

**8. Bear closure_predicate rows (sex_threshold + quota_threshold).** The bear artifact's `closures` list (top-level key, not per-row) carries two `closure_predicate` records: (a) `kind="sex_threshold"` covering BMUs 300, 301, 319, 580 (spring season closes upon exceeding 37% female harvest); (b) `kind="quota_threshold"` covering BMUs 411, 420, 440, 450, 510, 520, 600, 700 (season closes upon quota reached). These are child entities in S03.7's sense — they inherit parent `regulation_record.confidence` per ADR-017 §3 parent-inheritance rule (no separate `confidence` column on `closure_predicate`). The closure entities are indexed to bear regulation_record rows by BMU number. This is a child entity, inheriting parent confidence per ADR-017 §3 parent-inheritance rule; no audit signal exists on this entity itself.

S03.7.md § "11 plan decisions baked into the implementation" documents OQ-S7-5 and OQ-S7-6 (the closure_predicate attachment decisions): `sex_threshold` attaches to the spring season only (BMUs 300, 301, 319, 580; spring closure fires upon exceeding 37% female harvest); `quota_threshold` attaches to both `general_season` AND `archery_only_season` (BMUs 411, 420, 440, 450, 510, 520, 600, 700; quota-closure fires upon reaching the fall quota regardless of weapon type). The 20 total `closure_predicate` rows break down as: 16 `quota_threshold` across 8 BMUs × 2 fall seasons; 4 `sex_threshold` across 4 BMUs × 1 spring season. All 12 BMUs carrying closure predicates are correction-touched bear BMUs, so their parent `regulation_record.confidence` is `medium` (via edge case #3's demotion path). The inherited confidence value of `medium` is what an auditor attributes at query time — it is not a stored column on `closure_predicate`. Working note: `S03.7.md` (§ "11 plan decisions baked into the implementation" → OQ-S7-5 and OQ-S7-6, and § "Actual output counts (2026-05-15 dry-run)").

**9. B License cross-listing / OTC reclassification.** 162 B License `license_tag` identities were reclassified from `limited_draw` to `over_the_counter` in S03.8's T1 amendment to `_DEA_LICENSE_KIND_HEURISTIC`. The amendment inspects the `apply_by` column cross-row and cross-HD: an `apply_by` cell prefixed with `OTC:` (e.g., `"OTC:\nJun 15"`) is the OTC discriminator. Without this amendment, `draw_spec` rows would have been over-written for OTC-classified licenses (silent data corruption). This edge case affects `license_tag.kind` only — it does NOT affect `regulation_record.confidence`. Included for audit-traceability completeness as a cross-cutting correctness risk.

S03.8.md § "OTC discovery (Stage 2 probe finding)" documents the mechanism: S03.7's original 5-branch `license_tag.kind` heuristic inspected only `license_code` (any code matching the `B License: DDD-DD` pattern was classified `limited_draw`), without consulting `apply_by`. The Stage 2 probe found 162 DEA B License identities with `apply_by` cells reading `"OTC:\nJun 15"` — over-the-counter purchases with an application window, not lottery draws. The OTC-wins discipline keys by `license_code` alone (not by `(species, hd, license_code)` 3-tuple): any `license_code` appearing in at least one artifact row anywhere with `"OTC"` in `apply_by` reclassifies ALL cross-HD instances as `over_the_counter`. S03.8's Phase 1 re-upsert writes the corrected `kind` to existing DB rows via `upsert_license_tag`'s `kind = EXCLUDED.kind` clause. The corrected post-amendment breakdown is: 388 `limited_draw` + 162 `over_the_counter` + 239 `general` + 1 `statewide` = 790 unique DEA license_tag identities. The 388 `limited_draw` identities become S03.8's draw_spec population; OTC identities correctly receive no draw_spec rows. Working note: `S03.8.md` (§ "OTC discovery (Stage 2 probe finding)" and § "S03.7 amendment (T1) summary").

**10. HD-coded vs. prose-only license confidence distinction.** `_assign_row_confidence` in `extract_dea.py` assigns `HIGH` when `license_code` matches `\b\d{3}-\d{2}\b` (i.e., a hunt-code-keyed permit such as `Elk B License: 124-00`) and `MEDIUM` otherwise (prose-only license names such as `General Elk License`). This is the single most impactful rule: 601 HIGH rows and 589 MEDIUM rows in `dea-2026.json` (post-UAT-fix artifact). The rule is locked by `TestRowConfidenceAssignment` in `tests/test_extract_dea.py`. The subsequent MIN-aggregation at the section level means a single MEDIUM row in any section pulls the entire section — and both fan-out regulation_record rows — to MEDIUM.

S03.3.md § "Confidence assignment per ADR-017" provides the full decision table and explains why `re.search` (substring match) rather than `re.match` (full-string match) is correct: the actual DEA cell form is compound prose+code (`"Deer B License: 124-00"`) in 678 of 1190 rows; a full-string match would have incorrectly rated all 678 compound forms as MEDIUM. The HD code is a primary key linking the `regulation_record` row to the section's HD, making HD-coded cells objectively higher-quality structured data than prose-only cells. S03.3.md § "Live distribution at S03.3 post-dedup-fix" confirms 0 LOW rows — every row had a non-empty opportunity and every HD-coded row had at least one True season_coverage flag, so neither of the LOW-tier conditions in `_assign_row_confidence` fired on V1 data. At the `regulation_record` level, this rule produces the ~32 high / ~405 medium / 0 low population shown in §3.1's tier distribution table. The only HIGH regulation_record rows in V1 are antelope HDs with all-HD-coded rows (no prose-only General Antelope License rows in those sections) and the 2 elk sections (HD 250, HD 622) where every row is HD-coded. Working note: `S03.3.md` (§ "Confidence assignment per ADR-017", § "Why `re.search` (not `re.match`) for the HD-code pattern", and § "Live distribution at S03.3 post-dedup-fix").

## §2 Stratified audit sample

**Sort key:** All non-exception edge cases select the first 5 rows sorted by numeric HD number (or BMU number for bear) ascending. Where an edge case spans multiple artifact types (DEA vs. bear), bear BMU numbers are treated as comparable integers. This yields a deterministic, reproducible sample.

**Exceptions to the 5-row rule:**
- **Edge case #4 (statewide overlays):** Only 2 rows exist in V1 (`MT-STATEWIDE-antelope` and `MT-STATEWIDE-bear`). Both are included. Full population audited.
- **Edge case #8 (bear closure_predicate):** 12 affected BMUs split across 2 subtypes (4 sex_threshold + 8 quota_threshold). Sample is stratified: 3 of each subtype = 6 rows.

## §3 Audit results (T6)

**Methodology:** Each row is verified in strict 4-step order: (1) pull F12 inputs from the retrofitted working notes without reading artifact confidence; (2) apply ADR-017 §4–§5 rules independently to derive `framework_predicted`; (3) reveal `assigned_confidence` from the artifact; (4) compare. This order is load-bearing — reading the artifact value first would make the audit a tautology.

**ADR-017 rules applied (independent derivation):**
- Structured DEA table cell, HD-coded license_code (`\b\d{3}-\d{2}\b` match), non-empty opportunity, ≥1 True season_coverage, correction_touched=False → **HIGH** (row-level)
- Structured DEA table cell, prose-only license_code (no HD code), non-empty opportunity, correction_touched=False → **MEDIUM** (row-level)
- Statewide overlay row with `source="prose"` forced by orchestrator → **MEDIUM** (row-level; override of HD-code signal)
- Heading-anchored prose extraction (regex-anchored, bbox-cropped) → **MEDIUM** (row-level)
- Section-level MIN across contributing per-row tiers → regulation_record.confidence (ADR-017 §5)
- Correction-touched: demote one tier post-extraction (HIGH→MEDIUM, floor-clamped at MEDIUM per `pdf.py:141`) → applied before section-level MIN
- Child entity (closure_predicate) with no confidence column: inherits parent regulation_record.confidence per ADR-017 §3

**Notes on EC8 and EC9:**
- EC8 closure_predicate rows carry no stored `confidence` column. The "assigned_confidence" is the value an auditor attributes via ADR-017 §3 parent-inheritance; the parent regulation_record confidence is the ground truth.
- EC9 rows are audited at the individual row level (row-level extraction_confidence), not at the regulation_record.confidence level, because EC9's audit question is whether the row-level tier was correctly assigned before section-level MIN aggregation.

## §5 ADR-017 §7 trigger analysis (T8)

ADR-017 §7 defines three OR-ed deferral triggers. If any one fires, the affected section of the ADR moves to "deferred to M2" with a documented reason. All three are evaluated independently below.

---

### §5.1 Trigger 1 — Any documented edge case maps to >1 tier without contradiction

**DOES NOT FIRE.**

All 10 edge cases in §1 map deterministically to a single `regulation_record.confidence` tier under ADR-017's rules; none produces an unexplained split. The apparent multi-tier behavior in several edge cases is the MIN-aggregation rule operating correctly, not a contradiction:

- **EC2 (DEA MIN-tier aggregation):** Elk sections that mix HIGH (HD-coded B License rows) and MEDIUM (prose-only General Elk License rows) resolve to a single section-level confidence via `ConfidenceTier.min_tier`. The two all-HIGH sections (HD 250 and HD 622) and the 110 mixed-tier MEDIUM sections are fully explained by per-section row composition — which rows exist in each section is a data property, not a framework ambiguity. The framework's rule is: most-uncertain wins; the output is a single tier per `regulation_record`.
- **EC7 (Region 7 portions multi-HD emission):** HD 701 and HD 703 are emitted twice at the artifact layer (once per sub-section: North Yellowstone and South Yellowstone). Both emissions for each HD produce `high`; there is no within-HD tier split. The artifact-layer duplicate resolves to a single DB row via `ON CONFLICT DO NOTHING`.
- **EC4 (statewide overlay rows):** Both V1 statewide records (`MT-STATEWIDE-antelope`, `MT-STATEWIDE-bear`) produce `medium` by distinct but non-contradictory paths — orchestrator-forced `source="prose"` override and heading-anchored prose extraction, respectively. Each path yields a single tier; no contradiction exists between them.
- **All other edge cases (EC1, EC3, EC5, EC6, EC8, EC9, EC10):** Each maps to a single tier by a single deterministic rule (fan-out symmetry, correction-touched demotion, per-row window faithfulness, parent-inheritance, OTC reclassification, HD-code regex). None produces competing tier assignments.

The §3 audit confirms this empirically: all 50 sampled rows have `framework_predicted == assigned_confidence` with zero non-matches. Trigger 1 requires a documented edge case that maps to >1 tier _without_ the rules explaining the split; no such case exists.

---

### §5.2 Trigger 2 — Any tier has 0 rows in Montana data

**FIRES under the recommended LITERAL reading; does NOT fire under the alternative INTENT reading.**

The §3.1 tier distribution is unambiguous: `high=32, medium=405, low=0, total=437`. The `low` tier has zero rows in V1 Montana data. This is the live trigger condition per ADR-017 §7's text (lines 74-80): "Any tier has 0 rows in Montana data."

**LITERAL reading (recommended):** The ADR text contains no "by design" carve-out for the Trigger 2 condition. The text says "any tier has 0 rows in Montana data" without qualification; PM does not have interpretive license to read one in silently. Under this reading, `low=0` trips the trigger regardless of cause. ADR-017 §7's partial-defer mechanism ("wholly or partially") applies: only ADR-017's §4 LOW tier definition is the affected section. The LOW calibration — specifically, whether V1 Montana data correctly confirms the LOW-tier rule fires on appropriate inputs — is marked "deferred to M2." The rest of the framework (§1 inherited-confidence rule, §2 spatial carve-out, §3 per-VerbatimRule confidence, §4 HIGH and MEDIUM definitions, §5 MIN-aggregation) is unaffected and ships with the `m1` tag as-is. Per ADR-017 §7: "the ADR's affected section moves to 'deferred to M2' with a documented reason."

**INTENT reading (alternative):** The `low` tier has 0 rows in V1 Montana data because no V1 Montana extraction input satisfies the LOW-tier conditions defined in ADR-017 §4 and implemented in `_assign_row_confidence`: empty `license_code`, empty `opportunity`, or all-False `season_coverage`. Montana's DEA data is structured, well-formed, and consistent; its correction PDF touches only structured-table-cell rows (which start at HIGH, not MEDIUM). The MEDIUM→LOW demote path is not latent or broken — it is unit-tested in `TestDemoteOneTier` (`tests/test_pdf.py`) and its behavior is locked. Under this reading, "absent-by-data" is not the same as "absent-by-framework-gap": the rule exists, is implemented, and is tested; V1 data simply doesn't have inputs that reach it. Treating absence-by-data as a trigger would force framework amendments after every well-behaved dataset, which is not the spirit of ADR-017 §7's calibration mechanism.

**PM recommendation: LITERAL reading.** Four reasons:

1. The ADR text is the canonical source of truth in this project (per ADR-001: authority preserved). Reading qualifications into ADR-017 §7's trigger language that the text does not contain is interpretive license PM should not take unilaterally, especially when the user may have written the trigger language precisely to catch this case.
2. The partial-defer mechanism exists for exactly this situation. ADR-017 §7's "wholly or partially" language is designed to allow surgical deferral — flag the gap, ship the framework operationally intact, schedule M2 verification. The resulting amendment is light-touch (one-paragraph addendum to §4 acknowledging that LOW-tier calibration is pending M2 data) and does NOT change V1 operational behavior.
3. M2's Colorado dataset is expected to exercise the LOW-tier rule. Colorado's richer GMU + species + license-type matrix (CO GMU data is more granular, CO draw system has more license types, and CO regulations PDF structure may include rows with genuinely absent `opportunity` or all-False `season_coverage`) increases the likelihood of corner-case rows. The amendment lands calibrated on real data, not on unit tests alone.
4. Respecting the audit-trail-grade discipline of this project means accepting trigger outcomes even when the underlying cause is benign. The trigger fired; the process calls for a partial defer; that is what PM recommends.

The INTENT reading is defensible — it is not wrong — and the user has full authority to override to INTENT at review time. §6 makes the override path explicit.

---

### §5.3 Trigger 3 — Audit pass-rate <80%

**DOES NOT FIRE.**

The §3 audit returned **50 matches out of 50 sampled rows: pass-rate = 50/50 = 100%.** The threshold is 80%; 100% is comfortably above it. The math: 50 ÷ 50 = 1.000 = 100%, which is ≥ 0.80. Trigger 3 requires a pass-rate strictly below 80% — i.e., at most 39 matches out of 50. With 50 matches and 0 non-matches, Trigger 3 does not come close to firing. §4.2 documents the zero-non-match outcome and notes this is expected given that the framework is internally consistent on V1 Montana data.

## §6 Verdict — finalize or defer (T8)

- ADR-017 §2 (no inherited confidence for spatial entities): unchanged.
- ADR-017 §3 (per-VerbatimRule confidence): unchanged.
- ADR-017 §4 HIGH and MEDIUM tier definitions: unchanged.
- ADR-017 §5 (MIN-aggregation rule): unchanged.

The synthesis report's verdict here is PM's recommendation, not a commitment.
