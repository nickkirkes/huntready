## 1. Source choice

**Chosen source:** Montana State Library — MSDI Framework Boundaries, Layer 9.

**Why NOT MT FWP:**
Both the on-prem catalog (`https://fwp-gis.mt.gov/arcgis/rest/services?f=json`) and the ArcGIS Online org (`MtFishWildlifeParks`) were checked per the `.roughly/known-pitfalls.md` "MT FWP services" entry. Neither exposes a state-boundary layer. FWP publishes hunting districts, portions, restricted areas, and CWD zones — not the state outline itself. No credible FWP state-boundary endpoint was found.

**Why NOT TIGER 2020:**
TIGER is reachable (`https://www2.census.gov/geo/tiger/TIGER2020/STATE/tl_2020_us_state.zip`), but MSDI is strictly better on three grounds:
- State-published source (Montana State Library) rather than federal Census.
- GCDB-aligned at 1:24,000 — higher precision than TIGER's 1:100,000 state lines.
- Qualifies as `document_type='gis_layer'` (an ArcGIS FeatureServer query) with no Literal extension needed. TIGER would require either a semantic shoehorn (`'gis_layer'` on a Census shapefile) or adding `'reference_boundary'` to `SourceCitation.document_type` — an unnecessary schema change.

**Schema-Literal extension status:** The `'reference_boundary'` extension described as the plan default in T7 was NOT applied. `document_type='gis_layer'` covers the MSDI source cleanly per ADR-014. If a future state has no GIS-published boundary and falls back to TIGER, that story should reopen this question.

**Investigated:** 2026-05-03. Decision: implementer (S03.0 agent).

An operator can re-verify the SHA at any time:

```bash

  | shasum -a 256

Expected output (the SHA above, followed by `-`):

```
d90b40c8f290e8fbcd83c30e89c4edf645adb73f2a3e367b29b8637ee6f79d34  -

## 4. Load command

```bash
cd ingestion && DATABASE_URL=<your-direct-postgres-url> .venv/bin/python states/montana/load_state_boundary.py
```

`DATABASE_URL` is a direct Postgres connection string (not the Supabase REST URL). The script writes one `geometry` row (`id=MT-STATEWIDE-geom`, `kind='state'`) and commits atomically.

## 5. SHA drift policy

The URL is a live MapServer query, not a versioned static file. If the Montana State Library updates the MSDI Boundaries layer, the next curl will return different bytes and the SHA will drift. The load script will fail loud (`RuntimeError` with both observed and expected SHAs).

**When SHA drift occurs — operator must:**
1. Download the new payload and inspect the geometry (rough bbox check: longitude −116.05 to −104.04, latitude 44.36 to 49.0; area ~380,800 km²).
2. Validate it covers Montana correctly (compare to the existing DB row if one exists).
3. Update `STATE_BOUNDARY_SHA256` in `ingestion/states/montana/load_state_boundary.py`.
4. Update the SHA in this file (section 2 above).
5. Re-run the load command.

Do NOT bypass the SHA check (no `# noqa`, no hardcoded override, no silencing the RuntimeError). The check exists precisely because the live-query URL has no version pin.

## 6. Live-load gate

The story AC "one `geometry` row written with `kind='state'`" is satisfied by an operator running the load command in section 4 above with `DATABASE_URL` set. **CI does not satisfy this AC.** CI verifies script correctness via mocked HTTP and DB in `ingestion/tests/test_load_state_boundary.py` — it never hits the live MapServer or the live database.

The live load is intentionally operator-gated: it requires a real `DATABASE_URL`, network access to `gisservicemt.gov`, and a deliberate decision to write to the database.

## 7. Open question on `'reference_boundary'` Literal extension

T7's investigation found MSDI qualifies as `'gis_layer'`, making the `'reference_boundary'` extension unnecessary for S03.0. The plan's default extension was NOT applied; no schema change was made to `SourceCitation.document_type`.

If a future state has no GIS-published boundary and only a federal Census boundary (TIGER) as its source, that story should reopen the question: either (a) extend `SourceCitation.document_type` with `'reference_boundary'` across all three schema places (Pydantic, TypeScript, architecture.md per ADR-006), or (b) treat TIGER as `'gis_layer'` with an explanatory comment. The tradeoffs are documented in the S03.0 plan at `.roughly/plans/S03.0-schema-prep-plan.md` § T7.
