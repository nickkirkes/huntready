## Spec deviations applied

**Deviation #1 — id format string.** Spec line 1045 had `{state}-{jurisdiction_code}-{species_group}-{license_year}-binding-{geometry_id}-{role}` but S03.6.1 shipped `{state}-{jurisdiction_code}-{species_group}-{license_year}-{role}-{geometry_id}`. S03.10 imports `_JURISDICTION_BINDING_ID_FORMAT` from `load_regulation_records.py` to match. Locked by `TestIdStability::test_id_format_string_is_canonical` + `test_format_string_produces_ids_without_binding_literal`. Spec footnote added at line 1045.

**Deviation #2 — count band.** Spec AC #1087 had 1,500-3,500; planning produced 500-1,000 estimate; T0 probe confirmed ≈1000 projected. Band locked at `[400, 1100]` (intentionally wide pending T16). Spec footnote added at AC #1087. Narrow to ±30% around T16's empirical observation in a same-PR follow-up (or in S03.11/S03.12 if S03.10 closes first).

**Deviation #3 — source attribution mechanism.** Spec AC #1088 says "cites the original ArcGIS layer" but unspecified mechanism. Plan resolves: `SELECT id, source FROM geometry WHERE id = ANY(%s)` pre-fetch at load time (geometry table is authoritative per AC #1088; overlay fixture is derived). Inline in adapter (not a new `db.py` helper, per constraint #7).

**Deviation #4 — no-hunt-zone "nearby" rule.** Spec line 1062 had two-clause `ST_Touches OR ST_DWithin(centroid, centroid, 5000)`. **Empirical finding: centroid-to-centroid produces 0 matches for ALL 3 orphan zones at 5km on real Montana data.** Replaced with single-clause boundary-to-boundary `extensions.ST_DWithin(zone.geom, hd.geom, 5000)` on the native `geography` type. Reasons:
- Centroid-to-centroid: 0 matches (Glacier=0, Sun River=0, Yellowstone=0 at 5km centroid distance)
- Boundary-to-boundary: 8, 8, 13 matches respectively (covers the spec's "near OR touching" intent)
- Geography-native ST_DWithin is the canonical PostGIS pattern; sidesteps cross-cast statement_timeout per ADR-016 precedent
- No new ADR required — refines the implementation of an existing rule
Spec footnote added at line 1062.

## Schema-prefix audit (T8 SQL)

Per the user's S03.10 constraint #2: every PostGIS function call in raw SQL must be `extensions.`-qualified. Audited in `load_jurisdiction_bindings.py::_query_nearby_hds_for_zone`:

```sql
SELECT DISTINCT hd.id
FROM geometry zone
JOIN geometry hd
  ON hd.kind IN ('hunting_district', 'portion')
  AND extensions.ST_DWithin(zone.geom, hd.geom, %s)
WHERE zone.id = %s
ORDER BY hd.id
```

✅ `extensions.ST_DWithin` (only PostGIS call; correctly prefixed)
✅ Column name `geom` (not `geog`)
✅ No `::geometry` casts (geography-native)
✅ No `ST_Touches`, no `ST_Centroid` (Deviation #4 single-clause rule)
✅ `kind IN ('hunting_district', 'portion')` (matches T0-probed enum values)

Test `TestQueryNearbyHdsForZone::test_sql_uses_extensions_prefix_not_bare_name` locks the prefix invariant.

## DDL findings

**Column name:** `geom` (geography(MultiPolygon, 4326)), NOT `geog`. CLAUDE.md uses "geog" colloquially in one sentence; the actual DDL at `supabase/migrations/20260425000000_initial_schema.sql` uses `geom`. The plan's original SQL block used `geog` and was corrected mid-T0.

**Geometry kind enum values:** `cwd_zone`, `hunting_district`, `portion`, `restricted_area`, `state` (5 values). The plan's original T0 SQL used `'hunting_district_portion'` which doesn't exist; corrected to `'portion'` at T0.

**PostGIS schema:** lives in `extensions` namespace. All ST_* functions must be `extensions.`-prefixed in raw SQL. The schema is NOT on the connection's search_path, so bare names fail to resolve.

## M2 follow-ups (deferred, not blocking S03.10)

1. **`role='other_overlay'` semantic awkwardness for no-hunt-zone bindings.** The DDL role enum has 7 values; none semantically captures "this HD's regulations don't apply inside this zone." V1 ships with `'other_overlay'` (the only DDL-permitted choice). M2 candidate: add `'no_hunt_zone'` role enum value (requires DDL migration + ADR), or clarify `'other_overlay'` semantics in schema docs.

2. **Q19 (id-text-PK slug-drift)** does NOT propagate to `upsert_jurisdiction_binding` because that helper excludes identity fields from the UPDATE clause entirely (S03.6.1 OQ-S6.1-4). S03.10 does not introduce a local module-load drift guard. The pre-M2 Q19 ADR is still required for the 3 helpers it does affect.

3. **Cell-level source attribution.** S03.10 attributes binding source from the geometry table (a single source per geometry). If a future state's geometry has multi-source provenance (e.g., one HD with parts from two ArcGIS layers), the schema would need a multi-source field. Not a V1 concern.

4. **Count band narrowing.** After T16 produces the live count, narrow `_BINDING_COUNT_GUARD_BAND` to ±30% around the observed value. Update spec AC #1087's footnote with the empirical number.
