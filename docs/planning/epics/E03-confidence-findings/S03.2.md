# S03.2 confidence framework primitives — design notes

Per ADR-017 §6 artifact policy: working notes for the confidence-calibration framework as it lands in code. This file is deleted at the M1 tag.

## What was implemented

- `ConfidenceTier(str, Enum)` in `ingestion/ingestion/lib/pdf.py` — three members `LOW`, `MEDIUM`, `HIGH` with `.value` strings `"low"`, `"medium"`, `"high"`.
- `.rank` property reading from a module-level `_TIER_RANKS` dict (`LOW=0`, `MEDIUM=1`, `HIGH=2`).
- `__lt__` comparing the rank, returning `NotImplemented` for non-tier operands.
- `min_tier(tiers: Iterable[ConfidenceTier]) -> ConfidenceTier` using `min(tiers_list, key=lambda t: t.rank)`.
- `demote_one_tier(tier) -> ConfidenceTier` with the floor-clamped rule `HIGH→MEDIUM`, `MEDIUM→LOW`, `LOW→LOW`.
- Helpers and the enum live in `pdf.py` rather than a separate `confidence.py` because S03.2 is the only story building shared library code in this epic and a second module purely for two helpers + an enum was unjustified abstraction. If S03.6+ grows confidence-related helpers beyond the framework primitives, lift them into a dedicated module then.

## Why `str, Enum` mixin

The schema layer (`ingestion/ingestion/lib/schema.py:43`) defines `Confidence = Literal["high", "medium", "low"]`. With the `str, Enum` mixin, a `ConfidenceTier` instance IS a string (`ConfidenceTier.HIGH == "high"` is `True`), so it can be written directly to `regulation_record.confidence` without `.value` access at every callsite. This avoids spreading `tier.value` calls across S03.6+ ingestion code.

## Why explicit `key=` instead of relying on `__lt__` in `min_tier`

With `class ConfidenceTier(str, Enum)` and a custom `__lt__`, the MRO is `[ConfidenceTier, str, Enum, object]` — our `__lt__` is found first, so bare `min(tiers)` on enum instances IS rank-correct at runtime. We use `key=lambda t: t.rank` anyway because it is unambiguous (no MRO knowledge required to read) and survives any future change to the mixin or to `__lt__`. The `__lt__` is implemented for general comparability of `ConfidenceTier` instances; `min_tier` does not depend on it.

The lexicographic-MIN trap (`min(["high", "medium", "low"]) == "high"` because `h < l < m`) is locked by `test_pdf.py::TestMinTier::test_lexicographic_trap_explicit`. The test asserts three things:

1. `min_tier` returns `LOW` for `[HIGH, MEDIUM, LOW]` (rank-correct).
2. `min(tiers, key=lambda t: t.value)` returns `HIGH` (proves the trap is real for THIS enum, not just for plain strings).
3. Bare `min(["high", "low", "medium"])` returns `"high"` (documents the original Python idiom).

A future refactor that regresses to either of the naive forms (bare `min(tiers)` on raw strings, or keying on `.value`) fails (3) or (2) and the test catches it.

## Empty-iterable handling

`min_tier([])` raises `PdfExtractionError("min_tier received an empty iterable — caller must supply at least one tier")`. This is a fail-loud signal of an upstream bug — empty input means a row aggregation produced no signals, which should never happen for a record being written to the database. Per CLAUDE.md: "Don't add error handling for scenarios that can't happen … Only validate at system boundaries." The empty case here is a system boundary (the boundary between the aggregation step and the persistence step), and silently returning a default tier would mask the upstream error.

## Open questions deferred to S03.3+

None for S03.2. The framework primitives are deliberately minimal: an enum, a MIN, a demote-one. Per-story confidence assignment rules (which signals at what tier for which entity types) are S03.3-S03.5's domain and will accumulate findings in this directory under each story's file.
