# S03.2 confidence framework primitives — design notes

Per ADR-017 §6 artifact policy: working notes for the confidence-calibration framework as it lands in code. This file is deleted at the M1 tag.

## What was implemented

- `ConfidenceTier(str, Enum)` in `ingestion/ingestion/lib/pdf.py` — three members `LOW`, `MEDIUM`, `HIGH` with `.value` strings `"low"`, `"medium"`, `"high"`.
- `.rank` property reading from a module-level `_TIER_RANKS` dict (`LOW=0`, `MEDIUM=1`, `HIGH=2`).
- `__lt__` comparing the rank, returning `NotImplemented` for non-tier operands.
- `min_tier(tiers: Iterable[ConfidenceTier]) -> ConfidenceTier` using `min(tiers_list, key=lambda t: t.rank)`.
- `demote_one_tier(tier) -> ConfidenceTier` with the floor-clamped rule `HIGH→MEDIUM`, `MEDIUM→LOW`, `LOW→LOW`.
- Helpers and the enum live in `pdf.py` rather than a separate `confidence.py` because S03.2 is the only story building shared library code in this epic and a second module purely for two helpers + an enum was unjustified abstraction. If S03.6+ grows confidence-related helpers beyond the framework primitives, lift them into a dedicated module then.

## Why `str, Enum` mixin

The schema layer (`ingestion/ingestion/lib/schema.py:43`) defines `Confidence = Literal["high", "medium", "low"]`. With the `str, Enum` mixin, a `ConfidenceTier` instance IS a string (`ConfidenceTier.HIGH == "high"` is `True`), so it can be written directly to `regulation_record.confidence` without `.value` access at every callsite. This avoids spreading `tier.value` calls across S03.6+ ingestion code.

## Why explicit `key=` instead of relying on `__lt__` in `min_tier`

With `class ConfidenceTier(str, Enum)` and a custom `__lt__`, the MRO is `[ConfidenceTier, str, Enum, object]` — our `__lt__` is found first, so bare `min(tiers)` on enum instances IS rank-correct at runtime. We use `key=lambda t: t.rank` anyway because it is unambiguous (no MRO knowledge required to read) and survives any future change to the mixin or to `__lt__`. The `__lt__` is implemented for general comparability of `ConfidenceTier` instances; `min_tier` does not depend on it.

The lexicographic-MIN trap (`min(["high", "medium", "low"]) == "high"` because `h < l < m`) is locked by `test_pdf.py::TestMinTier::test_lexicographic_trap_explicit`. The test asserts three things:

1. `min_tier` returns `LOW` for `[HIGH, MEDIUM, LOW]` (rank-correct).
2. `min(tiers, key=lambda t: t.value)` returns `HIGH` (proves the trap is real for THIS enum, not just for plain strings).
3. Bare `min(["high", "low", "medium"])` returns `"high"` (documents the original Python idiom).

A future refactor that regresses to either of the naive forms (bare `min(tiers)` on raw strings, or keying on `.value`) fails (3) or (2) and the test catches it.

## Empty-iterable handling

`min_tier([])` raises `PdfExtractionError("min_tier received an empty iterable — caller must supply at least one tier")`. This is a fail-loud signal of an upstream bug — empty input means a row aggregation produced no signals, which should never happen for a record being written to the database. Per CLAUDE.md: "Don't add error handling for scenarios that can't happen … Only validate at system boundaries." The empty case here is a system boundary (the boundary between the aggregation step and the persistence step), and silently returning a default tier would mask the upstream error.

## Open questions deferred to S03.3+

None for S03.2. The framework primitives are deliberately minimal: an enum, a MIN, a demote-one. Per-story confidence assignment rules (which signals at what tier for which entity types) are S03.3-S03.5's domain and will accumulate findings in this directory under each story's file.

## Why `extract_text` accepts pdfplumber's word-grouping (ADR-008 boundary defense)

This story's `extract_text` primitive returns `page.extract_text()` from pdfplumber 0.11.x with no additional normalization. pdfplumber's word-grouping algorithm collapses repeated spaces (a `(hello  world) Tj` content stream returns as `"hello world"` with one space). Newlines from line-position changes ARE preserved.

A future reviewer reading this implementation against the story spec line 213 (`"no normalization, no whitespace squashing, no Unicode normalization"`) or AC #2 (`"extract_text returns source string with no normalization"`) may flag this as non-compliant. This section pre-defends against that read.

**ADR-008's actual concern is paraphrase prevention, not whitespace fidelity.** The ADR's canonical example is preserving statutory precision: `"a landowner must own at least 160 acres"` vs `"at least 640 contiguous acres"` — different rules that paraphrase would silently collapse. pdfplumber's word-grouping does NOT paraphrase: it preserves every word, every digit, every name, every punctuation mark, and the sequence of tokens. The `160` vs `640` distinction is fully preserved. ADR-008's failure mode is "synthesizing text beyond composition of verbatim pieces" — pdfplumber synthesizes nothing.

**The strict-literal alternatives are unworkable.** Two were considered and rejected:

1. *Char-level reconstruction via `page.chars`.* PDF content streams generally do NOT contain literal spaces between words; words are separated by `Td` positional moves or by independent `Tj` operations. `"".join(ch["text"] for ch in page.chars)` produces `"helloworld"`-style spaceless output. Recovering word boundaries requires inspecting glyph positions and injecting spaces — which is exactly what pdfplumber's word-grouping already does. Any honest char-level path would either be useless (no spaces) or a reimplementation of pdfplumber.

2. *Fail when byte-exact fidelity is impossible.* Byte-exact fidelity from rendered PDF glyphs is fundamentally impossible without access to the raw content-stream bytes (which pdfplumber does not expose at this layer). Adopting this rule means every call fails, every PDF rejected, the project ships nothing.

**The story spec's own escape hatch authorizes the current posture.** Line 213 reads: `"Pure pass-through is the baseline; deviations are documented exceptions."` pdfplumber's word-grouping is the platform-level deviation; `pdf.py::extract_text`'s docstring documents it explicitly (the "Platform note" paragraph names the trap). That documentation is the "documented exception" the spec contemplates.

**The plan review already adjudicated this.** The S03.2 plan went through two review passes; the second pass explicitly evaluated AC #2 interpretation and concluded the "no normalization beyond what pdfplumber itself does" reading is the only one implementable without a parallel char-level path that brings no real fidelity gain.

**If a future story genuinely needs byte-exact source text** (e.g., forensic comparison against a published correction PDF, or a hash-stable invariant), add a separate `extract_text_chars_raw(page) -> str` helper at that point with its own contract — do not retrofit `extract_text`. The rename-and-add path keeps the existing 404 tests stable and makes the byte-exact contract explicit at the callsite. As of S03.2 there is no such requirement.

**Re-litigating this in a code review is not productive** unless one of these conditions changes:

- A real S03.3+ failure mode is identified that double-space preservation would fix (i.e., a regulation that means something different with two spaces than with one — none anticipated)
- pdfplumber gains a new API exposing the un-normalized text-map (none in 0.11.x)
- ADR-008 is amended to redefine "verbatim" as byte-exact (would require explicit ADR change, not a S03.2-level decision)
