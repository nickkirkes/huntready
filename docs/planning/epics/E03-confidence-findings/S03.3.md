# S03.3 DEA booklet extraction — confidence calibration + real-PDF findings

Per ADR-017 §6 artifact policy: working notes for the confidence-calibration framework as it lands in code. This file is deleted at the M1 tag.

## What was implemented

`ingestion/states/montana/extract_dea.py` (~1540 LOC) extracts per-HD regulation rows from the Montana DEA hunting regulations booklet (141 pages) into a deterministic JSON artifact at `ingestion/states/montana/extracted/dea-2026.json`. The artifact carries 272 sections: 129 deer + 112 elk + 31 antelope = 272. The antelope group decomposes as 22 base HDs + 8 Region 7 portion-emissions (HDs 700-705, with HD 701 and HD 703 each emitting once per portions sub-section so they appear in both North and South Yellowstone) + 1 statewide overlay (`hd_number="STATEWIDE"`, license `900-20`). Per-row count: 1190 license rows (after row filters described below; deduplication of multi-page HD heading matches removed ~30 duplicate sections and ~177 duplicate rows from the initial post-cubic count, then the UAT-fix cycle added back +8 sections / +12 rows from Region 7 portions emission per the S03.3 UAT D2 fix).

**Post-implementation fix (page_reference on rows):** A contract mismatch was identified against AC S03.3 line 337 ("every extracted row carries a `PageReference`"). `DeaRowExtraction` was updated to include a `page_reference: PageReference` field. Every row in `_rows_to_license_extractions` and the single row in `_extract_statewide_antelope_overlay` now carry `page_reference`. The value is inherited from the enclosing section's `page_reference` (the section's starting page) — see "Deferred follow-ups" below for the per-row accuracy caveat.

13 public/private functions plus 5 TypedDicts (`SeasonCoverage`, `SeasonWindow`, `DeaRowExtraction`, `DeaSectionExtraction`, plus the row-fragment fields), with full re-run determinism (byte-identical SHA-256 across two runs of the CLI against the same PDF).

## Why `re.search` (not `re.match`) for the HD-code pattern

The original plan T9 used `re.match(r"^\d{3}-\d{2}$", license_code)` — a full-string match. Live extraction surfaced that the DEA PDF rarely uses pure-code license cells. The actual cell forms:

| Form | Count | Examples |
|---|---|---|
| Pure code | 1 | `"900-20"` (statewide antelope overlay only) |
| Compound prose + code | 678 | `"Deer B License: 124-00"`, `"Antelope License: 490-20"`, `"Elk B License: 170-00"` |
| Prose only | 690 | `"General Deer License"`, `"General Elk License"`, `"Region 5"` (an antelope sub-license footer) |

Full-match would have rated all 678 compound-form cells as `MEDIUM`, hiding the structural quality differential between **a cell that names an HD code in a regulated cell** vs **a cell that names a license type without an HD code**. The HD-coded cells are objectively higher-quality structured data — the HD code is a primary key linking S03.6's `regulation_record` row to the section's HD. `re.search(r"\b\d{3}-\d{2}\b", license_code)` correctly captures the embedded form. The pure-code form still matches.

This is documented inline in `_assign_row_confidence` and locked by `TestRowConfidence::test_table_source_with_compound_hd_code_high`.

## Real-PDF findings the plan did not anticipate

The S03.3 spec extrapolated from research notes that did not survive contact with the actual PDF. Adaptations applied during implementation, not after:

1. **One table per page, not per HD.** The booklet uses a single full-page table per page, with HD section delimiters embedded as data rows whose first cell contains `"HD NNN - Name"`. `_extract_hd_table` slices the per-HD subset out of each page-level table via `_slice_hd_rows`.

2. **Column headers use `"... SEASON DATES"` suffix.** The plan's constants used `"GENERAL"`, `"ARCHERY ONLY"`, etc. The real PDF uses `"GENERAL SEASON DATES"`, `"ARCHERY ONLY SEASON DATES"`, etc. The constants `_SEASON_COLUMNS`, `_SEASON_COLUMN_TO_KEY`, `_WEAPON_TYPE_OVERRIDE_BY_COLUMN` were corrected.

3. **`"-"` is the absent-season sentinel.** Empty cells are NOT used to denote "this license does not cover this season"; the dash is. `_is_season_cell_absent(cell)` treats `None` and `"-"` identically. Without this, every season would appear covered in `season_coverage`.

4. **Species banner rows embedded in the table.** The deer/elk page range uses `["DEER", None, ...]` and `["ELK", None, ...]` banner rows to switch species mid-page. There is no separate ELK banner page (the plan T10's `find_section(pdf, r"ELK\s+(?:HUNTING\s+)?REGULATIONS")` approach was discarded). `_split_rows_by_species` partitions HD rows into per-species buckets using the banner rows.

5. **License code carry-forward.** Real DEA tables group multiple opportunity sub-rows under a single license. Rows 2–N within a license group have `None` in the LICENSE/PERMIT cell and inherit the prior license code. T6's `_rows_to_license_extractions` carries forward the most-recent non-None license code (and, separately, the most-recent non-None opportunity for sub-opportunity rows).

6. **Antelope tables use a different column format.** The antelope section uses `"License"`, `"Opportunity"`, `"Quota"`, `"Archery Season Dates"`, `"Season Dates"` — not the 11-column deer/elk format. `_ANTELOPE_HEADER_SIGNATURE` and column aliases handle this.

7. **`_ANTELOPE_PAGES = (136, 141)`, not `(136, 142)`.** The PDF has 141 pages total, not 142.

8. **The 900-20 statewide row is a repeated table row, not a prose paragraph.** The plan T8 assumed a standalone prose paragraph; reality is a structured row appended after every per-HD antelope block on every antelope page. T8 parses it via a single regex against the page text.

9. **Window divergence within a single HD's general-season column.** Some HDs' GENERAL season legitimately differs by opportunity row (e.g., `'Either-sex WT'` is `Oct 24-Oct 30` while `'Antlered Buck MD'` is `Oct 24-Nov 29` in the same HD). The window-divergence WARNING (`_logger.warning` from `_aggregate_section_season_windows`) surfaces this as informational signal; first-observation-wins per spec line 307. **Reviewer should expect WARNING messages during a real-PDF run; they are signal, not noise.** The most prolific divergence is `'Elk B License: 699-01'` reporting `'Oct 22-Nov 29'` against the section's general-window `'Oct 24-Nov 29'` — verify against source PDF if a downstream consumer relies on per-license window precision.

## Why per-row `extracted_at` is sourced from the PDF manifest, not `datetime.now()`

T13's determinism check (re-run + diff = byte-identical) is the AC #1 gate ("produces deterministic `dea-2026.json`"). `datetime.now()` would silently break this on every re-run. `_load_extracted_at_from_manifest(pdf_path)` reads the manifest's `fetched_at` field — the timestamp at which the specific PDF version was anchored to disk. This means:

- The same PDF version → same `extracted_at` → byte-identical artifact across runs.
- A re-fetched PDF (drift, new version) → new manifest → new `extracted_at` → different artifact, **and** the per-PDF SHA drift marker from S03.1 would block downstream consumers until acknowledged.
- The manifest is fail-loud-required: missing manifest → `PdfExtractionError`, not a fallback to `datetime.now()`.

This binds the artifact's provenance to a specific PDF SHA. Any downstream consumer reading `extracted_at` learns the PDF version, not the wall-clock time of the extraction run.

## Cleanup-rule application notes (ADR-008 verbatim discipline)

The two cleanup regexes from spec lines 315–319 are applied only to structured `rows` cells, never to section-level `verbatim_text`:

- **Cell padding** `re.sub(r'\s{3,}', ' ', cell.strip())` fired on most cells with multi-line content (column headers split across 3 lines: `"EARLY\nSEASON\nDATES"` → `"EARLY SEASON DATES"`). Spot-checked 20 cells; behavior is correct.
- **Hyphenated rejoin** `re.sub(r'(?<=[a-z])-\n(?=[a-z])', '', text)` fired rarely (most DEA cells are short and don't wrap). Test counter-examples lock the regex against false-positive rejoins on `9/7-10/20` (date ranges) and `262-50` (license codes).

`extract_text`'s pdfplumber word-grouping (S03.2 confidence-findings doc § "Why `extract_text` accepts pdfplumber's word-grouping") provides the section-level `verbatim_text` baseline. ADR-008 paraphrase-prevention is met because numeric tokens, units, and lexical words are byte-exact within pdfplumber's word-grouped output.

## Post-cubic bug fixes (two correctness issues)

After the cubic-review pass, two latent correctness bugs were identified and fixed:

### Bug 1 (P1): Multi-page HD lookahead truncated on the first non-DEA page

`_extract_hd_table`'s continuation loop probed `_find_dea_table_on_page` for the next 1–2 pages. When a lookahead page had no DEA-format table (interstitial blank, footer, photo, or map page), the loop `break`ed instead of `continue`ing. The docstring claimed "Cap at 2 lookahead pages" — but in practice only the first probe ran. An HD whose continuation rows live on page N+2 with an interstitial page at N+1 would have been silently truncated.

The current DEA PDF doesn't appear to exhibit this pattern, but a future PDF revision with an embedded photo or footer page mid-section would break the extraction. Fix: change `break` → `continue` in the `cont_tm is None` branch. The 2-page cap is preserved by the `range(1, 3)` loop bound.

### Bug 2 (P2): Opportunity carry-forward leaked across license boundaries

`_rows_to_license_extractions` carried both `last_license_code` and `last_opportunity` forward across rows. The license code carry-forward is correct (sub-rows under a merged-cell license group should inherit the license). But the opportunity carry-forward had no scope check: when a row introduced a new `license_code` AND pdfplumber dropped its `OPPORTUNITY` cell, the previous license's opportunity silently inherited and corrupted the row.

Fix: when a new `license_code` is detected (the `if license_code:` branch fires), reset `last_opportunity = None`. The new license's first row must then carry its own opportunity, or extraction fails loud per ADR-001. Sub-rows within the same license group still inherit normally.

Applying Fix 2 surfaced 9 additional real-PDF rows (across deer/elk pages) that had a license-like first cell (`"ELK Hunting by Drawing Only"`) and `None` everywhere else — these are sub-section labels embedded in the multi-HD table, not regulation rows. They had been silently inheriting the previous opportunity and emitting as corrupt rows. The fix companion is a label-only row filter: skip rows where the first cell has content but every other cell is `None`. This generalizes the species-banner skip pattern (`['DEER', None, ...]` etc.) to any single-cell label.

Tests added: `TestLabelOnlyRowFilter` (2 tests) + `TestOpportunityCarryForwardScope` (3 tests, including the fail-loud case for new-license-with-missing-opportunity and the happy-path within-group carry-forward).

## Multi-page HD deduplication (post-cubic P1)

**Problem.** A subsequent review pass identified that `_iter_hd_sections` was yielding the same `hd_number` multiple times for any HD whose section spans pages. Three regex-matchable forms of an HD's heading exist on the live PDF:

1. `"HD NNN - Name"` — canonical heading at the section's start.
2. `"HD NNN - Name - Continued on the Next Page"` — footer annotation embedded in the start page's text (a courtesy pointer to the operator).
3. `"HD NNN - Name - Continued"` — repeated heading at the top of the continuation page.

The `_HD_HEADING_REGEX` (`r"^HD\s+(?P<num>\d{3})\s*-\s*(?P<name>.+?)\s*$"` with MULTILINE) matches all three forms, so a multi-page HD was yielded 2–3 times. The orchestrator then called `_extract_hd_table` for each yield independently, producing duplicate `DeaSectionExtraction` rows. Live impact pre-dedup: 20 deer/elk HDs duplicated, ~30 duplicate sections, ~177 duplicate rows in the artifact.

**Fix.** First-match-wins deduplication inside `_iter_hd_sections`:

```python
seen: set[str] = set()
for ...:
    for match in _HD_HEADING_REGEX.finditer(text):
        hd_number = match.group("num")
        if hd_number in seen:
            continue
        seen.add(hd_number)
        ...
```

This is safe because `_extract_hd_table`'s multi-page lookahead already merges continuation rows correctly when invoked once per HD (verified live: HD 240 deer ends on p58 with 9 rows, elk content is entirely on p59 with 7 rows — `_extract_hd_table(58, "240")` returns the 16-row union, then `_split_rows_by_species` partitions to 8 deer + 6 elk). The first match always reflects the canonical `hd_name` (without `- Continued` suffix), so the artifact's `hd_name` field is clean.

**Why the violation's hypothesis was wrong.** The violation suggested that continuation pages "don't repeat the HD header — the HD's rows simply start at the top of the table." Investigation showed continuation pages DO repeat the HD header (with `- Continued` suffix), and `_slice_hd_rows` correctly identifies them. The actual symptom was duplication, not silent drop. The fix is dedup, not the violation's proposed "collect rows until first HD header" approach.

### Defects fixed

- **D1 (P0) — per-row season_windows.** The previous `_aggregate_section_season_windows` helper computed section-level windows via first-observation-wins, then `_row_season_windows` filtered them per `season_coverage`. Real PDF rows have per-row date variation (HD 124 deer has three distinct General Season windows: `Oct 24-Oct 30`, `Oct 24-Nov 29`, `Oct 31-Nov 29`) — the artifact carried only the first observation across all 5 rows. **Fix:** deleted both helpers; per-row `season_windows` now reads each row's own column cells inline within `_rows_to_license_extractions`. The divergence-WARNING path was removed because divergence is the data, not an anomaly.

- **D2 (P0) — Region 7 / Portions sub-section slicer.** The antelope orchestrator used `_iter_hd_sections` + `_extract_hd_table` which only stopped at HD-numbered headers. Region 7's `"Portions of HDs A, B, C ... [directional qualifier]"` sub-sections (sub-section delimiters that follow a `"Region N"` header) were absorbed into the preceding HD-numbered section's data rows. HD 690 carried 4 phantom rows (`007-21`, `007-31`, `007-20`, `007-30`); Region 7 portions sub-sections were not represented at all. **Fix:** rewrote the antelope orchestrator path to walk pages directly, building "logical sections" from one of three delimiters: `HD NNN - Name`, `Portions of HDs A, B, ... [qualifier]`, or `Region N` (skipped). For Portions sub-sections, **option (a) per directive**: emit one `DeaSectionExtraction` per listed HD with the same row content; license codes (`007-21` etc.) disambiguate at S03.6 ingestion. New helpers: `_parse_portions_hd_list` and `_is_portions_subsection_row`.

- **D3+D4+D5+D6 (P1-P3) — STATEWIDE 900-20 column-faithful extraction.** The legacy `_extract_statewide_antelope_overlay` was a hand-rolled regex parser that synthesized a STATEWIDE row with column-position remap (Aug.15-Nov.08 mapped to `archery_only` because the extras text said "ArchEquip only", but the date was actually in the **Season Dates** column = `general` per the column-header mapping); stripped the `"Antelope License: "` prefix; stripped commas from `quota_range`; dropped the verbatim extras text; and overwrote `opportunity` with synthesized prose. **Fix:** deleted the entire helper and the `_ANTELOPE_900_ROW_RE` constant. ### Wrong assumptions in the original implementation

1. **"Section-level windows are sufficient because all rows in an HD share the same date structure."** False — DEA per-row windows genuinely vary by opportunity. ADR-008 requires the per-row variance be preserved verbatim.
2. **"The STATEWIDE 900-20 row is structurally different and warrants a hand-rolled parser."** False — the 900-20 row is a regular antelope-table row that happens to summarize statewide-applicable rules. Routing it through the same extraction path as per-HD rows yields the column-faithful output the directive requires; the only differentiator is `hd_number="STATEWIDE"`.
3. **"Antelope tables only contain HD-numbered sections."** False — Region 7's "Portions of HDs ..." sub-sections bind regulation rows to multiple sibling HDs without an HD-numbered delimiter. The slicer must recognize the Portions delimiter as a section boundary.
4. **"`weapon_type_override` should be derived from extras text when the date appears in an unexpected column."** False — column-position is the source of truth. The extras text ("ArchEquip only.") is a row-level annotation that survives via `weapon_types=["archery"]` on the row, not by remapping the date column.

## Deferred follow-ups (non-blocking for S03.3)

- **`weapon_types` per-row default.** Every per-HD row currently emits `["any_legal_weapon"]`; STATEWIDE emits `["archery"]` via orchestrator-side override (the `extras` text is the source of truth for the archery restriction). Real DEA license codes ending in `-30` (per Montana convention) are archery-only, but the per-license weapon eligibility is also expressible via `season_coverage`. S03.7's `season_definition.weapon_type` consumer reads `weapon_type_override` from `season_windows`, not `weapon_types` from rows; the current default is operationally correct. Revisit if S03.7 review surfaces a counter-example.
- **Per-row page-accurate provenance for multi-page HDs.** When an HD's section spans pages, all rows in the section currently inherit the section's starting page in `page_reference`. For multi-page HDs (e.g., HD 240 elk content lives on p59 but is tagged with p58), the row's `page_reference.page_num_1based` does not reflect the actual source page. Surface to S03.6 review if needed for downstream provenance.
