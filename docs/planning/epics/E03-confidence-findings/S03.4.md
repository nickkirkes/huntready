## Date-Arbitration Rule: Option B (doc-type-precedence), Not Option A

### What the spec said

The S03.4 epic spec (§ "Date-arbitration logic") specified Option A: "MAX `publication_date` wins." The rationale was that a later-dated document reflects more current agency intent.

### Why Option A is wrong for the 2026 Black Bear case

- Booklet `publication_date`: `2026-04-27`
- Correction `publication_date`: `2026-03-18`

Naive MAX would pick the booklet as the winner. The correction — which self-identifies as "Corrections to the 2026 Printed Black Bear Regulations" — would never apply. This silently inverts operator intent: the correction PDF exists precisely to amend the booklet, but Option A would suppress it entirely because its date is earlier.

This is not an edge case. A printed correction is distributed before the booklet goes to press (the correction was finalized in March; the booklet was finalized in April). The chronological order is a feature of the publication workflow, not a sign that the correction is "stale."

### Option B (shipped)

`document_type='correction'` always wins over `document_type='annual_regulations'`, regardless of `publication_date`. Date is used only for tiebreaking among sources of the same `document_type`.

This is the only defensible read of operator intent. The implementation is in `_merge_with_corrections` in `ingestion/states/montana/extract_black_bear.py`.

**Locking test:** `TestMergeWithCorrections::test_real_world_option_b_correction_wins_despite_earlier_date` in `ingestion/tests/test_extract_black_bear.py`. Asserts that correction content wins even when the booklet has a later `publication_date`.

## Decision NOT to Use `bmu_region_overrides.yaml`

The regional map on p. 8 of the Black Bear booklet is image-only and unparseable by pdfplumber. The spec anticipated needing a fallback YAML file mapping BMU numbers to regions.

In practice, no fallback is needed. Row 0 of the (transposed) BMU table carries explicit `"REGION N"` column markers that advance left-to-right across the table columns. `hd_region` is extracted directly by tracking the current region marker as each column is processed. The regional map on p. 8 is redundant given the table's own Row 0 markers.

No `bmu_region_overrides.yaml` was created. If a future state's BMU table omits region markers, this YAML fallback approach would become relevant.

## V1 Row-Level Source Attribution (AC #431 Granularity Note)

`BmuRowExtraction` carries a single `source_publication_date` per row, not per cell. When the correction touches any cell of a row, two things happen:

1. `applied_correction=True` is set on the row.
2. `source_publication_date` is updated to the correction's `publication_date` (`2026-03-18`).

AC #431 ("a row touched by the correction has `source.publication_date == 2026-03-18` for the touched cells") is satisfied at **row granularity** in V1. Per-cell attribution is NOT implemented — doing so would require splitting `BmuRowExtraction` into a cell-level structure with per-cell provenance fields, which is a schema change beyond V1 scope.

**Deferred to M2.** The unit test in T14 (`TestMergeWithCorrections::test_basic_merge_correction_wins_option_b`) verifies the row-level behavior and locks it in as the V1 contract.

## AC #428 Spec Discrepancy: 8 Quota-Closure BMUs, Not 9

AC #428 (epic line 428) and the S03.7 spec (line 639) both list 9 BMUs including BMU 530. Live-PDF discovery (2026-05-10) confirmed BMU 530 does not exist anywhere in the 2026 Black Bear booklet PDF — not in the BMU table, not in the closure-prose paragraph, not in any other page.

**Shipped artifact has 8 quota-closure BMUs: 411, 420, 440, 450, 510, 520, 600, 700.**

The drift guard in `_extract_closures` locks the expected set to exactly these 8 BMUs and raises `PdfExtractionError` if the first-sentence BMU scan returns a different count or set. This guard is scoped to the first sentence only (important — see the phone-number pitfall entry in `.roughly/known-pitfalls.md`).

**Action for downstream:** The AC reviewer and S03.7 implementor should note the spec is wrong for 2026. The epic lines referencing 9 BMUs (including 530) should be revised at the next epic revision. If BMU 530 reappears in a future year's booklet, the drift guard will surface it.

## Real-PDF Discoveries (S03.4)

These are the structural surprises that did not survive contact with the actual PDF — analogous to S03.3's "9 PDF discoveries."

1. **Transposed/rotated BMU table.** pdfplumber returns 10 rows × N columns where each column = one BMU; all cell text is doubly reversed (line order AND character order). `page.rotation == 0` — no PDF-level rotation hint. Every cell must pass through `_reverse_cell_text` before normalization.

2. **BMU table page range is pp. 9–12**, not pp. 10–11 as spec claimed. Four pages, not two.

3. **8 quota-closure BMUs, not 9.** BMU 530 absent from the 2026 PDF entirely.

4. **Correction PDF is prose-only** (no tables). It removes the entire "hound training season" column (Row 4 in the transposed extraction) for all BMUs. Operations were synthesized as N per-BMU correction records (one per BMU) targeting `(bmu_number, "hound_training_season_dates")`.

5. **Regional map (p. 8) is image-only.** `hd_region` derives from Row 0 of the BMU table via the `"REGION N"` column markers tracked left-to-right. No `bmu_region_overrides.yaml` needed.

6. **Permit Managed sub-table on p. 11**: 3 extra columns with different row semantics (BMUs 510-20, 520-20 — apply-by, permit-range, license-codes). Skipped in V1 via an `in_managed_block` state flag triggered by a `"Permit Managed Opportunities"` Row 0 marker.

7. **"Quota Managed Opportunities" is opportunity description, NOT a sub-table header.** Critical disambiguation: it appears as a row value inside the main table, not as a structural delimiter like "Permit Managed Opportunities."

8. **Female-subquota BMUs (300, 301, 319, 580) have asterisk-suffixed BMU IDs** in the table (`"300*"`, `"580*"`). `_BMU_NUMBER_REGEX` extended to `^(\d{3})\*?$` to strip the footnote marker at parse time.

9. **Quota cells appear in 3 word orderings** due to how pdfplumber reconstructs multi-line rotated cells: Pattern A `"= N Female subquota"`, Pattern B `"= N quota Harvest"`, Pattern C `"quota = Harvest N"`. `_parse_quota_cell` uses a primary regex and a word-order-invariant component-extraction fallback (`_QUOTA_COUNT_REGEX` + `_QUOTA_KIND_REGEX`).

10. **Closure-prose page (p. 7) is two-column.** Right-column bbox crop required: `page.crop((306, 0, 612, 792))`. Default `extract_text` reads both columns top-to-bottom and interleaves them incorrectly.

11. **Phone number in quota-closure paragraph split across lines** (`"1-800-385-\n7826"`). `_DIGIT_LINEBREAK_RE` added to collapse digit-bordered hyphenated line breaks before BMU scanning.

12. **Drift guard for closure BMU list scoped to first sentence only** — phone-number digit fragments (385, 444, 800) would otherwise contaminate the BMU set comparison if the full paragraph text were scanned.
