## Scope Decision: V1 Species Only

The Legal Descriptions PDF (`mt-fwp-legal-descriptions-2026-2027-2026-02-03.pdf`) covers **nine** species sections; only **three** have matching geometry rows in the V1 Montana DB.

| Section | PDF pages | V1 in scope? | Namespace |
|---|---|---|---|
| Region Legal Descriptions | 3–4 | No | (no geometry rows) |
| Antelope HDs | 5–9 | **Yes** | `antelope` |
| Bighorn Sheep HDs | 10–12 | No | (no geometry rows) |
| Bison HDs | 13 | No | (no geometry rows) |
| Black Bear MUs | 14–18 | **Yes** | `bear` |
| Deer and Elk HDs | 19–35 | **Yes** | `deer-elk-lion` |
| Moose HDs | 36–42 | No | (no geometry rows) |
| Mountain Goat HDs | 43–45 | No | (no geometry rows) |
| Mountain Lion HDs | 46–53 | No | (no geometry rows) |
| Contacts | 54–56 | — | skip |

**Decision:** the extractor walks ONLY V1 pages (5–9, 14–18, 19–35). Out-of-scope sections are NOT processed and their headings are NOT surfaced as `unmatched`. This keeps the 10% fail-loud threshold meaningful — the rate is computed over V1 candidates only.

**Rationale:** without scoping, ~108 of ~215 headings (>50%) would land in `unmatched` (Bighorn 18 + Bison 4 + Moose 36 + Goat 26 + Lion 24), trivially tripping the threshold despite no parser regression.

## Threshold Denominator: Excludes "Portion of HD NNN" Sub-Headings

The PDF contains ~29 "Portion of HD NNN" sub-headings (e.g., `Portion of HD 103 North Fisher:`, `Portion of HD 170 Valley Portion:`). These map to `portion` geometry rows whose IDs end in opaque slugs (`elPt22`, `wtPt12`, `portion-of-hd-103-north-fisher`, etc.) that are **not derivable from heading text alone**.

The plan documented portions as explicitly out of V1 matcher scope. T7's matcher returns `("unmatched", {"reason": "portion_sub_heading_not_resolvable_to_opaque_slug"})` for these blocks.

**Decision:** T9 excludes the `portion_sub_heading_not_resolvable_to_opaque_slug` reason from BOTH the threshold numerator and denominator. The threshold detects HD/CWD parser regressions; counting by-design unresolvable sub-headings would trip the threshold on every run, defeating its purpose.

The spec invited this calibration (§ 483): "The 10% threshold is the V1 first-cut; tighten or loosen during S03.5 implementation if reality shows it should be different."

`extract_legal_descriptions.py` `main()` logs:
- `unmatched_regression` (HD/CWD mismatches — counted toward threshold)
- `unmatched_portion` (by-design unresolvable — surfaced for human review, NOT counted)
- `threshold_rate` (the rate over regression-unmatched only)

## Citation ID Slug: Biennial vs. Annual

The citation id in `sources.yaml` is `mt-fwp-legal-descriptions-2026-2027` (biennial slug, per the URL pattern). The output artifact is `legal-descriptions-2026.json` (annual slug per spec, consistent with `dea-2026.json` and `black-bear-2026.json`).

**Rationale for keeping the mismatch:** the artifact's `source_id` field carries the canonical biennial citation id so S03.6 can look up the citation correctly. The filename annual slug is consumer-facing convention; the source id is the authoritative pointer.

**Open question for M2:** if the 2027 PDF turns out to be a separate annual booklet rather than a continuation of the 2026/2027 biennial, the citation slug will need a re-anchor. Validate at first operator re-fetch.

## Discoveries Documented During Implementation

### D1 — Three-column PDF layout

Each content page (5–53) is divided into three columns at approximately:
- col1: x 36–195pt
- col2: x 210–355pt
- col3: x 390–555pt

`pdfplumber.Page.extract_text()` on the full page interleaves columns and is unreadable. The extractor crops per-column (T3's `_extract_three_column_text`) and concatenates the three column streams in left-to-right order before applying heading regex.

### D2 — Rotated sidebar banners are non-upright glyphs

Every content page carries a vertical chapter label (e.g., `klE & reeD`, `epoletnA`) as non-upright text in the outer margin. These appear in column crops unless filtered. The extractor filters via `page.crop(bbox).filter(lambda c: c.get("upright", True))`.

### D3 — CWD heading text wraps across lines

The narrow column width causes the CWD anchor phrase ("Those portions of...") to wrap onto a continuation line — and even the trailing word "Zone:" of the Kalispell heading wraps. The T4 regex (`_CWD_HEADING_RE`) cannot require the colon or anchor phrase; it matches only the zone-name prefix:

```
^\s*(Libby CWD Management Zone|Kalispell Area CWD Management(?:\s+Zone)?)\b
```

T7's `_canonicalize_cwd_name` appends " Zone" to a captured name lacking the suffix, producing the canonical lookup key.

### D4 — Libby CWD heading appears once per column

On page 19, the Libby zone's description spans all three columns. The heading text appears in each column (with three different body fragments). T6's `_consolidate_cwd_blocks` merges all blocks with the same `cwd_name` into a single block whose body concatenates the three fragments in column order.

After consolidation, the Libby block's `verbatim_description` is ~4,342 chars (compared to ~706 for the Kalispell block, which spans one column on page 21).

### D5 — HD heading regex must exclude newline from name capture

The original regex `[^:]{3,80}?` for the HD name was greedy across newlines. A prior-body fragment like `"93 in Missoula County."` immediately preceding `"261 East Bitterroot: That portion..."` would let the regex capture `"93"` as the HD number with the lazy name span absorbing `"in Missoula County.\n261 East Bitterroot"`.

Result: HD 93 (not in DB) surfaced as unmatched; HD 261 (in DB) was silently absorbed into the false-positive match. The same pattern hit HD 262.

**Fix:** changed `[^:]{3,80}?` → `[^:\n]{3,80}?`. After the fix, matched count went from 141 (with 2 duplicate IDs from corrupted false matches) → 139 (unique IDs, no duplicates); regression-unmatched went 2 → 0. Locked by `TestArtifactRegression::test_regression_unmatched_rate_below_10_percent`.

### D6 — Pages 10–13 + 36–53 walked-but-out-of-scope

Bighorn, Bison, Moose, Goat, Lion sections are NOT walked by `_walk_v1_pages` (the `_V1_SECTIONS` tuple omits them). Their headings would all be unmatched and are out of scope; surfacing them inflates the threshold rate without adding signal.

## UAT Package (≥2 descriptions for human review)

The S03.5 acceptance criteria require operator UAT against the PDF for ≥2 descriptions — one HD and one CWD or restricted-area. Recommended candidates from the matched artifact:

### Candidate 1: HD 215 (Antelope) — `MT-HD-antelope-215-geom`

- Source PDF page: 5 (col 1 of antelope section)
- Heading text in PDF: `215 East Deer Lodge: Those portions of Granite County lying within the following described boundary...`
- Extracted `verbatim_description` length: ~457 chars
- UAT check: open the PDF, locate the heading on page 5, compare the prose to the artifact's `matched[]` entry for `MT-HD-antelope-215-geom`. Confirm all source words are byte-exact preserved; only whitespace differs (Cleanup Rule A collapse).

### Candidate 2: Libby CWD Management Zone — `MT-CWD-zone-libby-cwd-management-zone-geom`

- Source PDF page: 19 (heading appears in all three columns)
- Heading text in PDF: `Libby CWD Management Zone: Those portions of Lincoln county lying within the following described boundary...`
- Extracted `verbatim_description` length: ~4,342 chars (concatenation of three column fragments)
- UAT check: open PDF page 19, verify the boundary description across all three columns is captured in the merged `verbatim_description` field. The merged body preserves all three column streams' content separated by single spaces (Cleanup Rule A).

### Why two heading shapes?

Heading-to-geometry-id matching is heuristic. Validating one HD (regex-derived ID via species namespace) and one CWD (hardcoded lookup with name canonicalization) exercises both matching paths. If both UAT samples pass, the matching algorithm is faithful.

## Re-Run Determinism

`legal-descriptions-2026.json` is byte-deterministic across re-runs when the input PDF + manifest are unchanged. The extracted_at timestamp comes from the manifest, not `datetime.now()`. Re-extracting with identical inputs produces a byte-identical artifact.

## Anomalies Surfaced for Future Stories

### MT-STATEWIDE-geom missing from `geometry-overlays.json`

The state-boundary geometry row `MT-STATEWIDE-geom` (kind=`state`) was written to production via S03.0 but is NOT present in the V1 fixture. The fixture predates S03.0. S03.5's `_load_geometry_lookup` does NOT include it in `all_v1_ids`, so it does NOT appear in `unlinked`.

For S03.6: the `unlinked` array of `legal-descriptions-2026.json` does not include `MT-STATEWIDE-geom`. S03.6 will need to handle the state row separately (it has no legal description in this PDF; that's documented and correct).

### Two HDs absent from PDF but present in DB (now surfaced in `unlinked`)

The 98-row `unlinked` array contains a small number of HDs whose headings genuinely don't appear in the Legal Descriptions PDF (separate from the 55 portion and 54 restricted-area entries). This is expected — the PDF is a published booklet that may not cover every HD/BMU. These will need M2 follow-up if any one of them is later flagged by the response composer for missing prose.
