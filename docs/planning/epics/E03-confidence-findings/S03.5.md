## Scope Decision: V1 Species Only

The Legal Descriptions PDF (`mt-fwp-legal-descriptions-2026-2027-2026-02-03.pdf`) covers **nine** species sections; only **three** have matching geometry rows in the V1 Montana DB.

| Section | PDF pages | V1 in scope? | Namespace |
|---|---|---|---|
| Region Legal Descriptions | 3–4 | No | (no geometry rows) |
| Antelope HDs | 5–9 | **Yes** | `antelope` |
| Bighorn Sheep HDs | 10–12 | No | (no geometry rows) |
| Bison HDs | 13 | No | (no geometry rows) |
| Black Bear MUs | 14–18 | **Yes** | `bear` |
| Deer and Elk HDs | 19–35 | **Yes** | `deer-elk-lion` |
| Moose HDs | 36–42 | No | (no geometry rows) |
| Mountain Goat HDs | 43–45 | No | (no geometry rows) |
| Mountain Lion HDs | 46–53 | No | (no geometry rows) |
| Contacts | 54–56 | — | skip |

**Decision:** the extractor walks ONLY V1 pages (5–9, 14–18, 19–35). Out-of-scope sections are NOT processed and their headings are NOT surfaced as `unmatched`. This keeps the 10% fail-loud threshold meaningful — the rate is computed over V1 candidates only.

**Rationale:** without scoping, ~108 of ~215 headings (>50%) would land in `unmatched` (Bighorn 18 + Bison 4 + Moose 36 + Goat 26 + Lion 24), trivially tripping the threshold despite no parser regression.

## Threshold Denominator: Excludes "Portion of HD NNN" Sub-Headings

The PDF contains ~29 "Portion of HD NNN" sub-headings (e.g., `Portion of HD 103 North Fisher:`, `Portion of HD 170 Valley Portion:`). These map to `portion` geometry rows whose IDs end in opaque slugs (`elPt22`, `wtPt12`, `portion-of-hd-103-north-fisher`, etc.) that are **not derivable from heading text alone**.

The plan documented portions as explicitly out of V1 matcher scope. T7's matcher returns `("unmatched", {"reason": "portion_sub_heading_not_resolvable_to_opaque_slug"})` for these blocks.

**Decision:** T9 excludes the `portion_sub_heading_not_resolvable_to_opaque_slug` reason from BOTH the threshold numerator and denominator. The threshold detects HD/CWD parser regressions; counting by-design unresolvable sub-headings would trip the threshold on every run, defeating its purpose.

The spec invited this calibration (§ 483): "The 10% threshold is the V1 first-cut; tighten or loosen during S03.5 implementation if reality shows it should be different."

`extract_legal_descriptions.py` `main()` logs:
- `unmatched_regression` (HD/CWD mismatches — counted toward threshold)
- `unmatched_portion` (by-design unresolvable — surfaced for human review, NOT counted)
- `threshold_rate` (the rate over regression-unmatched only)

The citation id in `sources.yaml` is `mt-fwp-legal-descriptions-2026-2027` (biennial slug, per the URL pattern). The output artifact is `legal-descriptions-2026.json` (annual slug per spec, consistent with `dea-2026.json` and `black-bear-2026.json`).

**Open question for M2:** if the 2027 PDF turns out to be a separate annual booklet rather than a continuation of the 2026/2027 biennial, the citation slug will need a re-anchor. Validate at first operator re-fetch.

## Discoveries Documented During Implementation

### D1 — Three-column PDF layout

Each content page (5–53) is divided into three columns at approximately:
- col1: x 36–195pt
- col2: x 210–355pt
- col3: x 390–555pt

`pdfplumber.Page.extract_text()` on the full page interleaves columns and is unreadable. The extractor crops per-column (T3's `_extract_three_column_text`) and concatenates the three column streams in left-to-right order before applying heading regex.

### D2 — Rotated sidebar banners are non-upright glyphs

Every content page carries a vertical chapter label (e.g., `klE & reeD`, `epoletnA`) as non-upright text in the outer margin. These appear in column crops unless filtered. The extractor filters via `page.crop(bbox).filter(lambda c: c.get("upright", True))`.

### D3 — CWD heading text wraps across lines

The narrow column width causes the CWD anchor phrase ("Those portions of...") to wrap onto a continuation line — and even the trailing word "Zone:" of the Kalispell heading wraps. The T4 regex (`_CWD_HEADING_RE`) cannot require the colon or anchor phrase; it matches only the zone-name prefix:

```
^\s*(Libby CWD Management Zone|Kalispell Area CWD Management(?:\s+Zone)?)\b
```

T7's `_canonicalize_cwd_name` appends " Zone" to a captured name lacking the suffix, producing the canonical lookup key.

### D4 — Libby CWD heading appears once per column

On page 19, the Libby zone's description spans all three columns. The heading text appears in each column (with three different body fragments). T6's `_consolidate_cwd_blocks` merges all blocks with the same `cwd_name` into a single block whose body concatenates the three fragments in column order.

After consolidation, the Libby block's `verbatim_description` is ~4,342 chars (compared to ~706 for the Kalispell block, which spans one column on page 21).

### D5 — HD heading regex must exclude newline from name capture

The original regex `[^:]{3,80}?` for the HD name was greedy across newlines. A prior-body fragment like `"93 in Missoula County."` immediately preceding `"261 East Bitterroot: That portion..."` would let the regex capture `"93"` as the HD number with the lazy name span absorbing `"in Missoula County.\n261 East Bitterroot"`.

Result: HD 93 (not in DB) surfaced as unmatched; HD 261 (in DB) was silently absorbed into the false-positive match. The same pattern hit HD 262.

**Fix:** changed `[^:]{3,80}?` → `[^:\n]{3,80}?`. After the fix, matched count went from 141 (with 2 duplicate IDs from corrupted false matches) → 139 (unique IDs, no duplicates); regression-unmatched went 2 → 0. Locked by `TestArtifactRegression::test_regression_unmatched_rate_below_10_percent`.

### D7 — HD heading anchor phrase wraps across newline

A second regex defect, surfaced 2026-05-13: HD 705 (Prairie/Pines-Juniper Breaks) reads `"705 Prairie/Pines-Juniper Breaks: Those\nportions of Carter, ..."` in the column-cropped stream — the literal space between `"Those"` and `"portions"` was a line break, so the original regex (`Those portions?` with a literal space) failed to match. HD 705 silently surfaced in `unlinked` while HD 704's body absorbed the 705 heading text.

**Fix:** changed `Those portions?` → `Those\s+portions?` (and same for `That\s+portion`). The whitespace-flex anchor matches across line breaks while still rejecting arbitrary text between the words (`\s` only matches whitespace chars). Locked by `TestHeadingRegex::test_hd_anchor_phrase_wraps_across_newline` and `TestArtifactRegression::test_hd_705_antelope_matched_via_wrapped_anchor`.

Matched count: 139 → 156 (+17 HDs whose headings had wrapped anchors).

### D8 — Footer strip too shallow; running footer leaked into HD bodies

The Legal Descriptions PDF carries a running `"Visit fwp.mt.gov <page#>"` footer on every content page (top ≈ 716 on a 756pt-tall page). The initial `_FOOTER_STRIP_PT = 20.0` produced a crop cutoff at `page.height - 20 = 736` — above the footer. HD 704's `verbatim_description` ended with `"Visit fwp.mt.gov 9"` (the last antelope HD on page 9).

**Fix:** increased `_FOOTER_STRIP_PT` from 20.0 → 50.0, putting the cutoff at top=706 — well above the footer's top y of 716. Locked by `TestArtifactRegression::test_no_running_footer_leak_in_verbatim_descriptions`.

### D9 — Locator clause leaked into bodies; Cleanup Rule C added

Beyond the heading number+name (which the regex correctly excluded), the FWP "locator clause" — `"of <county list> lying within the following[ -]described boundary:"` — bled into every matched body because the regex consumed only `"Those portions"` / `"That portion"`, not the entire structural prefix. Before Rule C, all matched bodies opened with structural metadata like `"of Granite County lying within the following described boundary: Beginning at..."` (or `": Thos portions of..."` for CWDs).

Per the V1 spec (epic line 475), `verbatim_description` is the "full prose boundary description, verbatim" — the prose, not the locator metadata. The boundary description itself always opens with `"Beginning at..."` (or its column-truncated / lowercase variants `"Beginnin"` / `"Begin"` / `"Beg"` / `"beginning"`).

**Fix:** added **Cleanup Rule C**: search body for `\b[Bb]eg\w*\s+\w` and slice to that position. The `\s+\w` discriminator avoids matching the end-of-body `"the point of beginning."` (period after "beginning" means no word char follows). If no anchor is found, log a WARNING naming the geometry_id and preserve body unmodified (ADR-001 fail-safe). Discovered 2026-05-13; extended to lowercase 2026-05-14 (HD bear-309 reads `"beginning at the intersection..."`). Locked by `TestBuildArtifact::test_locator_clause_stripped`, `TestBuildArtifact::test_locator_clause_stripped_with_truncated_beg`, `TestBuildArtifact::test_rule_c_fail_safe_when_no_beg_anchor`, and `TestArtifactRegression::test_all_matched_bodies_start_with_boundary_prose`.

### D10 — Heading regex missed 8 FWP variants; major regex overhaul

Post-merge review (2026-05-14) surfaced that 35 matched entries carried foreign HD content (one body absorbed 5 complete HD descriptions) and 53 HDs that exist in the PDF were silently in `unlinked`. Root cause: the original `_HD_HEADING_RE` required `"Those portions"` / `"That portion"` as a contiguous anchor phrase after the colon, missing 7 additional FWP heading patterns:

1. **Canonical** (already matched): `"215 East Deer Lodge: Those portions of ..."`.
2. **No locator clause** (newly matched): `"301 Hyalite-Portal: Beginning at ..."` — heading goes directly from colon to boundary prose; no `"Those portions"` anchor at all.
3. **"The" typo** (newly matched): `"302 Tendoys: The portion of ..."` — FWP-side typo for "That portion".
4. **Truncated anchor word** (newly matched): `"303 Lima Peaks-Nicholia: That p\n..."` — column wrap cuts the anchor word mid-letter.
5. **Wrapped HD name** (newly matched, secondary alternation): `"309 Gallatin Valley Weapons Rest\nArea: That portion of ..."` — the HD name spans two lines, no colon on the first line.
6. **No-colon heading** (newly matched, secondary alternation): `"311 Lower Gallatin-Madison-Horseshoe\nThose portions of Madison, Park, G\n..."` — heading line has NO colon at all; locator clause starts on next line.
7. **"HD NNN" prefix** (newly matched): `"HD 525 Beartooth/Absaroka: Those portions ..."` — FWP-side variant using "HD " prefix before the number.
8. **Name with embedded period** (newly matched): `"200 North St. Regis: Those portions of Mineral and Sanders ..."` — the "St." abbreviation contains a period.

The header strip was also reduced from 40pt → 25pt (no running title actually exists; the 40pt strip was clipping HD heading lines that sit at top≈29pt of each column). The optional `_PORTION_HEADING_RE` was relaxed to accept `"Portion of NNN ..."` without an intermediate `"HD "` (PDF uses both forms).

**Fix summary:**

```python
# Two alternations: primary (colon-terminated) + secondary (no-colon)
_HD_HEADING_RE = re.compile(
    # Primary: heading line ends with ":" + anchor lookahead
    r"^\s*(?:HD\s+)?(?P<num_a>\d{2,3})\s+"
    r"(?P<name_a>[^:\n]{3,80}?(?<!\.)(?:\n[^:\n.\d][^:\n.]{0,39})?)(?<!\.):"
    r"\s+(?=(?:Beg|Th|HD)\w*)"
    r"|"
    # Secondary (case 6): heading line has NO colon; anchor on next line
    r"^\s*(?:HD\s+)?(?P<num_b>\d{2,3})\s+"
    r"(?P<name_b>[A-Z][^:\n]{2,80}(?<!\.))"
    r"\s*\n\s*(?=(?:Th|Beg)\w*)",
    re.MULTILINE,
)
```

Two `(?<!\.)` lookbehinds prevent body-text bleed: a name part ending in `"."` cannot be followed by a wrap continuation or a colon. Without these, body fragments like `"41 to Twin Bridges, the point of beginning.\nNOTE: ..."` would match as if `"41"` were an HD number.

Matched count: 156 → 228 (+72 HDs from variants 2-8 newly captured). Regression-unmatched: 0. Bleed cases: 35 → 0. Locked by `TestHeadingRegex::test_hd_regex_rejects_body_text_followed_by_note_line`, `test_hd_regex_matches_hd_prefix_form`, `test_hd_regex_matches_name_with_period`, `test_hd_regex_matches_no_colon_heading_via_secondary_alt`.

### D11 — CWD consolidation must canonicalize names before deduping

`_consolidate_cwd_blocks` keyed dedup on raw `block.cwd_name`. A future PDF where two CWD heading instances have different raw captures that canonicalize to the same key (e.g., `"Libby CWD Management Zone"` vs `"Libby CWD Management"` truncated) would emit duplicate matched entries with the same `geometry_id` — last-write-wins corruption in S03.6.

**Fix:** consolidate using `_canonicalize_cwd_name(block.cwd_name)` as the dedup key, matching the matcher's lookup behavior. Defensive fix; current PDF doesn't trigger it (artifact byte-identical to pre-fix), but the contract is now lock-step with the matcher.

### D6 — Pages 10–13 + 36–53 walked-but-out-of-scope

Bighorn, Bison, Moose, Goat, Lion sections are NOT walked by `_walk_v1_pages` (the `_V1_SECTIONS` tuple omits them). Their headings would all be unmatched and are out of scope; surfacing them inflates the threshold rate without adding signal.

## Anomalies Surfaced for Future Stories

### MT-STATEWIDE-geom is hardcoded into `all_v1_ids` (fixture predates S03.0)

The state-boundary geometry row `MT-STATEWIDE-geom` (kind=`state`) was written to production via S03.0 but is NOT present in the V1 `geometry-overlays.json` fixture (the fixture predates S03.0). To honor the module docstring's contract that all V1 geometry rows land in `unlinked` when they have no matching heading, `_load_geometry_lookup` hardcodes the `MT-STATEWIDE-geom` row into `all_v1_ids`.

For M2 / future fixture refresh: when `geometry-overlays.json` is regenerated to include the post-S03.0 state row, the hardcoded injection can be removed (the loader's dedup-via-`setdefault` already prevents double-counting). For now, the hardcode is documented inline and locked by `TestGeometryLookup::test_statewide_present_even_when_fixture_omits_it`.

### HDs absent from PDF but present in DB

The `unlinked` array contains 98 HD rows whose headings genuinely don't appear in the Legal Descriptions PDF (separate from the 55 portion and 54 restricted-area entries and the 1 state row). This is expected — the PDF is a published booklet that may not cover every HD/BMU. These will need M2 follow-up if any one of them is later flagged by the response composer for missing prose.
