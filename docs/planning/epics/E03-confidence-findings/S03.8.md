# S03.8 working note — draw_spec ingestion

**Story scope:** Third E03 database-write adapter. Reads `dea-2026.json` (already committed by S03.3); writes `draw_spec` rows for every limited-draw `license_tag` from S03.7 (except `900-20`) and backfills `license_tag.draw_spec_key` in the same atomic transaction. Also applies a T1 amendment to S03.7's `_build_dea_license_tags` heuristic to reclassify 160 "B License" identities from `limited_draw` to `over_the_counter` before S03.8's draw_spec builder runs.

## 1. Story summary

S03.8 is specified at `docs/planning/epics/E03-regulation-text-ingestion.md` lines 726-779. It writes **390 `draw_spec` rows** for the DEA limited-draw license_tag identities confirmed by the post-amended-heuristic probe (see § 3 below), then backfills `license_tag.draw_spec_key` via `UPDATE license_tag SET draw_spec_key = ? WHERE id = ?` after all `draw_spec` rows are committed. Also installs `_KNOWN_CROSS_LISTING_OVERRIDES` with one V1 entry (`Elk B License: 210-03` → quota=300) to resolve the single cross-listing structural conflict surfaced during Stage 6 review (see `## Stage 6 PDF investigation` below). The `900-20` statewide antelope license_tag receives no `draw_spec` row per Q12 deferral (epic lines 748-761); its `license_tag.draw_spec_key` stays NULL and `verbatim_rule` carries the full "First and only choice. ArchEquip only." text. S03.8 also amends `load_seasons_and_licenses.py:_build_dea_license_tags` (T1) to fix the OTC misclassification discovered during Stage 2 probing, ensuring the 160 reclassified identities are corrected in the DB via S03.8's Phase 1 re-upsert before any draw_spec rows are written.

## 2. Path 2-folded decision context

The T1 heuristic amendment was originally a candidate for a standalone S03.7-amendment story. PM resolved 2026-05-17 to fold it into S03.8 rather than split it out: the OTC reclassification is load-bearing for S03.8 correctness (the draw_spec builder must not emit rows for OTC identities), the fix is surgical (a pre-pass + one branch change in `_build_dea_license_tags`), and the `upsert_license_tag` call at S03.8 Phase 1 naturally writes the corrected `kind` to existing DB rows via `kind = EXCLUDED.kind`. A separate story would have required its own PR, plan, and DB re-run with no additional safety over folding it.

## 3. OTC discovery (Stage 2 probe finding)

**Finding:** 160 DEA "B License" `license_tag` identities had `apply_by` cells reading `"OTC:\nJun 15"` — they are over-the-counter purchases with an application window, not lottery drawings. S03.7's original 5-branch heuristic classified them as `kind='limited_draw'` because it inspected only `license_code` (specifically: any code matching the `B License: DDD-DD` pattern) without consulting `apply_by`.

**Cross-row OTC-wins discipline:** Any `(species, hd_number, license_code)` identity with at least one artifact row showing the substring `"OTC"` in `apply_by` reclassifies as `over_the_counter` for ALL rows of that identity. A single OTC row is sufficient to override limited-draw classification for the identity.

**Post-amended-heuristic, post-dedup identity counts (PM-confirmed baseline 2026-05-17):**

| kind | identity count |
| --- | --- |
| `limited_draw` | 390 |
| `over_the_counter` | 160 |
| `general` | 239 |
| `statewide` | 1 |
| **TOTAL DEA** | **790** |

This matches the 790 unique-id count from S03.7's cubic-review fix. The 390 `limited_draw` identities are the draw_spec population for S03.8.

## Stage 6 PDF investigation: HD 210 cross-listing conflict

Stage 6's silent-failure-hunter surfaced exactly 1 structural conflict among
the 32 cross-listed `(hunt_code, year)` PKs from S03.8's 390 (license_tag,
draw_spec) pairs:

| PK | License_tag id | Quota | Source |
| --- | --- | --- | --- |
| `('Elk B License: 210-03', 2026)` | `MT-HD-210-elk-elk-b-210-03-2026` | **300** | DEA p. 53 row 7 (home HD) |
| | `MT-HD-211-elk-elk-b-210-03-2026` | 200 | DEA p. 53 row 21 (cross-list) |
| | `MT-HD-212-elk-elk-b-210-03-2026` | 200 | DEA p. 54 row 12 (cross-list) |
| | `MT-HD-216-elk-elk-b-210-03-2026` | 200 | DEA p. 57 row 13 (cross-list) |

**OPPORTUNITY SPECIFIC DETAILS column text (identical on all 4 rows):**
> "Valid on private lands in HDs 211, 212, 216 and south portion of 210
> (Rattling Gulch-Henderson Creek)."

**Outcome of PM-directed 15-min PDF probe (2026-05-18):** confirmed X1 (real
per-HD-cap semantic). Home-HD quota=300 = total drawable; cross-listed
quota=200 = per-HD allocation cap. NOT an S03.3 extraction defect (PDF text
matches the artifact exactly).

**V1 resolution:** `_KNOWN_CROSS_LISTING_OVERRIDES[("Elk B License: 210-03", 2026)]
= {"quota": 300, "rationale": ...}` in `load_draw_specs.py`. The validator
`_validate_cross_listing_consistency` applies the override + WARN log;
canonical quota=300 written to DB; cross-listed cap=200 values dropped.

**Per-HD-cap structural gap → Q17.** The current `draw_spec` schema cannot
model jurisdiction-keyed per-HD allocation caps. Documented in
[docs/open-questions.md Q17](../../open-questions.md) (M2 ADR-candidate) and
[docs/planning/epics/E03-deferred-items/draw-mechanics.md](../E03-deferred-items/draw-mechanics.md)
§ "Per-HD allocation caps for cross-listed B licenses".

- "Antelope (apply by) ..............June 1" → `("antelope", "license"): 2026-06-01`, `("antelope", "b_license"): 2026-06-01`

**DEA p. 10** (Resident License Chart):

- "Deer B License - Drawing only ... June 1" → `("deer", "b_license"): 2026-06-01`
- "Elk B License - Drawing only ... June 1" → `("elk", "b_license"): 2026-06-01`

**DEA p. 11** (Nonresident License Chart): same values — cross-validates the resident chart entries on p. 10.

The lookup keyed on `(species_group, license_kind_token)` where `license_kind_token` is derived from `license_tag.kind` + `license_code` prefix. V1 expected **zero safety-net hits** (confirmed by `test_lookup_zero_hits_against_real_artifact` in `tests/test_load_draw_specs.py`). Any firing in production is a drift signal: either an S03.3 extraction regression or a new license type in a future PDF revision that bypasses per-row `apply_by` inspection.

## 5. Hardcoded V1 defaults rationale

**`choices.count=1, points_used_in_choices=[1]`** — MT has no preference-point system. "First and only choice" semantics for `900-20` differ from preference-point ranks and are deferred to M2 per Q12 (see `docs/planning/epics/E03-deferred-items/draw-mechanics.md`). For V1 limited-draw B licenses there is one application slot; `count=1` and `points_used_in_choices=[1]` are the schema-minimal faithful representation.

**`pools = [AllocationPool(share=1.0, selection="unweighted_random")]`** — Single pool elides MT's 10% nonresident cap (MCA Title 87-2-106). A two-pool model (resident 90% / nonresident 10%) is architecturally supported by `AllocationPool` but requires per-hunt residency-distribution data not present in the DEA PDF. See `docs/planning/epics/E03-deferred-items/draw-mechanics.md` for the deferred M2 entry.

**`draw_phase = "primary"`** — MT Surplus Drawing (leftover allocation after primary draw) is not extracted from S03.3; it would require a new extraction pass on a different DEA section. All V1 rows are primary-phase.

**`parameters = None`** — Q12 hard-defer per epic lines 748-761. Any case requiring `parameters` triggers flag-and-defer: WARN log + entry in `docs/planning/epics/E03-deferred-items/draw-mechanics.md` + run-summary count. V1 expected zero hits.

## 6. S03.7 amendment (T1) summary

**File changed:** `ingestion/states/montana/load_seasons_and_licenses.py`, function `_build_dea_license_tags`.

**Change:** A pre-pass computes `_otc_identities: frozenset[tuple[str, str, str]]` — the set of `(species_group, hd_number, license_code)` tuples where any artifact row for that identity has `apply_by` containing the substring `"OTC"`. The per-row "B License" branch then consults the set:

```python
kind = "over_the_counter" if identity in _otc_identities else "limited_draw"
```

All other branches (STATEWIDE, General, `Permit:`, `Antelope License:`, fail-loud else) are unchanged.

**Effect:** 160 license_tag identities reclassified at the next re-run of `_build_dea_license_tags`. S03.8's Phase 1 re-upsert writes the corrected `kind` to existing DB rows via `upsert_license_tag`'s `kind = EXCLUDED.kind` clause. This must complete before S03.8's draw_spec builder runs, so the builder sees only the 390 true `limited_draw` identities.
