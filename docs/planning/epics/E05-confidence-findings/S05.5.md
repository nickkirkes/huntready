## Group B procedure (operator runs post-merge)

Precondition: S05.2 (`load_gmus.py`) and S05.4 (`load_restricted_areas.py`) have been run
against the production project so the CO `geometry` rows exist (≈186 GMUs + 10 restricted
areas + `CO-STATEWIDE-geom`).

### 1. Set environment

```bash
export DATABASE_URL="postgresql://...service-role DSN..."
```

### 2. Run the builder

```bash
cd /Users/nickkirkes/rowdycloud/code/huntready
ingestion/.venv/bin/python ingestion/states/colorado/build_overlay_fixture.py
```

Writes both `ingestion/states/colorado/fixtures/geometry-overlays.json` (kept rows) and
`ingestion/states/colorado/fixtures/geometry-overlays-dropped.json` (audit log). The build
fails loud if any `cwd_zone`/`restricted_area` geometry is an orphan not on the allowlist,
or if any fixture-referenced id is missing from the loaded geometry set.

### 3. Threshold-recalibration check (per Spatial Correctness SHOULD-FIX / ADR-016 §4)

Inspect `geometry-overlays-dropped.json` plus the kept fixture's `intersects` rows:

- **Borderline drops:** `overlap_pct` in `[0.005, 0.02]`
- **Borderline relabels:** `overlap_pct` in `[0.98, 0.995]`

If either band's count differs from MT proportions by **>10%**, recalibrate
`COVER_RELABEL_THRESHOLD` / `COVER_DROP_THRESHOLD` per ADR-016 §4 and update the four
numeric values in the threshold edge-tests symmetrically (the test pattern is the lock, not
the literal numbers). Otherwise MT thresholds carry forward unchanged. Record the decision
in this note's "Group B verification record" slot.

### 4. UAT visual spot-check (S05.5 is a UAT-gating story)

Confirm in the generated fixture:

- **Every** `kind='gmu'` row has a `self` relationship with `role_for_e03='primary_unit'`
  (programmatic — cannot orphan).
- **Multi-part GMU self-rows** — cross-reference `multipart-gmus.json` (S05.2 Group B
  output): each multi-part GMU still has exactly one self-row.
- **The 10 restricted-area ids appear as orphans** (in `EXPECTED_CO_RA_ORPHAN_IDS`,
  INFO-logged at build time) rather than as GMU children — consistent with
  `restricted-area-discovery.md`. If any of the 10 instead appears as a GMU child
  (covers/intersects), that is new information worth recording (a park that genuinely
  overlaps a GMU); it does not fail the build (the allowlist is permissive for orphans).
- **CWD coverage is vacuously satisfied** — zero `cwd_zone` rows (S05.3 documented gap), so
  no GMU↔CWD relationships are expected.

### 5. Commit the live-run fixtures

```bash
git add ingestion/states/colorado/fixtures/geometry-overlays.json \
        ingestion/states/colorado/fixtures/geometry-overlays-dropped.json
git commit -m "chore(S05.5): commit Colorado overlay fixtures from live build"
```

Both files are byte-deterministic across runs (`sort_keys=True`, `indent=2`,
`ensure_ascii=False`, trailing newline, sorted lists, `overlap_pct` rounded to 6 decimals,
atomic tmp+rename) — re-running the builder against the same DB state produces no diff.

## `_JURISDICTION_BINDING_ID_FORMAT` contract for E06

The builder's module docstring documents the contract E06's CO binding loader must honor:
import `_JURISDICTION_BINDING_ID_FORMAT = "{state}-{jurisdiction_code}-{species_group}-{license_year}-{role}-{geometry_id}"`
from S03.6.1's `load_regulation_records.py` (or its CO equivalent) so `CO-STATEWIDE-{species}`
binding ids derive symmetrically — preserving the UPSERT no-op contract. Statewide
`(state, state)` rows are **not** pre-emitted in this fixture (per Schema Stress-Test
SHOULD-FIX); E06 reads `CO-STATEWIDE-geom` from the geometry table at binding-derivation
time, mirroring S03.6.1's MT-STATEWIDE-bear pattern.

## Line-citation-drift flags found during discovery (epic vs reality)

Recorded per Bundle A pitfall #1 (line-citation drift is the #1 E05 failure mode). The epic
prose at S05.5 (lines 482-515) carried two snippet inaccuracies that the implementation
corrected against the actual `overlays.py`:

1. **Placeholder types** — the epic snippet typed the new mapping
   `Mapping[OverlayChildKind, GeometryRole]`. Neither `Mapping` nor a type named
   `GeometryRole` exists in `overlays.py`. The actual (and shipped) annotation is
   `dict[OverlayChildKind, GeometryRoleForE03]`, matching the existing
   `ROLE_FOR_E03_BY_CHILD_KIND` definition.
2. **Dropped `hunting_district` entry** — the epic's `ROLE_FOR_BINDING_BY_CHILD_KIND` snippet
   (lines 504-509) omitted the `"hunting_district" → "primary_unit"` entry, but AC line 548
   requires preserving it. The shipped dict keeps both `hunting_district` and `gmu` → `primary_unit`.

Both are snippet/illustration drift, not blocking design errors; the ACs themselves
(lines 547-563) describe the correct behavior.

## Stage-6 review record (Group A)

Parallel triad (code-reviewer + static-analysis + silent-failure-hunter) returned **no
Critical findings**. Applied in one review-fix cycle:

- **W1 (convergent — SFH + code-reviewer):** strengthened `test_all_children_covered_no_raise`
  with a `caplog` assertion that the ADR-016 orphan INFO message is NOT emitted when a
  restricted_area is genuinely GMU-covered. Because all 10 CO RAs are allowlisted, a broken
  `parent_kind == "hunting_district"` copy-paste in `_validate_coverage` would have let the
  coverage check pass vacuously (covered → allowlisted-orphan, no raise). The absence of the
  orphan log line is now the lock that the `parent_kind == "gmu"` filter actually works.
- **I1:** `DroppedOverlayPair.parent_kind` docstring (`overlays.py`) corrected from
  "Always `hunting_district` in V1" → MT `hunting_district` / CO `gmu`.
- **I2 (convergent):** `OverlayFixtureRow.role_for_e03` docstring (`overlays.py`) now points
  at the canonical `ROLE_FOR_BINDING_BY_CHILD_KIND` (was the deprecated alias).

**W2 rejected (documented):** silent-failure-hunter proposed an `if ras and not ra_rows:
raise` count guard. Rejected — per `restricted-area-discovery.md` (line ~170) **all 10 CO
restricted areas are expected orphans** (NPs/NMs/AFA are adjacent to, not contained by,
GMUs), so zero `restricted_area` covers/intersects rows is the **expected** CO outcome. That
guard would false-positive-fail the build on the correct case. The existing no-GMU fail-loud
guard already covers the catastrophic empty/unpopulated-table scenario; CWD is a documented
S05.3 gap (zero rows). A live Group B run will confirm the actual orphan/child split.
