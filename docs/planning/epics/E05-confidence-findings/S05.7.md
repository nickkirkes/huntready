## `spatial-test-points.json` generation (the load-bearing item)

### Decision: Option A (spec-faithful, Group B operator output)

`ingestion/states/colorado/fixtures/spatial-test-points.json` is **NOT committed at-merge**. This mirrors S05.5's `geometry-overlays.json` (Group-B-pending) exactly.

**Why Option A:** AC #2 requires "real `shapely.representative_point()` from the actual loaded geometry — no invented points." At-merge, no CO geometry rows exist in the production DB (all S05.0/S05.2/S05.4 writes are themselves operator-pending). Authoring named lat/lng coordinates at-merge (Option B) would therefore constitute invented points, violating AC #2. Option A defers fixture generation to the operator's batched live-write session, producing coordinates derived from the real geometry via `ST_PointOnSurface`.

### Generator query (run in the production project after CO rows are loaded)

```sql
SELECT id, kind,
       extensions.ST_AsText(
           extensions.ST_PointOnSurface(
               extensions.ST_GeomFromText(extensions.ST_AsText(geom), 4326)
           )
       ) AS pt
FROM geometry
WHERE state = 'US-CO'
ORDER BY kind, id;
```

Cross-reference: `docs/runbooks/E05-colorado-geometry-verification.md` Section 0.

### Per-kind selection rules

From the query output, select:

- **GMUs (`kind='gmu'`):** ≥3 entries, including the first entry of `ingestion/states/colorado/fixtures/multipart-gmus.json` as the named multi-part anchor (the anchor is the load-bearing AC #703 test point). If `multipart-gmus.json` is empty at operator run time, document as N/A + data observation (do not silently skip).
- **Restricted areas (`kind='restricted_area'`):** ≥1 representative zone from the 10 V1 federal no-hunt zones. Note: all 10 CO restricted areas are `EXPECTED_CO_RA_ORPHAN_IDS` orphans (NPs/NMs/AFA are spatially adjacent to GMUs, not contained by them) — `ST_Covers` returns the zone's own row; a point inside a RA may ALSO return an adjacent GMU row if the zone overlaps a GMU boundary; both are correct.
- **Statewide (`kind='state'`):** 1 entry for `CO-STATEWIDE-geom` — use the `ST_PointOnSurface` output (any CO interior point).
- **CWD zones (`kind='cwd_zone'`):** **omitted** — CO publishes no CWD zone geometry (S05.3 documented gap, 2026-06-03). No CWD test point.
- **Portions (`kind='portion'`):** **omitted** — CO has no portion geometry; this kind does not exist in the CO adapter.
- **Outside-CO negative control:** exactly 1 point just over the WY or KS line (e.g., `lng=-104.0, lat=41.5` in Wyoming). This is the **only** non-`ST_PointOnSurface` coordinate in the fixture — it is documented as a deliberate human-authored negative control, not derived from `ST_PointOnSurface`, because it must lie outside CO geometry by definition. `ST_Covers` with `AND state = 'US-CO'` must return zero rows for this point.

### Fixture JSON schema

```json
{
  "version": 1,
  "points": [
    {
      "name": "<descriptive name, e.g. GMU 54 representative point>",
      "lat": <float>,
      "lng": <float>,
      "expected_kind": "<gmu|restricted_area|state>",
      "expected_id_pattern": "<full id or prefix pattern, e.g. CO-GMU-54-geom>",
      "expected_role_for_e03": null,
      "notes": "<optional; multi-part anchor, orphan RA, negative control, etc.>"
    }
  ]
}
```

**`expected_role_for_e03` is `null` for ALL CO points.** E05 writes zero `jurisdiction_binding` rows — bindings (including `role='no_hunt_zone'` on restricted areas per ADR-021) are entirely E06 territory. Carrying `null` here is correct and intentional; the field will remain `null` when E06 adds bindings because the fixture's purpose is geometry identity, not role verification.

## Decisions log

| Decision | Rationale |
|---|---|
| **Option A: `spatial-test-points.json` is Group B operator output** | AC #2 prohibits invented points; no CO geometry exists in the production DB at-merge. Mirrors S05.5's `geometry-overlays.json` Group B posture exactly. Option B (authoring named coordinates at-merge) was rejected. |
| **No CWD test point** | CO publishes no CWD zone geometry. S05.3 (2026-06-03) exhaustively searched three paths (CPW ArcGIS service catalog, ArcGIS Online org scan, PDF hand-trace) and found no polygonal CWD zone. `kind='cwd_zone'` does not appear in the CO geometry table. The CWD coverage invariant in `build_overlay_fixture.py` is vacuously satisfied. |
| **No `portion` kind** | CO uses GMUs as primary units; no portion sub-geometries are defined. The `portion` handling branch in `build_overlay_fixture.py` (and in S05.5's `OverlayChildKind`) is MT-specific infrastructure that has no CO analog. |
| **`expected_role_for_e03` is `null` for all CO points** | E05 writes zero `jurisdiction_binding` rows. The binding role (`no_hunt_zone` for restricted areas per ADR-021; `primary_unit` for GMUs) is assigned by E06's CO binding loader. There is no role to assert against in E05's spatial verification. |
| **CO restricted areas are all orphans** | All 10 V1 federal no-hunt zones (4 NPs + 5 NMs + 1 DOD AFA) are in `EXPECTED_CO_RA_ORPHAN_IDS` in `build_overlay_fixture.py`. They are spatially adjacent to GMUs, not contained by them (same pattern as MT's Glacier NP / Sun River / Yellowstone orphans). `ST_Covers` returns a restricted area's own row when queried from a point inside it; an adjacent GMU row may also appear and is not an error. |
| **Cross-state regression Step 7 is green at-merge** | `test_co_binding_loader_sql_filters_by_state_co_pollution_guard` is a mocked test (no live DB); it passes in CI at-merge. It is recorded in the Group B table for UAT-completeness, but its CI status is already confirmed — it does not need a live-DB run to evidence the AC. |

## Data observations / open items

No blocking items are expected from the Group B live run. Anticipated observations:

- **`multipart-gmus.json` AC #703 fallback:** If S05.2's Group B live run finds that CO has zero multi-part GMUs (i.e., `multipart-gmus.json` is empty), the named anchor step is N/A — document this as a data observation in this note and do NOT silently skip. The AC explicitly permits the N/A path.
- **CO restricted-area spot-checks may return two rows** (restricted area + overlapping GMU). This is correct behavior, not an error; document any such cases during Step 1 for E06 situational awareness (E06's no-hunt-zone binding loader will need to know which GMUs each NP/NM overlaps).
- **Threshold-recalibration check (per ADR-016 §4):** after generating `geometry-overlays.json`, inspect `geometry-overlays-dropped.json` for borderline drops (`overlap_pct` in `[0.005, 0.02]`) and borderline relabels (`overlap_pct` in `[0.98, 0.995]`). If either band's count differs from MT proportions by >10%, recalibrate `COVER_RELABEL_THRESHOLD` / `COVER_DROP_THRESHOLD` per ADR-016 §4. Record the decision in the Group B verification record above (under Step 1 notes or as an additional row).
