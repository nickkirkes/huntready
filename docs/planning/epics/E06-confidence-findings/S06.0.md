**Question.** E05 S05.3 empirically confirmed CO publishes no CWD-zone geometry (license/hunt-code-keyed model). How should CO's CWD reporting be modeled in V1?

**Decision (verbatim outcome).** Option **(c) license-keyed** — retain Q18's V1 license-keyed disposition: **0 typed CWD `reporting_obligation` rows**; CWD text lives in `regulation_record.additional_rules`. No new table.

**Rationale.** Matches the empirical CO model (S05.3 found CWD is managed by hunt code / GMU, not mapped zone polygons) and the S05.3 closure recommendation. PRD 002 §"Open decisions resolved during M2" (L174) frames Q18 as an ADR-candidate triggered "by first CPW CWD sampling row that doesn't fit `regulation_record.additional_rules`" — option (c) means that trigger does not fire; the V1 disposition is recorded without an ADR.

**Downstream spec implications.** **S06.9** (reporting_obligation ingestion): no CWD `reporting_obligation` rows; the `cwd_sample` enum value remains defined but unused for CO V1; CWD prose flows into `regulation_record.additional_rules` at the S06.6 regulation_record layer. No `public.*` table; no schema change.

### D2 — Known Issue #6 (`_VALID_ROLE_FOR_E03` subset gate) → **Hardcoded path** [PM rec confirmed]

**Question.** How do CO no-hunt zones get their `jurisdiction_binding.role`? MT's `_VALID_ROLE_FOR_E03` frozenset (`ingestion/states/montana/load_jurisdiction_bindings.py:417`) is deliberately 4 values `{primary_unit, portion, restricted_area, cwd_management_zone}` and excludes `no_hunt_zone`; MT routes no-hunt zones through a separate hardcoded `_build_no_hunt_zone_bindings` (`:592`) that bypasses the overlay-fixture gate.

**Decision (verbatim outcome).** **Hardcoded path** — CO mirrors MT's `_build_no_hunt_zone_bindings`. The gate frozenset stays narrow at 4 values; CO no-hunt zones flow through a dedicated hardcoded builder, not the overlay-fixture path.

**Rationale.** Lower blast radius (zero MT changes; the gate is documented as intentionally narrow and widening it would require updating the frozenset + MT tests), and consistent with the established MT pattern. The S05.6 CO scaffold already pre-ships the `query_nearby_gmus_for_zone` reference function (`ingestion/states/colorado/load_jurisdiction_bindings.py:85`) for exactly this hardcoded path.

**Downstream spec implications.** **S06.10** (CO binding loader): CO no-hunt zones use a hardcoded `_build_no_hunt_zone_bindings`-analog that writes `role='no_hunt_zone'` directly (per ADR-021), bypassing `_VALID_ROLE_FOR_E03`. The MT gate frozenset is **not** widened. S06.10 promotes the S05.6 scaffold's `query_nearby_gmus_for_zone` reference function to a live function body.

### D5 — Multi-source provenance for the 10 S05.4 no-hunt-zone geometry rows → **(b) accept split-provenance** [confirmed]

**Question.** Those rows' `geometry.source` is currently the PAD-US citation (`co-usgs-padus-arcgis-Federal_Fee_Managers_Authoritative_PADUS-0-2026`, `document_type='gis_layer'`). When E06 populates `verbatim_rule` from the CPW Big Game brochure, the regulatory authority for the prohibition is CPW (`annual_regulations`) but the geometry provenance is PAD-US. The single `source` field forces a choice: (a) UPDATE `source` to the CPW citation; (b) accept split-provenance with PAD-US winning the `source` field; (c) ADR + migration adding a multi-source provenance field.

**Decision (verbatim outcome).** Option **(b) accept split-provenance.** PAD-US wins the `geometry.source` field (`document_type='gis_layer'`); the CPW regulatory text reaches the rows via `verbatim_rule` only. No migration, no `source` mutation.

**Rationale.** Text faithfulness is preserved via `verbatim_rule` (ADR-008), while `source` stays honest about the geometry's origin (PAD-US). No schema change, no consumer breakage. PRD 002 §"Open decisions resolved during M2" (L176) frames "Multi-source geometry provenance" as an ADR-candidate triggered "if CPW splits a single GMU's geometry across multiple data sources" — D5 is a distinct trigger (PAD-US geometry + CPW text on the **same** row); accepting split-provenance for V1 leaves that Q a **deferred ADR-candidate** (no new field).

**Option-(a) consumer grep (recorded for completeness even though (b) was chosen, per AC #137).** Grepped `document_type` across `mcp-server/` + `web/` + `ingestion/`: the only hit on a geometry's `source.document_type` is the **type definition** at `mcp-server/src/types/schema.ts:50`. There are **zero runtime consumers** that branch on `geometry.source.document_type` (mcp-server is a 3-file stub; web has 10 TS files with zero `document_type` references). So option (a) would have caused no breakage today — but (b) was chosen, so the question is moot.

**Downstream spec implications.** **S06.5** (no-hunt-zone `verbatim_rule` population): UPDATE **only** the `verbatim_rule` column on the 10 geometry rows; leave `source` (PAD-US / `gis_layer`) untouched. Consistent with D6 (the targeted helper touches only `verbatim_rule`).

### D6 (conditional 6th) — `db.update_geometry_verbatim` helper → **Built in S06.5, not S06.0**

**Question.** S06.5 must populate `verbatim_rule` on the 10 no-hunt-zone geometry rows via a targeted UPDATE. Add a fail-loud `db.py` helper now, or do an in-script UPDATE? (Template: `update_legal_description` at `ingestion/ingestion/lib/db.py:362`.)

**Decision (verbatim outcome).** **Targeted helper, built in S06.5 — not S06.0.** Lock the decision now: S06.5 populates `verbatim_rule` via a **dedicated targeted UPDATE helper that fails loud on `cur.rowcount == 0`** (mirroring `update_legal_description` / `update_license_tag_draw_spec_key`), **NOT** by reusing `upsert_geometry`. The helper is built in S06.5 where a test exercises it; **no `db.py` helper lands in S06.0**.

**Rationale.** Unlike S03.6.1's `upsert_jurisdiction_binding` (which S03.10 had to reuse, so the id-format encoding contract had to be locked early), there is **no cross-story encoding contract** to lock here — so no early introduction. Building the helper in S06.5, alongside the test that exercises it, keeps the helper and its test co-located. The helper touches **only** the `verbatim_rule` column, leaving `source` (PAD-US / `gis_layer`) untouched — consistent with D5=(b).

**Downstream spec implications.** **S06.5**: add `db.update_geometry_verbatim(conn, geometry_id, verbatim_rule)` analog of `update_legal_description` (`db.py:362`) — `UPDATE geometry SET verbatim_rule = %s WHERE id = %s`, fail-loud on `cur.rowcount == 0`, no-commit (caller-controlled txn), + a `test_db.py` test class. ~30 LOC. No S06.0 code change.
