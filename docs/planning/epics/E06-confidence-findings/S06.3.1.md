## Phase B — R17 row-fusion split port (Known Issue #11)

Ported the pdfplumber row-fusion splitter from `extract_black_bear.py` (bear's "Rule R17") to
`extract_big_game.py`, where it is **big-game Rule R16** (per-module rule numbering; big-game had no R16).

### The adaptation — NOT a verbatim port

Bear's `_split_fused_block_row` is **strict**: it splits every present block cell on `\n` and **raises** if
any cell's part-count ≠ N (the hunt-code count). That is correct for bear, where a fused row doubled *all*
columns. **Big-game fusion is partial-column**: some cells carry a single *shared* value that applies to
both logical rows (e.g. one `valid_gmus='82'` shared by `D-M-082-O2-R` and `D-M-082-O3-R`). A verbatim port
would have **raised on all four** big-game fused rows.

The big-game split rule (`n` = number of full hunt codes in the Hunt Code cell, ≥ 2), per present block cell
split on `\n` into `k` parts:

- `k == n` → **distribute** (part *i* → synthetic row *i*)
- `k == 1` → **broadcast** (shared single value → all *n* rows; empty/`None` → `None`)
- otherwise → **fail loud** (`PdfExtractionError`, ADR-001 — genuine misalignment)

Wired into the single standard/dual/triple block path in `_parse_table_block`
(`for srow in _split_fused_block_row(sub_row, block): ...`). The Season Choice (`8col`) path is **deliberately
not wired** (distinct column layout; no fused Season-Choice rows exist — verified). `extract_big_game.py`
remains state-agnostic-clean (no `ingestion/lib/` edits); R16 stays per-extractor, mirroring `_split_valid_gmus`.

### Count reconciliation — spec's "9" → empirical **4**

cells" with "dropped codes"). The empirical artifact scan finds exactly **4 fused rows → 4 dropped codes**:

All four recovered codes are confirmed present as their own rows in the regenerated artifact. Per the
