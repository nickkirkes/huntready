## What shipped

A state-agnostic ArcGIS-fetch hardening + the CO restricted-area loader's immediate workaround, so the
republished PAD-US 4.1 service is fetched with a resolvable OBJECTID and any future ArcGIS-host
republish fails loud at the right layer with an actionable diagnostic.

**4 files changed (+176 / −23):**
- `ingestion/states/colorado/load_restricted_areas.py` — `_RA_OUT_FIELDS` now includes `"OBJECTID"`
  (append-not-prepend); the manifest-hash loop calls `arcgis._require_objectid` (was `_read_objectid` +
  manual `if oid is None: raise`).
- `ingestion/ingestion/lib/arcgis.py` — new strict `_require_objectid(feature, *, oid_field=None) -> int | str`
  that scans only `properties`/`attributes` and raises `ArcGISError` (diagnostic naming the failure mode,
  the republish/`outFields` root cause, and forensic feature keys) — **no `feature["id"]` fallback**.
  `fetch_features`' manifest-hash loop now calls it. `_read_objectid` is unchanged (keeps the fallback)
  for its 6 error-message callers and the dedup loop, which legitimately tolerate `None`.
- `ingestion/tests/test_arcgis.py` — new `TestReadObjectidFailsLoud` (4 tests, incl. the behavior-change
  lock `test_top_level_id_only_raises`).
- `ingestion/tests/test_load_co_restricted_areas.py` — new `TestStateAdapterOutFieldsIncludeObjectid`
  AST-walk over the whole CO adapter dir.

**Design decision:** added a *new* strict helper rather than mutating `_read_objectid`, because
`_read_objectid` has 6 error-message-context callers (`arcgis.py:335,490,499,510,527,559`) + the dedup
loop that must keep tolerating `None`. AC #624's "(or its raising caller)" sanctions this. The behavior
change is localized to the two OID-critical manifest-hash loops, which already treated `None` as fatal —
now they also refuse the masking `feature["id"]` fallback.

## Pitfall candidate (T7 — doc-writer lands final wording)

Under `.roughly/known-pitfalls.md` § "Integration — ArcGIS": an ArcGIS host can republish a layer such
that the top-level GeoJSON `id` is omitted unless `OBJECTID` is in the request `outFields` — a loader
whose hardcoded `_*_OUT_FIELDS` lacks `"OBJECTID"` then resolves every feature's OID to `None` and fails
loud with 0 rows. **Forensic signal of the same republish:** the response native CRS shifts (PAD-US
`4269 → 3857`), captured as a warning by `lib/arcgis._check_and_fix_projection`. Mitigation: include
`"OBJECTID"` in every hardcoded outFields tuple (locked by `TestStateAdapterOutFieldsIncludeObjectid`),
and use `_require_objectid` (not `_read_objectid`) on any OID-critical path so the gap surfaces with the
`outFields`-fix diagnostic instead of the silent `feature["id"]` fallback.
