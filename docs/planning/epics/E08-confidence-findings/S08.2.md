## §1 — ADR-024 addendum content (for the human to fold into ADR-024 — PM does NOT edit the ADR)

### Chosen path

**`postgres.js`** (the `postgres` npm package) connecting **directly** to Supabase over the Postgres wire protocol, with the SELECT-only-role DSN held as a **Workers Secret** (`SUPABASE_READONLY_DSN`), a **per-request** client (`createDbClient` called inside the `fetch` handler, `max: 1`), targeting the Supabase **Supavisor transaction-mode pooler** (`prepare: false`).

### Why `postgres.js`, not `@neondatabase/serverless`

`@neondatabase/serverless` has two modes: its `fetch`-based HTTP mode speaks only to Neon's proprietary SQL-over-HTTP endpoint (not Supabase); its WebSocket `Pool` mode needs a `wsproxy` deployed in front of Postgres. Neither targets Supabase cleanly without additional infrastructure. `postgres.js` speaks the standard Postgres wire protocol and runs in **both** Node (TCP — the vitest Node pool, enabling real database tests without mocking the Worker runtime) and workerd (`cloudflare:sockets`, enabled by the already-present `nodejs_compat` compatibility flag). No proxy or additional service required.

### Why direct + Workers Secret, not Hyperdrive, for V1

**Lighter foundation for V1.** Hyperdrive is a Cloudflare cloud resource that must be provisioned via `wrangler hyperdrive create --connection-string=…` — a more involved Group B operator step than a single `wrangler secret put`. The `src/db.ts` access layer takes a plain `dsn: string` parameter, so adopting Hyperdrive later **changes only how `index.ts` sources the DSN** — the `src/db.ts` code is unchanged. Hyperdrive is recorded here as the **drop-in optimization** for when pooled high-concurrency is needed; it is not needed for the V1 read workload.

### ADR-024 `Proposed → Accepted` flip is Group-B-contingent

**PM flags this to the human; PM does not edit the ADR.** The driver is chosen and the access-layer code + role-level write-rejection ship at Group A merge. However, the read-only-*enforcement* claim from the *edge runtime* is not proven until the live edge read + rejected write are captured in Group B (§6 below). **Do NOT flip ADR-024 to `Accepted` at Group A merge** — flag it as Group-B-contingent and flip after the Group B runbook is completed.

## §2 — Workers connection-lifecycle posture

`workerd` cannot reuse outbound sockets across request invocations. Every Worker invocation is isolated: sockets created in one invocation are not visible to any other. This means:

- **`createDbClient` is called inside `fetch`** (wired at S08.3). Calling it at module scope would construct a client that workerd's runtime can never meaningfully share.
- **`max: 1`** in the `postgres()` options reflects this — a pool larger than 1 would queue connections that can never be reused across invocations, wasting resources without benefit.
- **`.end()` after use** — the client must be closed after each request to release the connection. The `close()` method on `DbClient` calls `sql.end({ timeout: 5 })`, giving in-flight queries five seconds to drain.
- **`prepare: false`** — REQUIRED for Supabase's Supavisor transaction-mode pooler. Named prepared statements are connection-scoped; the pooler remaps connections per-transaction, so a prepared statement created on one connection may be routed to a different connection on the next round-trip ("prepared statement does not exist"). Simple-query protocol avoids this.

**Implication for E09/E10:** budget **one connection acquisition per tool call**. Never assume a cross-request connection pool. Each tool handler calls `createDbClient(dsn)` at the top of its request path, runs its queries, and calls `client.close()` in a `finally` block before returning. If Hyperdrive is later adopted, it moves the pooling server-side at the Cloudflare edge — the per-invocation acquisition model in application code is unchanged.

## §3 — Q14 serving-half finding

The SELECT-only-role **DSN** is a Postgres connection string — entirely distinct from the Supabase API-key question (publishable/secret keys). These are two independent credential surfaces:

| Surface | Format | Who holds it |
|---------|--------|--------------|
| Supabase API key (`SUPABASE_SECRET_KEY`) | `sb_secret_*` / legacy JWT | `SUPABASE_SECRET_KEY` Workers Secret (ingestion + any PostgREST path) |
| SELECT-only-role DSN | `postgresql://...` | `SUPABASE_READONLY_DSN` Workers Secret (serving stack) |

**Chosen connection-string target:** the Supabase **Supavisor transaction-mode pooler** format:

```
postgresql://huntready_readonly:<password>@<project-ref>.pooler.supabase.com:6543/postgres
```

The query string should include `?sslmode=require`. The legacy direct format (`postgresql://...:<password>@db.<project-ref>.supabase.co:5432/postgres`) also works but the pooler is preferred for workerd's connect-per-invocation pattern (the pooler keeps server-side connections warm, reducing per-request latency).

**Q14 status: this ADVANCES the serving half of Q14.** The serving stack now has a documented, chosen connection-string format. It does **not** fully close Q14 because:

1. The E01 RLS-verification-runbook half (runbooks using legacy `anon` / `service_role` JWT keys in curl commands) stays open — not M3's to close.
2. The final DSN format (project-ref, role password) is confirmed at Group B when the operator composes the real SELECT-only-role DSN against the dev Supabase project.

Record in `docs/open-questions.md` Q14's M3 note (already present) — no further PM action until the E01 runbook question surfaces.

## §4 — Hyperdrive caching forward-note

**N/A under the chosen direct path** — there is no Hyperdrive resource and no Hyperdrive cache to configure. This section exists for completeness and to guard against a future adopter enabling caching without reading this note.

If Hyperdrive is adopted later (the drop-in optimization path described in §1): caching must be **off or set to a short TTL** (for example, `max_age: 0`). Regulatory freshness (`meta.data_freshness`) is a **correctness property** per ADR-024 and ADR-001 — a stale cached result that answers "is this season still open?" with last week's data is a regulatory safety failure, not just a staleness inconvenience. When Hyperdrive caching is configured, a **Hyperdrive cache-purge step belongs in the E11 re-ingestion/deploy runbook** (between "re-ingest completes" and "deploy new Worker"). Default-on caching is not assumed.

## §5 — CI-vs-local AC disposition

Both the PostGIS `ST_*` WKT-round-trip smoke (Test B) and the role-level write-rejection (Test C) run **in CI** — not just locally.

The `.github/workflows/ci.yml` CI job provisions a `postgis/postgis:16-3.4` service container alongside the test run. The provisioning step applies:

1. `mcp-server/tests/fixtures/ci-substrate.sql` — creates the `geometry` table with a minimal representative row.
2. `supabase/grant-readonly-role.sql` — creates the `huntready_readonly` role + `GRANT SELECT` + RLS ALLOW-SELECT policies.
3. `ALTER ROLE huntready_readonly PASSWORD 'ci_readonly_pw';` — sets a throwaway password for the ephemeral localhost-only CI Postgres (carries `# pragma: allowlist secret`).

The `npm run test:ci` step then sets `TEST_READONLY_DSN=postgresql://huntready_readonly:ci_readonly_pw@localhost:5432/huntready_test` (also `# pragma: allowlist secret`). The four live DB tests in `tests/db.test.ts` (skipped when `TEST_READONLY_DSN` is absent) run against this substrate in every CI push/PR.

**Equivalent local run path** for a developer who wants to run the live DB tests without CI:

```sh
# 1. Start a local PostGIS container (ephemeral; remove after testing)
docker run --rm \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=huntready_test \
  -p 5432:5432 \
  postgis/postgis:16-3.4

# Admin connection via libpq env vars (no credential-in-URL needed).
export PGHOST=localhost PGPORT=5432 PGUSER=postgres PGDATABASE=huntready_test
export PGPASSWORD=postgres  # pragma: allowlist secret  (throwaway, localhost-only)

# 2. Apply the CI substrate (creates the geometry table + seed row)
psql -v ON_ERROR_STOP=1 -f mcp-server/tests/fixtures/ci-substrate.sql

# 3. Apply the GRANT SQL (creates role + SELECT grant + RLS policies)
psql -v ON_ERROR_STOP=1 -f supabase/grant-readonly-role.sql

# 4. Set a throwaway local password on the role
psql -v ON_ERROR_STOP=1 -c "ALTER ROLE huntready_readonly PASSWORD 'localpw';"  # pragma: allowlist secret

# 5. Run the tests with the role DSN set (throwaway, localhost-only)
cd mcp-server
TEST_READONLY_DSN=postgresql://huntready_readonly:localpw@localhost:5432/huntready_test npm test  # pragma: allowlist secret
```

Without `TEST_READONLY_DSN` set, the four live DB tests skip cleanly (the `describe.skipIf(!DSN)` block in `tests/db.test.ts`). The remaining 18 tests run unconditionally (the top-level CI-DSN guard always runs).

**Note on the `extensions` schema:** the CI substrate's `geometry` table stores `geom` as `geography(MultiPolygon, 4326)`. The `postgis/postgis:16-3.4` image installs PostGIS in the `public` schema by default, not `extensions`. The `ST_VALID_SMOKE_SQL` constant therefore uses `extensions.ST_IsValid(...)` — which resolves correctly against the dev/prod Supabase projects (where PostGIS is installed with `SCHEMA extensions`) but would fail against a plain Docker PostGIS unless the schema is set up to match. The CI substrate SQL runs `CREATE SCHEMA IF NOT EXISTS extensions; CREATE EXTENSION IF NOT EXISTS postgis SCHEMA extensions;` to mirror the Supabase layout, making the `extensions.`-prefixed calls portable between CI and Supabase.

## §6 — Group B operator runbook (operator-pending; non-blocking)

Apply against the **dev Supabase project** (`eklivzoomtdluedzlyai`) after the S08.2 PR merges. Captures live in this note; PM ticks Group B ACs in a follow-up doc-only commit.

### Step 1 — Apply the GRANT SQL to dev

```sh
supabase db query --db-url "$DATABASE_URL" < supabase/grant-readonly-role.sql
```

Or use `psql` with the dev project's direct connection string:

```sh
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f supabase/grant-readonly-role.sql
```

Verify idempotency: running it a second time should succeed silently (the `DO $$ IF NOT EXISTS $$` guards handle re-runs).

### Step 2 — Set the role password (out of band)

```sql
ALTER ROLE huntready_readonly PASSWORD '<generated>';
```

Generate a strong random password (e.g., `openssl rand -base64 32`). Store it only in the Workers Secret — never in source.

### Step 3 — Compose the SELECT-only DSN and set the Workers Secret

The Supavisor transaction-mode pooler DSN format for the dev project:

```
postgresql://huntready_readonly:<password>@<project-ref>.pooler.supabase.com:6543/postgres?sslmode=require
```

Set it as the Workers Secret (run from `mcp-server/`):

```sh
wrangler secret put SUPABASE_READONLY_DSN
```

Paste the DSN at the prompt. It is never written to disk.

### Step 4 — Verify the role can READ (FORCE-RLS gotcha)

The dev Supabase project has `FORCE ROW LEVEL SECURITY` on every app table. The deny-all policies target the `anon` and `authenticated` roles; `huntready_readonly` is neither, so it needs an explicit ALLOW-SELECT policy — which `grant-readonly-role.sql` creates (the `huntready_readonly_select` policy via the `DO $$ LOOP $$` block). Verify the policy landed:

```sql
-- Run AS the service-role (admin) DSN:
SELECT tablename, policyname
  FROM pg_policies
 WHERE policyname = 'huntready_readonly_select'
 ORDER BY tablename;
```

Expected: one row per app table.

Then verify the role can actually read a row:

```sql
-- Run AS the huntready_readonly role:
SELECT id FROM geometry WHERE state = 'US-MT' LIMIT 1;
```

If this returns zero rows or errors with a row-level-security message, the ALLOW policy did not land — check `pg_policies` as above.

**`search_path` check for `extensions.ST_*`:** the Supavisor pooler may reset `search_path` per transaction. Verify the `extensions` schema is reachable for PostGIS function calls:

```sql
-- Run AS huntready_readonly:
SHOW search_path;
-- Expected output should include "extensions" OR the function call below should succeed
SELECT extensions.ST_IsValid(
  extensions.ST_GeomFromText(extensions.ST_AsText(geom), 4326)
)
FROM geometry
WHERE state = 'US-MT'
LIMIT 1;
```

If `extensions.ST_*` fails with "schema does not exist", the role needs `GRANT USAGE ON SCHEMA extensions TO huntready_readonly` — which `grant-readonly-role.sql` includes (step 3 in the SQL file). Re-run the GRANT SQL if it was applied before the `extensions` schema existed.

### Step 5 — Confirm a read from the deployed Workers edge runtime

After deploying (or redeploying) the Worker with the `SUPABASE_READONLY_DSN` secret set, exercise the `READ_SMOKE_SQL` path (wired at S08.3) and verify:

- A read against the **Montana corpus** (`state = 'US-MT'`, frozen at `m1`) returns at least one `geometry` row.
- The `ST_VALID_SMOKE_SQL` PostGIS smoke returns `{ id: string, valid: true }` for that row.

### Step 6 — Confirm a write is provably rejected from the edge runtime

Issue an `INSERT` through the serving stack's `DbClient.query()` method (e.g., via a temporary diagnostic endpoint). The role-level rejection must surface as a Postgres error with **SQLSTATE `42501`** (`permission denied for table geometry`). A mock that throws application code is **not** sufficient — the rejection must come from Postgres itself.

### Group B ACs to tick

- [ ] `huntready_readonly` role exists on dev Supabase with SELECT privilege and ALLOW-SELECT RLS policies (verify via `pg_roles` + `pg_policies` queries above).
- [ ] A SELECT against the Montana corpus from the deployed Workers edge runtime returns at least one row.
- [ ] The PostGIS `extensions.ST_IsValid(extensions.ST_GeomFromText(extensions.ST_AsText(geom), 4326))` smoke returns `valid = true` for a Montana geometry row.
- [ ] A write attempt from the serving stack is rejected with SQLSTATE `42501` from the Postgres role (not application code).

## §9 — Forward-notes for S08.3 (the access layer's first live call site)

S08.2 ships the access layer but does **not** wire `createDbClient` into the `fetch` handler — the live call site is S08.3's internal health check. Carry these into S08.3:

- **`close()` lifecycle in the Worker.** `createDbClient` is constructed per request (`max: 1`); `close()` (`sql.end({ timeout: 5 })`) must be called after each request. In `workerd`, do this via `ctx.waitUntil(client.close().catch((e) => console.error("db close error", e)))` **after** the `Response` is returned — do NOT `await client.close()` inline before returning, or a teardown error becomes a spurious 500 on an otherwise-successful read. The `boundary.test.ts` module-scope guard already locks that `createDbClient` is called inside `fetch`, never at module scope.
- **The smoke SQL constants are the proven read idiom.** S08.3's health check (and E09/E10's tools) read through `READ_SMOKE_SQL` / `ST_VALID_SMOKE_SQL` (or the same `extensions.`-prefixed + WKT-round-trip pattern) — never re-derive the cast workaround.
- **`fetch_types` perf (future-hardening, not S08.3-blocking).** `postgres.js` issues one `pg_type` catalog query per new connection (`fetch_types: true` default). With per-request `max: 1` that is one extra round-trip per request. Acceptable for the V1 low-concurrency workload; revisit (`fetch_types: false`, or Hyperdrive pooling) if/when concurrency grows.
