## Objective

E09 implements the three regulation-stack MCP tools — `get_regulations`, `list_seasons`, `get_tag_requirements` — against the Montana corpus, built entirely on the E08 foundation. They share the most query/join logic (all read the regulation/season/license/draw-spec/reporting stack through `jurisdiction_binding` fan-out and the link tables), so **`get_regulations` is the flagship that designs the shared layer once** (jurisdiction resolution, the bounded composite query, coverage-signal logic, and the generalized response mechanisms); `list_seasons` and `get_tag_requirements` are narrower reads over the same tables that reuse it.

When E09 closes: each tool returns a spec-valid response validating against its **own declared `outputSchema`**, with `structuredContent` as the source of truth, read-only annotations set, `sources[]` on every response, and schema-version-gated rows surfaced in `meta.warnings` (never silent-dropped). `verbatim_rule` and `confidence` pass through byte-identically. Out-of-scope queries return a structured `coverage: "none"` (not empty, not error). Tools are schema-generic (no MT-specific branches) so Colorado works at `m2` without server changes.

See [PRD 003 §"E09 — Regulation-stack tools"](../prds/003-M3-canonical-interface.md) (outcome + exit criteria), `architecture.md` §"The MCP server" (tool signatures), §"Response shape: `GetRegulationsResponse`" (the Shape C envelope, amended 2026-06-30 — `data_freshness` nullable), and §"Data model" (the 6 entities + link tables). Load-bearing ADRs: [ADR-001](../../adrs/ADR-001-authority-preserved.md), [ADR-006](../../adrs/ADR-006-schema-versioned-from-day-one.md), [ADR-008](../../adrs/ADR-008-verbatim-regulation-text.md), [ADR-010](../../adrs/ADR-010-decomposed-entity-model.md), [ADR-011](../../adrs/ADR-011-shape-c-response-envelope.md), [ADR-012](../../adrs/ADR-012-draw-mechanics-sibling-entity.md), [ADR-013](../../adrs/ADR-013-server-returns-structure-client-composes-presentation.md), [ADR-017](../../adrs/ADR-017-confidence-calibration.md).

**The four E08-resolved decisions are baked into the ACs below** (decided 2026-06-30): (1) `data_freshness` nullable + `sources` empty on `coverage:"none"`; (2) relax the empty-`tools/list` bootstrap + an exact-tool-set lock; (3) builder-owned warning accumulation + per-tool gating test; (4) audit Rec 5 — a single-read-path guard.

Surfacing it is an `architecture.md` §"Response shape" addition. **RESOLVED 2026-06-30 (human-approved): surface** — added a new `additional_rules: AdditionalRulesSection | null` section (`architecture.md` + ADR-011 amended; `Warning.section` gains `"additional_rules"`); S09.1 implements it. The read path is GiST + composite-PK equijoins — no ingestion-time `statement_timeout` risk.
- **Coverage boundary added:** jurisdiction-resolved-but-species-absent → `jurisdiction:"full"`, `species:"none"`, `overall:"partial"`, **non-empty `sources` + non-null `data_freshness`** (not collapsed to `none`).

- **`mcp-server/src/server.ts`** — `createMcpServer()` + the `initializeEmptyToolRegistry` register-then-remove bootstrap (**S09.1 deletes it when the first real tool registers** — decision 2); tools register via `server.registerTool(name, { description, inputSchema, outputSchema, annotations }, handler)`.
- **`mcp-server/src/types/response.ts`** (+ barrel `types/index.ts`) — `GetRegulationsResponse` + section/`Resolved*` types; E09 adds the `list_seasons` / `get_tag_requirements` output types here (serving-composition, **not** the 6-entity `schema.ts` sync). `types/schema.ts` holds the 6 entity types + `DrawSpec` (reused, not redefined).
- **DB join (from `supabase/migrations/` + `schema.py`):** `regulation_record` composite PK `(state, jurisdiction_code, species_group, license_year)`; the link tables `regulation_season` / `regulation_license` / `regulation_reporting` join back on **all four** PK columns (not a single id) + their entity id; `license_season` (`license_tag_id`↔`season_definition_id`, per ADR-018); `jurisdiction_binding` flat FK cols (`regulation_record_state`/`_jurisdiction_code`/`_species_group`/`_license_year`) + `geometry_id` + `role`; `geometry.geom geography(MultiPolygon,4326)` (GiST-indexed) + `geometry.state`; `draw_spec` composite PK `(state, hunt_code, year)` (+ `point_system`/`pools`); `license_tag.draw_spec_key` is a **jsonb soft FK** `{state, hunt_code, year}` (LEFT-JOIN `draw_spec` on its composite PK; non-draw tags → `draw_spec: null`). **Enum-drift caution** (the project's documented line-drift mode): if any AC/test enumerates `geometry.kind` or `jurisdiction_binding.role`, cite the **migration-current** sets (`kind` gained `'state'` in `20260504032424`; `role` gained `'no_hunt_zone'` in `20260603000000` — 8 values), not the initial migration's. Resolution reads `geometry.state`, not `kind`, so it's not load-bearing here.
- **`species` → `species_group` (no serving-time mapping; identical for MT + CO):** the DB `species_group` column stores **granular** values only — `mule_deer`, `whitetail`, `elk`, `pronghorn`, `bear` — for **both** states; the DEA `deer`→(`mule_deer`,`whitetail`) and `antelope`→`pronghorn` fan-out/rename happens **at ingestion** (`load_regulation_records.py` `_DEA_SPECIES_FANOUT`), so there is **no `deer`/`antelope` row in the DB** and no MT-specific serving branch. (Note: `license_tag.species` is a **separate denormalized text column** — the species filter keys on `regulation_record.species_group` reached via the link tables, NOT on `license_tag.species`.) `architecture.md:161`'s `species_group` comment (`"deer" | … | "antelope"`) is stale vs the loaded corpus — do not inherit it (forward-note flagged).
- **Jurisdiction resolution (`get_regulations`):** point → `geometry` via `extensions.ST_Covers(geom, extensions.ST_GeogFromText('SRID=4326;POINT(<lng> <lat>)'))` → `jurisdiction_binding` (on `geometry_id`) → `regulation_record` (via the flat FK cols) → seasons/tags/reporting via the link tables. **`ST_Covers(polygon_geom, point)` = polygon-covers-point** (geom is arg 1); WKT is **`POINT(longitude latitude)` — lng first** (lat/lng transposition is the classic silent bug → wrong/empty jurisdiction). `ST_Covers` runs **natively on geography** — the point is built with `ST_GeogFromText`, so the resolution predicate needs **no WKT round-trip / no `::geometry` cast**; the round-trip is required only for ancillary *geometry-only* selects (e.g. `ST_AsText`/`ST_NumGeometries` of a boundary). **`state` is an *output* of resolution** (read `geometry.state` from the covering row), not a precondition — the GiST `ST_Covers` lookup need not be pre-filtered by state; **all *downstream* link-table queries carry the resolved `state`** (no cross-state fan-out).
- **Test gating:** live-DB tests use `describe.skipIf(!process.env.TEST_READONLY_DSN)` + a top-level CI guard asserting the DSN is set in CI; they run in CI against the dev corpus. **Baseline floor: 133 passed + 6 skipped (local) / 139 (CI)** — E09 grows it additively.

### S09.1: `get_regulations` — flagship Shape C composite + generalized response foundation

**Status:** Not started

**As an** agentic client asking "I'm hunting {species} at {lat,lng} on {date} — what applies?"

**So that** the canonical interface's headline tool proves the whole regulation-stack pattern, and the shared layer (jurisdiction resolution, bounded join, coverage signals, generalized response mechanisms) exists for `list_seasons`/`get_tag_requirements` to reuse

**UAT: yes** (Group A static/unit/CI-live closes at-merge; PM-run faithfulness spot-check is the UAT sign-off, non-blocking)

**S09.2 and S09.3 depend only on S09.1a** (the foundation) — they can start once S09.1a merges. **Each AC below is tagged `[1a]`/`[1b]`.**

**Context:**

The flagship. Resolves `(lat, lng)` → `geometry` (PostGIS `extensions.ST_Covers` + `ST_GeogFromText('SRID=4326;POINT(<lng> <lat>)')`; **`state` is read from the covering `geometry.state` — an output of resolution, not a pre-filter; the GiST `ST_Covers` lookup need not be state-pre-filtered, and all *downstream* link-table queries carry the resolved `state`**) → `jurisdiction_binding` (on `geometry_id`) → `regulation_record(s)` (flat FK cols) → seasons/tags/methods/reporting/contacts via the link tables, composed into the Shape C envelope. Registers the **first** real MCP tool — so this story **deletes the `initializeEmptyToolRegistry` bootstrap** (decision 2) and **replaces BOTH the empty-registry lock AND the now-stale additive-extension test in `mcp-server/tests/server.test.ts`** with an exact-registered-tool-set assertion (`tools/list` is exactly the V1 tools registered so far, no internal/health tools — the UAT #1 guard). It also lands the cross-cutting foundation the other two reuse:

- **Decision 1:** make `meta.data_freshness` nullable in `types/response.ts` + `output-schema.ts`; drop `sources.min(1)`; `buildDataFreshness` returns `null` on empty (the throw becomes a "non-empty-when-called" guard). A `coverage:"none"` response carries `sources: []` + `data_freshness: null` — invariant **`data_freshness` null iff `sources` empty**.
- **Decision 3 (specified):** **`gateBySchemaVersion` STAYS PURE** — it keeps its `{included, warnings}` return shape (so `health-check.ts` + its existing tests are unaffected; do not refactor the signature). The "can't-forget" property comes from making **`buildStructuredToolResult` the sole writer of `meta.warnings`**: it takes the accumulated warnings (the handler collects each section's `gateBySchemaVersion(...).warnings` into one array) as a **required argument** and is the only place that populates `meta.warnings`. A handler that gates but doesn't thread its warnings through is caught by the per-tool gating test. Generalize `buildStructuredToolResult` to be **schema-parameterized** (`buildStructuredToolResult(payload, schema, warnings, renderText)`) so S09.2/S09.3 reuse it with their own `outputSchema`. **The current `buildStructuredToolResult` hard-calls `renderThinText`, which is typed to `GetRegulationsResponse` and reads `meta`/section fields the narrower envelopes lack — so thin-text rendering must also be parameterized** (a per-tool `renderText(payload): string`). Every tool still emits `content[0].text` as a **mechanical, field-derived** derivative (presence/coverage/warning inventory) — never a summarizing/paraphrasing "overview" (ADR-013/ADR-008). (`architecture.md`'s "overview text assembled from the structured sections" prose is loose — the impl is mechanical, not an overview.)
- **Rec 5:** a single-read-path guard (lint/test that only `db.ts` opens a DB connection).

**`additional_rules` surfacing — ✅ RESOLVED 2026-06-30 (surface):** `regulation_record.additional_rules: VerbatimRule[]` (HD-wide `NOTE:` lines, the statewide Bear ID Test rule — verbatim text with per-rule `confidence`, one of only **two** confidence carriers per ADR-017 §3) is now surfaced via a **new always-present null-bearing section `additional_rules: AdditionalRulesSection | null`** (`{ rules: VerbatimRule[]; source }`) — `architecture.md` §"Response shape" + ADR-011 amended (and `Warning.section` gained `"additional_rules"`). S09.1 implements it in `types/response.ts` + the zod `outputSchema`: `rules` reuse the existing `VerbatimRule` type verbatim; the section is **populated-but-empty (`rules: []`) when a record resolved with no such prose, `null` only when no record resolved** (`coverage.overall:"none"`). Text surfaced byte-identically (ADR-008); each rule's `confidence` is its own (per ADR-017 §3, independent of the parent-inherited section confidences).

**Coverage tri-state boundary cases (lock all three):** total gap → all sections null, `sources: []`, `data_freshness: null`, `coverage.overall:"none"`. **Mixed: point resolves to a known HD but the species has no `regulation_record` there → `jurisdiction:"full"`, `species:"none"`, `overall:"partial"`, and `sources` is NON-empty + `data_freshness` NON-null** (the jurisdiction resolved and is cited) with the season/tag sections null — do **not** collapse species-absent-under-a-resolved-jurisdiction into `coverage:"none"`/`sources:[]`. Populated-but-empty section (e.g. `reporting.obligations: []`) ↔ `coverage:"full"` ("not required" vs "no data").

**Coverage tri-state (the heart of Shape C — get it exactly right):** `meta.coverage.{jurisdiction, species, overall}`. Jurisdiction `none` = no geometry covers the point (or no binding for the resolved state); species `none` = species not in the dataset for the resolved jurisdiction; `overall` aggregates. A null section pairs with `coverage:"none"`; a populated-but-empty section (e.g., `reporting.obligations: []`) pairs with `coverage:"full"` ("not required" vs "no data"). A total gap → all sections null, `sources: []`, `data_freshness: null`, `coverage.overall:"none"` — structured, not empty, not error.

**Bounded query:** a small constant (target: one) of `db.query` calls — **no per-`jurisdiction_binding` fan-out**. The measurable gate: an instrumented test asserts the handler issues **≤N `db.query` calls independent of overlay/binding count** (overlays aggregated via `json_agg` in the bounded query), validated on the largest MT district (max-16 overlays). Connect-per-request; **`await client.close()` in a `finally`** (swallow teardown errors so a close failure isn't a 500) — the reachable pattern E08's `health-check.ts` uses; `ctx.waitUntil` is **not** available to a tool callback (no `ExecutionContext` on `RequestHandlerExtra`).

**🔒 Design decision (locked 2026-07-01 — do NOT relitigate): `date` drives `SeasonsSection.status`, it does NOT filter `windows[]`.** `get_regulations` returns the **full** season structure for the resolved jurisdiction/species (all `windows`) and sets the date-derived `status` (`in_season` / `out_of_season` / `no_season_defined` / `conditionally_closed` / `unknown`). The `windows` list is intentionally NOT filtered to only ranges active on `date`. Rationale: (1) `status:"out_of_season"` is a schema value **distinct** from `no_season_defined` — date-filtering would collapse the two (out-of-season would always carry an empty `windows`), destroying the distinction and hiding a hunter's upcoming/past seasons; (2) filtering to "today's windows" is a client presentation choice, not a server one (ADR-013 — server returns structure, client composes); (3) `list_seasons(state, species, year)` (S09.2) returns the same `ResolvedSeasonWindow` set with **no date parameter at all** — windows are the season structure, not a date slice. A review raised "date not used to filter windows" as a gap (2026-07-01); it was assessed and **rejected as by-design** — the date's job is `status`. Changing this would be a Shape C `SeasonsSection` contract change requiring an `architecture.md` amendment + `list_seasons` reconciliation, not a bug fix.

**Relevant ADRs:** [ADR-011](../../adrs/ADR-011-shape-c-response-envelope.md), [ADR-013](../../adrs/ADR-013-server-returns-structure-client-composes-presentation.md), [ADR-001](../../adrs/ADR-001-authority-preserved.md)/[ADR-008](../../adrs/ADR-008-verbatim-regulation-text.md), [ADR-006](../../adrs/ADR-006-schema-versioned-from-day-one.md), [ADR-012](../../adrs/ADR-012-draw-mechanics-sibling-entity.md) (embedded `draw_spec`).
