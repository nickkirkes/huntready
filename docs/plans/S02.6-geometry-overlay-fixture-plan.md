# Implementation Plan: S02.6 — Geometry Overlay Fixture

**Story:** S02.6 from `docs/planning/epics/E02-geometry-ingestion.md` lines 458–570.
**Branch:** `feat/S02.6-geometry-overlay-fixture`

## Context

E02 has loaded 349 Montana geometries (235 HDs + 55 portions + 57 restricted areas + 2 CWD zones). E03 will create `jurisdiction_binding` rows once `regulation_record` rows exist, but it cannot do that today (regulation_record is E03 territory). S02.6 closes the handoff gap by computing the spatial topology *now* and persisting it as a JSON fixture that E03 will consume.

The fixture captures four parent→child relationship classes:

1. **HD → HD self-references** (programmatic, not spatial) → `role_for_e03='primary_unit'`
2. **HD → Portion** containment (`ST_Covers(geog, geog)`) → `role_for_e03='portion'`
3. **HD → CWD zone** containment OR intersection (`ST_Intersects` then `ST_Covers` discriminator) → `role_for_e03='cwd_management_zone'`
4. **HD → Restricted Area** (same pattern as CWD) → `role_for_e03='restricted_area'`

### T3: Spatial-query SQL + per-relationship row generators (~5 min)
**Files:** `ingestion/states/montana/build_overlay_fixture.py`
**Depends on:** T2
**Action:** Implement the four overlay-row producers (HD self + 3 spatial queries) as pure functions returning `list[OverlayFixtureRow]`.
**Details:**
- Define module-level SQL constants. Each query MUST use geography-native operators (no `::geometry` cast — see `.roughly/known-pitfalls.md`):
  ```python
  HD_TO_PORTION_SQL = """
  SELECT a.id AS parent_id, b.id AS child_id
  FROM geometry a, geometry b
  WHERE a.kind = 'hunting_district' AND b.kind = 'portion'
    AND a.state = %s AND b.state = %s
    AND ST_Covers(a.geom, b.geom)
  """

  HD_TO_OVERLAY_SQL = """
  SELECT a.id AS parent_id, b.id AS child_id,
         ST_Covers(a.geom, b.geom) AS is_covered
  FROM geometry a, geometry b
  WHERE a.kind = 'hunting_district' AND b.kind = %s
    AND a.state = %s AND b.state = %s
    AND ST_Intersects(a.geom, b.geom)
  """
  ```
  (`HD_TO_OVERLAY_SQL` is parameterized on child kind so CWD zone and Restricted Area share the SQL.)
- Implement `_fetch_hd_ids(conn) -> list[str]`: `SELECT id FROM geometry WHERE kind = 'hunting_district' AND state = %s ORDER BY id`. Returns sorted IDs.
- Implement `_build_hd_self_rows(hd_ids: list[str]) -> list[OverlayFixtureRow]`: for each HD id, append a row with `parent_geometry_id == child_geometry_id == hd_id`, `parent_kind="hunting_district"`, `child_kind="hunting_district"`, `relationship="self"`, `role_for_e03="primary_unit"`. No SQL.
- Implement `_build_hd_portion_rows(conn) -> list[OverlayFixtureRow]`: execute `HD_TO_PORTION_SQL` with `(MT_STATE_CODE, MT_STATE_CODE)`, build rows with `relationship="covers"`, `child_kind="portion"`, `role_for_e03=ROLE_FOR_E03_BY_CHILD_KIND["portion"]`.
- Implement `_build_hd_overlay_rows(conn, child_kind: OverlayChildKind) -> list[OverlayFixtureRow]`: execute `HD_TO_OVERLAY_SQL` with `(child_kind, MT_STATE_CODE, MT_STATE_CODE)`. For each row, set `relationship = "covers" if is_covered else "intersects"` and `role_for_e03 = ROLE_FOR_E03_BY_CHILD_KIND[child_kind]`.
- Replace the T2 stub `_collect_overlay_rows(conn, explain=...)` with: fetch HD ids → build self rows → build portion rows → build CWD overlay rows (`child_kind="cwd_zone"`) → build Restricted-Area overlay rows (`child_kind="restricted_area"`) → concatenate and return. Log INFO counts per relationship type.
- Use `with conn.cursor() as cur:` for each query (matches `db.py` convention).
- Do NOT call `conn.commit()` — this is a read-only script.
**Verify:** `cd ingestion && .venv/bin/mypy ingestion/states/montana/build_overlay_fixture.py`

### T4: Coverage-invariant validation + custom exception (~4 min)
**Files:** `ingestion/states/montana/build_overlay_fixture.py`
**Depends on:** T3
**Action:** Validate every Portion / CWD zone / Restricted Area appears as child of ≥1 HD; collect orphans, then raise once with the full list.
**Details:**
- Define `class OverlayFixtureError(ValueError):` near the top of the script (after constants, before SQL). Module-level docstring: "Raised when the overlay fixture violates the S02.6 coverage invariant."
- Implement `_validate_coverage(conn, rows: list[OverlayFixtureRow]) -> None`:
  - Query `SELECT id, kind FROM geometry WHERE state = %s AND kind IN ('portion', 'cwd_zone', 'restricted_area') ORDER BY kind, id` to enumerate all geometry rows that the invariant requires to appear as children.
  - Build a set of `(child_kind, child_geometry_id)` from `rows` where `parent_kind == "hunting_district"` and `relationship in ("covers", "intersects")`.
  - For each `(id, kind)` from the DB query, check membership; collect missing IDs into a list grouped by kind.
  - If any orphans: `raise OverlayFixtureError(f"orphan geometries (no HD parent): {orphans!r}")`. Include each kind's orphan ids in the message so a human can diagnose immediately.
  - Pattern reference: collect-then-raise mirrors `_duplicate_ids` in the loaders.
- Also validate every fixture-referenced `geometry_id` exists in the `geometry` table (JSON-level FK check):
  - Query `SELECT id FROM geometry WHERE state = %s` once; build an `id_set: set[str]`.
  - Iterate `rows`; collect any `parent_geometry_id` or `child_geometry_id` not in `id_set`.
  - If any missing: `raise OverlayFixtureError(f"fixture references unknown geometry ids: {missing!r}")`.
- In `_collect_overlay_rows`, after building all rows, call `_validate_coverage(conn, rows)` before returning.
**Verify:** `cd ingestion && .venv/bin/mypy ingestion/states/montana/build_overlay_fixture.py`

### T5: Deterministic JSON write (~3 min)
**Files:** `ingestion/states/montana/build_overlay_fixture.py`
**Depends on:** T4
**Action:** Replace the T2 `_write_fixture` stub with a writer that produces byte-identical output across runs.
**Details:**
- Implement `_write_fixture(rows: list[OverlayFixtureRow]) -> None`:
  - Sort rows: `sorted_rows = sorted(rows, key=lambda r: (r["parent_geometry_id"], r["child_geometry_id"], r["relationship"]))`. Three-key sort because a single (parent, child) pair could in principle have multiple relationship rows in future expansions; keying on relationship makes the order stable today and forward-compatible.
  - `MT_FIXTURE_DIR.mkdir(parents=True, exist_ok=True)` (defensive — directory already exists today).
  - Write atomically via tmp + rename: write to `OVERLAY_FIXTURE_PATH.with_suffix(".json.tmp")` then `.replace(OVERLAY_FIXTURE_PATH)`.
  - Use `json.dumps(sorted_rows, indent=2, sort_keys=True, ensure_ascii=False)` and append a trailing `"\n"` (POSIX convention; matches existing JSON fixtures in the directory).
  - Log INFO: `"wrote %d overlay rows to %s", len(sorted_rows), OVERLAY_FIXTURE_PATH`.
**Verify:** Implementation-only; `mypy` covers the typing. Determinism is asserted by tests in T7.

### T6: --explain flag emits EXPLAIN ANALYZE to stderr (~3 min)
**Files:** `ingestion/states/montana/build_overlay_fixture.py`
**Depends on:** T3
**Action:** When `--explain` is passed, run each spatial query under `EXPLAIN (ANALYZE, FORMAT TEXT)` and emit the plan to stderr with marker lines.
**Details:**
- Implement helper `_emit_explain(conn, label: str, sql: str, params: tuple[object, ...]) -> None`:
  - `with conn.cursor() as cur: cur.execute("EXPLAIN (ANALYZE, FORMAT TEXT) " + sql, params); rows = cur.fetchall()`
  - `print(f"# EXPLAIN ANALYZE: {label}", file=sys.stderr)` then `for row in rows: print(row[0], file=sys.stderr)` then `print("", file=sys.stderr)` (trailing blank line for readability).
- In `_collect_overlay_rows`, when `explain=True`, call `_emit_explain` for each of the three spatial query labels: `"HD→Portion"`, `"HD→CWD zone"`, `"HD→Restricted Area"`. Pass the same SQL + params used in the data fetch so the plan reflects the actual query.
- The HD self-reference path is not a SQL query; do not include it in EXPLAIN output.
- Plain runs (no `--explain`) must NOT execute EXPLAIN — keeps non-debug runs free of side effects.
**Verify:** Tests in T7 cover the wiring; manual verification deferred to live-load step (post-merge).

## Conventions

- **Geography-native PostGIS operators only.** Never `::geometry` cast in WHERE — see `.roughly/known-pitfalls.md` and epic spec lines 472–474.
- **Fail loud, collect-then-raise.** Mirrors `_duplicate_ids` in the loaders. Never silently skip.
- **Deterministic JSON.** `json.dumps(..., indent=2, sort_keys=True, ensure_ascii=False)` + sorted row list + trailing newline. Atomic write via `.tmp` + `.replace`.
- **No `print`; logging only.** `logger = logging.getLogger(__name__)`; INFO for layer counts and output path; DEBUG for per-row events. Exception: `--explain` writes raw plan text to stderr via `print(..., file=sys.stderr)` — that's the documented diagnostic channel.
- **CLI shape mirrors the loaders.** `main(argv: list[str] | None = None) -> int`, argparse, `logging.basicConfig`, `with db.connect() as conn:`, `if __name__ == "__main__": raise SystemExit(main())`.
- **No `any` types** (project-wide rule from CLAUDE.md). Use `object` or specific types.
- **ADR references:** ADR-004 (Postgres + PostGIS), ADR-010 (decomposed entity model), ADR-003 (ingestion is upstream and offline).
