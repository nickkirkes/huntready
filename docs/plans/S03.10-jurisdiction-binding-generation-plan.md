# Implementation Plan: S03.10 — jurisdiction_binding generation (Montana)

Plan-format-version: 1

## Source

`docs/planning/epics/E03-regulation-text-ingestion.md` lines 967-1097 (the S03.10 section), with three planning-level adjustments that override the as-written spec (see "Spec deviations" below).

## Goal

Create the first cross-cutting binding adapter `ingestion/states/montana/load_jurisdiction_bindings.py` that reads `geometry-overlays.json` + queries `regulation_record` rows + writes one `jurisdiction_binding` per qualified pair in a single atomic transaction. Also covers (a) statewide bindings to `MT-STATEWIDE-geom`, (b) no-hunt-zone Option A bindings via spatial nearby-rule, (c) source-citation attribution from the geometry table, (d) per-species cross-species filter.

## Spec deviations (PM-pre-approved at Stage 2 gate)

These three deviations from the written spec are baked into the plan:

1. **ID format** — spec line 1045 reads `{state}-{jurisdiction_code}-{species_group}-{license_year}-binding-{geometry_id}-{role}` but S03.6.1 shipped + locked `{state}-{jurisdiction_code}-{species_group}-{license_year}-{role}-{geometry_id}` at [load_regulation_records.py:116-117](ingestion/states/montana/load_regulation_records.py#L116). **Implementer imports `_JURISDICTION_BINDING_ID_FORMAT` from `load_regulation_records.py`**. Spec line 1045 gets a strike-through + footnote in T14.

2. **Row-count band** — spec AC #1087 says `1,500-3,500`. Initial planning estimate of `~770` was too narrow because it ignored 3-species fan-out on shared `MT-HD-deer-elk-lion-N-geom` geometries (mule_deer + whitetail + elk each get their own reg_record but share the same parent geometry — so the 235 fixture self-rows fan out by species multiplicity, not 1:1). Revised reasoning: 235 self-rows × per-geometry-species-fan-out + 155 portions × eligible-species + 188 RAs × cross-species filter + 8 CWDs × 3 deer/elk/whitetail = wide range from ~500 to ~900 overlay bindings + ~30-100 no-hunt-zone = **~500-1,000 total**. Plan's `_BINDING_COUNT_GUARD_BAND` is deliberately wide at `[400, 1100]` until **T16 (live run) produces the empirical count**, at which point the band gets narrowed to ±30% around the observed value in a follow-up PR (or in this PR if T16 runs before close). The wide band still catches a 2x regression. Spec AC #1087 gets revised to `400-1,100` in T14 with the same "narrow after T16" note.

3. **Source attribution** — spec AC #1088 says "cites the original ArcGIS layer" but doesn't specify the mechanism. Plan resolves: **`SELECT id, source FROM geometry WHERE id = ANY(%s)` pre-fetch at load time**. Authoritative; avoids re-deriving citation logic.

4. **No-hunt-zone "nearby" rule simplification — spec deviation surfaced during T0 probe (2026-05-23)**. Spec line 1062 says `ST_Touches OR ST_DWithin(centroid, centroid, 5000)`. Empirically the centroid-to-centroid clause produces **0 matches for ALL 3 orphan zones at 5km** (Glacier NP, Sun River, Yellowstone NP — every centroid pair > 5km apart). Replacement rule: **single clause `extensions.ST_DWithin(zone.geom, hd.geom, 5000)`** on the native `geography` type. This is boundary-to-boundary distance in meters; it covers ST_Touches (touching = 0m distance) and "near" simultaneously. Live counts confirm zero-match-free: Glacier=8, Sun River=8, Yellowstone=13 nearby HDs. The geography-native variant is also the canonical PostGIS pattern for lat/lon "nearby" queries. T8 + T14's spec correction reflect this; T15's working note documents the rationale + the empirical centroid-result evidence. Side note: `PostGIS` lives in the `extensions` schema in this Supabase project, so all `ST_*` calls in raw SQL must be schema-qualified (`extensions.ST_DWithin`); psycopg / SQLAlchemy use of ORM-mediated calls is unaffected, but our raw-SQL helper needs the prefix.

```python
class _OverlayRow(TypedDict):
    parent_geometry_id: str
    child_geometry_id: str
    parent_kind: str
    child_kind: str
    relationship: str
    role_for_e03: str

_REQUIRED_OVERLAY_KEYS = frozenset(_OverlayRow.__annotations__.keys())

def _load_overlay_fixture(path: Path) -> list[_OverlayRow]:
    with path.open() as f:
        data = json.load(f)
    raw_rows = data if isinstance(data, list) else data.get("relationships", [])
    if not isinstance(raw_rows, list):
        raise RuntimeError(
            f"overlay fixture at {path} has unexpected shape: "
            f"top-level is neither list nor dict-with-'relationships' key"
        )
    rows: list[_OverlayRow] = []
    for idx, element in enumerate(raw_rows):
        if not isinstance(element, dict):
            raise RuntimeError(
                f"overlay fixture row {idx} is not a dict: got {type(element).__name__}"
            )
        missing = _REQUIRED_OVERLAY_KEYS - element.keys()
        if missing:
            raise RuntimeError(
                f"overlay fixture row {idx} missing required keys: "
                f"{sorted(missing)}; present keys: {sorted(element.keys())}"
            )
        rows.append(cast(_OverlayRow, element))
    return rows
```

```python
# Step A: self-row short-circuit — always primary_unit eligible regardless of species prefix
if overlay_row["child_geometry_id"] == parent_geometry_id:
    return True

# Step B: parent geometry must match the species-expected prefix (defensive structural check —
# the caller should have filtered to the right parent already, but this catches a logic bug fast)
if species_group in {"elk", "mule_deer", "whitetail"}:
    if not parent_geometry_id.startswith("MT-HD-deer-elk-lion-"):
        return False
elif species_group == "pronghorn":
    if not parent_geometry_id.startswith("MT-HD-antelope-"):
        return False
elif species_group == "bear":
    if not parent_geometry_id.startswith("MT-HD-bear-"):
        return False
else:
    raise RuntimeError(f"unhandled species_group: {species_group!r}")

# Step C: filter child by per-species accept set on child_geometry_id prefix + child_kind
child_gid = overlay_row["child_geometry_id"]
role_for_e03 = overlay_row["role_for_e03"]

if role_for_e03 == "portion":
    # Per-species portion namespace gate (per spec line 1028-1032 filter table)
    if species_group == "elk":
        return child_gid.startswith("MT-HD-elk-")
    if species_group == "mule_deer":
        return child_gid.startswith("MT-HD-mule-deer-")
    if species_group == "whitetail":
        return child_gid.startswith("MT-HD-whitetail-")
    if species_group == "pronghorn":
        # antelope HDs and antelope portions share `MT-HD-antelope-` prefix; disambiguate via kind
        return child_gid.startswith("MT-HD-antelope-") and overlay_row["child_kind"] == "portion"
    if species_group == "bear":
        return child_gid.startswith("MT-HD-bear-")

if role_for_e03 == "restricted_area":
    if child_gid.startswith("MT-restricted-bigame-"):
        return True  # species-agnostic — all big-game species
    if child_gid.startswith("MT-restricted-elk-"):
        return species_group == "elk"  # elk-only discriminator
    return False  # unknown restricted-area namespace — fail closed

if role_for_e03 == "cwd_management_zone":
    return species_group in {"elk", "mule_deer", "whitetail"}  # CWD doesn't apply to bear or antelope

# Unknown role_for_e03 — fail closed (T3's _VALID_ROLE_FOR_E03 frozenset elsewhere will fail loud)
return False
```

```python
def _derive_parent_geometry_id(reg_record: RegulationRecord) -> str:
    """Map a regulation_record to its parent geometry_id.

    Patterns:
        MT-HD-deer-elk-lion-N  → MT-HD-deer-elk-lion-N-geom  (deer / elk / whitetail / mule_deer reg_records)
        MT-HD-antelope-N       → MT-HD-antelope-N-geom       (pronghorn reg_records, per-HD)
        MT-HD-bear-N           → MT-HD-bear-N-geom           (bear reg_records, per-BMU)
        MT-STATEWIDE-antelope  → MT-STATEWIDE-geom           (per ADR-018)
        MT-STATEWIDE-bear      → MT-STATEWIDE-geom           (per ADR-018)

    Fails loud on any other jurisdiction_code pattern (forces explicit handling for new states/anchors).
    """
    jc = reg_record.jurisdiction_code
    if jc in {"MT-STATEWIDE-antelope", "MT-STATEWIDE-bear"}:
        return _MT_STATEWIDE_GEOM_ID
    # All non-statewide jurisdiction_codes get a `-geom` suffix
    if jc.startswith(("MT-HD-deer-elk-lion-", "MT-HD-antelope-", "MT-HD-bear-")):
        return f"{jc}-geom"
    raise RuntimeError(
        f"unhandled jurisdiction_code pattern: {jc!r}. "
        f"Known patterns: MT-HD-deer-elk-lion-N, MT-HD-antelope-N, MT-HD-bear-N, "
        f"MT-STATEWIDE-antelope, MT-STATEWIDE-bear."
    )
```

```python
def _fetch_geometry_sources(
    conn: psycopg.Connection[tuple[object, ...]],
    geometry_ids: set[str],
) -> dict[str, SourceCitation]:
    """Pre-fetch source citations for every geometry id that will appear in a binding row.

    Per AC #1088 the geometry table is the authoritative source-of-record (the overlay
    fixture is derived). This adapter-local SELECT keeps the `db.py` surface limited to
    write/mutation helpers per S03.10 constraint #7. Fails loud if any requested id is
    missing — that means the overlay fixture references a geometry absent from the DB,
    which is a structural fixture/DB sync bug.
    """
    if not geometry_ids:
        return {}
    with conn.cursor() as cur:
        cur.execute(
            "SELECT id, source FROM geometry WHERE id = ANY(%s)",
            (sorted(geometry_ids),),
        )
        rows = cur.fetchall()
    sources = {row[0]: SourceCitation.model_validate(row[1]) for row in rows}
    missing = geometry_ids - sources.keys()
    if missing:
        sample = sorted(missing)[:10]
        suffix = "..." if len(missing) > 10 else ""
        raise RuntimeError(
            f"overlay fixture references {len(missing)} geometry ids that are "
            f"absent from the geometry table: {sample}{suffix}"
        )
    return sources
```

```python
def _build_statewide_bindings(
    statewide_reg_records: list[RegulationRecord],
    mt_statewide_source: SourceCitation,
) -> list[JurisdictionBinding]:
    """Emit one primary_unit binding per statewide regulation_record → MT-STATEWIDE-geom.

    Per ADR-018: any role beyond 'primary_unit' for a statewide binding requires
    an ADR amendment.

    V1 statewide reg_records: MT-STATEWIDE-antelope, MT-STATEWIDE-bear.
    The MT-STATEWIDE-bear binding was already written by S03.6.1; this builder
    re-derives it identically (same id format) so the loop UPSERTs as a no-op.
    """
    bindings: list[JurisdictionBinding] = []
    for rr in statewide_reg_records:
        role = "primary_unit"
        bid = _JURISDICTION_BINDING_ID_FORMAT.format(
            state=rr.state,
            jurisdiction_code=rr.jurisdiction_code,
            species_group=rr.species_group,
            license_year=rr.license_year,
            role=role,
            geometry_id=_MT_STATEWIDE_GEOM_ID,
        )
        bindings.append(JurisdictionBinding(
            id=bid,
            regulation_record_state=rr.state,
            regulation_record_jurisdiction_code=rr.jurisdiction_code,
            regulation_record_species_group=rr.species_group,
            regulation_record_license_year=rr.license_year,
            geometry_id=_MT_STATEWIDE_GEOM_ID,
            role=role,  # type: ignore[arg-type]  # Literal narrowing
            verbatim_rule=None,
            source=mt_statewide_source,
        ))
```

```python
def _query_nearby_hds_for_zone(
    conn: psycopg.Connection[tuple[object, ...]],
    zone_id: str,
) -> list[str]:
    """Return the geometry ids of every HD whose boundary is within the
    configured distance of the zone boundary (geography-native ST_DWithin).

    Per Spec Deviation #4, this uses single-clause
    `extensions.ST_DWithin(zone.geom, hd.geom, 5000)` on native geography type
    (NOT the spec's two-clause `ST_Touches OR ST_DWithin(centroid, centroid)`
    rule, which empirically returned 0 matches on real data at 5km).  Filter
    is `kind = 'hunting_district'` — portions excluded because the downstream
    `rrs_by_parent.get(portion_id, [])` lookup would silently drop them (see
    `_derive_parent_geometry_id` — only produces HD-shaped geometry ids).

    See ADR-016 § no-hunt-zone-handling, spec lines 1059-1067, and plan
    § "Spec deviations" #4 for the full rationale.
    """
    sql = """
        SELECT DISTINCT hd.id
        FROM geometry zone
        JOIN geometry hd
          ON hd.kind = 'hunting_district'
          AND extensions.ST_DWithin(zone.geom, hd.geom, %s)
        WHERE zone.id = %s
        ORDER BY hd.id
    """
    # kind = 'hunting_district' (S02.2 load_hds.py) — portions EXCLUDED because
    # regulation_record is HD-keyed in V1; portion ids never appear as keys in
    # rrs_by_parent (per `_derive_parent_geometry_id`) and would be silently
    # dropped by `_build_no_hunt_zone_bindings`.  Adding `'portion'` to this
    # filter is a silent-failure regression — see TestQueryNearbyHdsForZone
    # `test_sql_excludes_portions_regression_guard`.
    # ST_DWithin on native geography (boundary-to-boundary, meters) — single clause supersedes
    # spec's two-clause ST_Touches OR ST_DWithin(centroid, centroid, 5000) per Spec Deviation #4
    # (centroid-to-centroid empirically returned 0 matches for all 3 orphan zones at 5km).
    # extensions.* prefix is required: PostGIS lives in the `extensions` schema in this project.
    with conn.cursor() as cur:
        cur.execute(sql, (_NO_HUNT_ZONE_NEARBY_DISTANCE_M, zone_id))
        return [row[0] for row in cur.fetchall()]
```
