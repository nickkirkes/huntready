# Implementation Plan: S03.6 — regulation_record ingestion + geometry.legal_description population

Plan-format-version: 1

## Story

`docs/planning/epics/E03-regulation-text-ingestion.md` lines 522-604. First E03 database-write story. Reads three extraction artifacts (`dea-2026.json`, `black-bear-2026.json`, `legal-descriptions-2026.json`) and writes V1 Montana's anchor `regulation_record` rows + populates `geometry.legal_description` from S03.5's matched entries.

## Key decisions baked into this plan

**OQ1 resolution — Option (c): `regulation_record.verbatim_rule` does NOT exist; section-level text decomposes into downstream entities.**
- DEA section text decomposes via S03.7 (`season_definition.verbatim_rule`, `license_tag.verbatim_rule`) and the NOTE-line capture in this story (`regulation_record.additional_rules: VerbatimRule[]`).
- Bear per-BMU text decomposes via S03.7 (`season_definition.verbatim_rule` per window with `closure_predicate`; `license_tag.verbatim_rule` for hound NR license) and S03.9 (`reporting_obligation.verbatim_rule` for closure prose).
- This story writes NOTE lines to `additional_rules` only. The artifact's full `verbatim_text` field stays in the JSON artifact (committed to repo) as a debug/audit aid; **not denormalized into the DB**.
- Epic line 561-571 (verbatim chain mapping table) and line 565 are contradicted by this decision — `T11` amends the epic with a footnote and strike-through.
- This decision warrants an open-question entry in `docs/open-questions.md`; no new ADR (refinement of ADR-008's decomposition story, not a new commitment).

**OQ7 resolution — Row-count fail-loud guard introduced in this story.**
- Spec estimate: ~514. Actual artifact-derived count: ~436. Threshold: written count must be ≥70% and ≤130% of 514. Concrete bounds (after `int(...)` truncation): `int(514 * 0.70) = 359` ≤ written ≤ `int(514 * 1.30) = 668`. Below or above → exit non-zero with diagnostic.

**OQ8 resolution — `page_reference` collapse uses `pdf.page_reference_to_str()` (already shipped in S03.2).**

**Empty `legal_description` → write `NULL` not `""`** (review N4).

### T5: Implement NOTE-line parser (~4 min)

**Files:** `ingestion/states/montana/load_regulation_records.py`

**Depends on:** T3

**Action:** Add pure function `_extract_note_lines(verbatim_text: str, citation: SourceCitation, confidence: Confidence) -> list[VerbatimRule]`. Parses `NOTE:` prefixed lines (and their continuation lines) out of a section's full-page `verbatim_text` and constructs `VerbatimRule` instances for each.

**Details:**
- Regex pattern (line-anchored, multi-line continuation):
  ```python
  _NOTE_LINE_RE = re.compile(
      r"^NOTE:\s*(?P<body>.*?)(?=\n^NOTE:|\n^[A-Z]{3,}|\Z)",
      re.MULTILINE | re.DOTALL,
  )
  ```
  - Anchors at line start (`^` with `re.MULTILINE`).
  - Captures body up to the next NOTE-line OR up to the next ALL-CAPS-LIKE line (e.g., `ARCHERY APPLY EARLY GENERAL HERITAGE LATE OPPORTUNITY SPECIFIC` — table column headers — terminate the NOTE).
  - DOTALL allows continuation lines within the body.
- Normalize captured body: `body = match.group("body").strip()`; collapse internal whitespace via `body = re.sub(r"\s+", " ", body)`. This is **extras-collapse only** (matches S03.3's extras-only collapse rule per `extract_dea.py:44-60` docstring); we are normalizing the rule prose, not paraphrasing — ADR-008-safe because the content is identical, only whitespace runs are normalized for human readability.
- Skip empty bodies after strip (defensive; shouldn't fire on real data).
- Construct `VerbatimRule(text=f"NOTE: {body}", page_reference=citation.page_reference, confidence=confidence, source=citation)`. **Prepend `"NOTE:"`** so the verbatim_rule preserves the literal source phrasing — readers of `additional_rules` see the original NOTE prefix.
- Return list in source order.
- **Free-prose flag-and-surface:** If the section has prose paragraphs that don't begin with `NOTE:` and aren't part of the table rows, this parser misses them. For V1 we accept the gap (the AC notes this is intentional and any miss should be flagged in S03.6.md working note). T9 documents the limitation; S03.12 UAT spot-checks.

**Verify:** Unit tests in T7 cover: single NOTE, multi-NOTE in one section, multi-line NOTE (continuation), NOTE terminated by ALL-CAPS header, section with no NOTEs (returns empty list), whitespace collapse. Run `cd ingestion && .venv/bin/pytest tests/test_load_regulation_records.py -k note -x -v`.

### T9: Implement legal_description write path + empty-string guard + count guard (~5 min)

**Files:** `ingestion/states/montana/load_regulation_records.py`, `ingestion/tests/test_load_regulation_records.py`, `docs/planning/epics/E03-confidence-findings/S03.6.md` (new)

**Depends on:** T2, T3

**Action:** Add `_legal_description_writes(legal_desc_artifact: dict) -> list[tuple[str, str | None]]` that returns `(geometry_id, text_or_None)` pairs from the `matched` array with the empty-string guard applied. Add `_assert_count_within_guard(written_count: int) -> None` helper. Create the working note.

**Details:**
- `_legal_description_writes`:
  ```python
  def _legal_description_writes(legal_desc_artifact: dict) -> list[tuple[str, str | None]]:
      writes = []
      for entry in legal_desc_artifact["matched"]:
          text = entry["verbatim_description"].strip() if entry.get("verbatim_description") else None
          if not text:
              text = None  # NULL out empty/whitespace-only
          writes.append((entry["geometry_id"], text))
      return writes
  ```
  - Returns list of (geometry_id, text|None) tuples. The caller (T10) loops and calls `db.update_legal_description(conn, id, text)`. The `update_legal_description` helper raises if `id` is unknown — fail-loud.
- `_assert_count_within_guard(written: int) -> None`:
  ```python
  def _assert_count_within_guard(written: int) -> None:
      lower = int(_SPEC_ESTIMATE_TOTAL * _COUNT_GUARD_MIN_RATIO)   # 359
      upper = int(_SPEC_ESTIMATE_TOTAL * _COUNT_GUARD_MAX_RATIO)   # 668
      if not (lower <= written <= upper):
          msg = (
              f"regulation_record count guard failed: wrote {written} rows; "
              f"expected approximately {_SPEC_ESTIMATE_TOTAL} (acceptable range "
              f"{lower}-{upper}, ±30% of spec estimate). This indicates a "
              f"catastrophic regression in one of the extraction artifacts. "
              f"Investigate before re-running."
          )
          raise RuntimeError(msg)
  ```
- Unit tests added to `test_load_regulation_records.py`:
  - `TestLegalDescriptionWrites`:
    - `test_normal_text_passed_through`: matched entry with non-empty `verbatim_description` → tuple's text matches.
    - `test_empty_string_becomes_none`: matched entry with `verbatim_description=""` → tuple's text is None.
    - `test_whitespace_only_becomes_none`: `verbatim_description="   \n  "` → tuple's text is None.
    - `test_unmatched_array_ignored`: synthetic artifact with `matched=[]` and `unmatched=[...]` → returns `[]` (unmatched entries are NEVER written; this is by-design per S03.5's contract).
  - `TestCountGuard`:
    - `test_count_within_range_passes`: 436 (actual), 359 (lower bound — `int(514 * 0.70)`), 668 (upper bound — `int(514 * 1.30)`) → no raise.

**Watch for:**
- Determinism: artifact iteration order must be preserved end-to-end (artifact ordering → record ordering → cursor execution ordering). Re-runs with identical artifacts must produce identical SQL execution sequences.
- Transaction boundary: single `conn.commit()` after all writes (records + legal_description updates). A failure mid-write rolls back everything.
- The `cur.rowcount == 0` check in `update_legal_description` is critical — without it, a matcher bug (legal-descriptions writing a `geometry_id` that doesn't exist in `geometry` table) would silently no-op and pass tests.
- `additional_rules` is a `list[VerbatimRule]`, not `Optional[list]`. Empty list is the correct "no NOTEs" representation; DB default is `'[]'`.
- The bear artifact has `extraction_confidence: "medium"` on **every** row in V1 (all 35 are correction-touched). The aggregation logic must still handle other tiers defensively — V2 corrections may touch only a subset of BMUs.
