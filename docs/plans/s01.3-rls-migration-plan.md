# Implementation Plan: S01.3 — RLS Deny-All Migration

## Context

Per ADR-002 and ADR-004, PostgREST must be structurally disabled so the MCP server is the only read path to regulation data. S01.2 created all 10 tables but applied no RLS. This migration adds three defense-in-depth security layers to every table, ensuring `anon` and `authenticated` roles cannot access data — only `service_role` (which has the `bypassrls` attribute) can.

## Tasks

### T1: Create RLS deny-all migration (~5 min)
**Files:** `supabase/migrations/20260425000001_rls_deny_all.sql`
**Action:** Create a timestamped migration file that applies three security layers to all 10 tables
**Details:**

Create the migration file with a clear header comment explaining the purpose and referencing ADR-002 and ADR-004.

For each of the 10 tables (`regulation_record`, `season_definition`, `license_tag`, `draw_spec`, `reporting_obligation`, `geometry`, `jurisdiction_binding`, `regulation_season`, `regulation_license`, `regulation_reporting`), apply these three statements in order:

**Layer 1 — Enable and force RLS:**
```sql
ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;
ALTER TABLE <table> FORCE ROW LEVEL SECURITY;
```
`FORCE` ensures RLS applies even to the table owner (`postgres` role). `service_role` is unaffected because its bypass comes from the `bypassrls` role attribute, not table ownership.

**Layer 2 — Deny-all policies:**
```sql
CREATE POLICY "Deny all access for authenticated" ON <table>
  FOR ALL TO authenticated USING (false) WITH CHECK (false);
CREATE POLICY "Deny all access for anon" ON <table>
  FOR ALL TO anon USING (false) WITH CHECK (false);
```

**Layer 3 — Explicit REVOKE (defense-in-depth):**
```sql
REVOKE ALL ON TABLE <table> FROM anon, authenticated;
```
Removes Supabase's default `ALTER DEFAULT PRIVILEGES` grants.

Group statements by table with a comment separator for each table. Order tables in the same order as the S01.2 migration for consistency:
1. `regulation_record`
2. `season_definition`
3. `license_tag`
4. `draw_spec`
5. `reporting_obligation`
6. `geometry`
7. `jurisdiction_binding`
8. `regulation_season`
9. `regulation_license`
10. `regulation_reporting`

**Verify:** `supabase db push --dry-run` (if available) or manual review that SQL is syntactically valid. Pre-commit hooks won't trigger on `.sql` files.
**UI:** no

## Blast Radius
- Do NOT modify: `supabase/migrations/20260425000000_initial_schema.sql` or any other existing file
- Watch for: The table named `geometry` — Postgres won't confuse it with the PostGIS type here since we're referencing it via `ALTER TABLE` not `CREATE TABLE`

## Conventions
- Migration timestamp sorts after S01.2's `20260425000000` (using `20260425000001`)
- Same table ordering as S01.2 for reviewability
- Policy names follow the pattern from the epic spec: `"Deny all access for <role>"`
- References ADR-002 (MCP canonical interface) and ADR-004 (Supabase + PostGIS + RLS commitment)
